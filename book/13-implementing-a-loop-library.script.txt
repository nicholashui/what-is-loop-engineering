In this chapter, you'll learn how to build a loop library of your own. That's a curated, version-controlled collection of reusable loop definitions your team can browse, copy, and run. We'll define a single loop entry schema, so every loop is described the same way. We'll lay out a repository structure to store them. We'll show how to make the library searchable. And most importantly, we'll show how to turn a static library entry into a running loop, using the machinery from the build chapter. By the end, you'll be able to take the forty-five patterns from the previous chapter, or your own, and turn them into an executable library.

So let's start by distinguishing a catalog from a library. The previous chapter was a catalog, a read-only tour of loops other people wrote. A loop library is something stronger. It's a living artifact in your own repository, where each entry is structured data, not prose, so it can be searched, validated, versioned, and, the crucial step, executed. The difference is the same one this book keeps returning to. Prose describes a loop. A library defines it precisely enough that a machine can run it.

Implementing a library means making three decisions and then wiring them together. First, a schema, one consistent shape for describing every loop, so entries are comparable and machine-readable. Second, a store, where the entries live and how they're organized, versioned, and reviewed. And third, a runner, how a library entry becomes a running loop assembled from the eight components.

Let's take them in that order, starting with the schema. The foundation of a library is a single, agreed-upon description for what a loop is. Notice that every entry in the previous chapter implicitly answered the same questions: who wrote it, what it's for, how it works, and when it stops. Make those questions explicit fields, and you have a schema. A good schema mirrors the eight components, so a library entry maps one-to-one onto a runnable loop.

The practical form is a Markdown file with YAML frontmatter. It's readable by a human and parseable by a machine, which is exactly what a library entry needs. The frontmatter carries identity and discovery fields, like an id, name, author, and category, and then the loop contract itself: the trigger, the generator, the evaluator, the isolation strategy, the state file, the stopping conditions, the connectors, and the skills. Those are exactly the eight components. A library entry is therefore a complete, declarative description of a loop. Nothing about how to run it lives outside the file.

Two design principles make this work. First, every component is a field, so the entry is the contract. And second, the prompt is a reference, not inline text. The actual agent instructions live in a separate prompt file, which keeps prompts reusable across loops and editable on their own.

Not every loop needs every field, but a small core should be mandatory, so the library never contains an unrunnable entry. The id, name, author, and category are required for identity and discovery. The purpose is required as a one-line checkable intent. The generator's prompt file is required, because there's no loop without a generator. And the stopping conditions are required, because an unbounded loop is a defect, not an entry. The evaluator is strongly recommended, because without it, "done" is just the agent's say-so. The single most important rule is this: no entry may lack a stopping condition. That one constraint is what keeps a library of autonomous loops from being a library of runaway processes.

Now the store. With a schema fixed, the store is straightforward: a directory of entry files under version control. Plain files in git give you history, review, branching, and diffs for free, the same properties that make the loops themselves trustworthy. The layout is one file per loop, foldered by category, mirroring the five categories from the previous chapter. Prompts, rubrics, and skills are shared, first-class files, so the same rubric can back many loops. And a machine-checkable schema lives right in the repo, so CI can validate every entry on every change. That's the library applying an evaluator to itself.

The library should hold itself to the standard it preaches. A pre-merge check that validates every entry against the schema is the minimum bar. It enforces three non-negotiables: a valid shape, a present stopping condition, and an existing prompt file. Run it in CI, and the library can never merge an entry that would fail to run, or run forever.

Next, making it searchable. A library only earns its name when people can find the right loop quickly. Because every entry carries structured frontmatter, search is mostly a matter of reading that metadata and projecting it into something browsable. Two low-cost mechanisms cover almost every need. There's a generated index: a small script reads every entry's frontmatter and writes a grouped, linked README, just like the catalog from the previous chapter. And there's faceted, tag-based filtering: because category, tags, and maturity are structured fields, you can filter without a database. The metadata is the search index.

Now the step that separates a real library from a fancy notes folder: turning a library entry into a running loop. A library entry is a declaration. Running it means handing that declaration to a runner that assembles the eight components and drives the lifecycle. It's the exact production loop from the build chapter, now parameterized by the entry instead of hard-coded.

The runner's job is a faithful translation, field by field. The trigger decides when to start. The isolation strategy creates a git worktree on a throwaway branch per attempt. The generator's prompt file and inputs render the prompt and invoke the agent. The evaluator runs the objective gate, then the critic or rubric gate. The state file persists progress and recycles it into the next prompt. And the stopping conditions decide whether to continue, merge and stop, or escalate. Each maps directly onto a stage of the lifecycle.

The point is not the specific script. It's the separation. The runner is written once and contains all the orchestration. The library entries contain only declarations. Add a hundred loops, and the runner doesn't change. That's the payoff of the schema. A loop becomes data, and running any loop is the same code path.

This is also where parameterized loops come in. Many catalog loops had blanks, like "stop after N successes," or "repository task," or "quality threshold." A library makes those blanks first-class inputs, supplied at run time. So you can run the same entry against different targets with no edits to the entry. A single well-parameterized entry replaces a dozen near-duplicate copies. That's how a library stays small as it grows. You add parameters, not forks.

So how do you get to your first library? You don't build the whole thing up front, any more than you built the whole loop up front in the build chapter. Grow it the same subtractive way. First, promote one working loop: take a loop you already run, write it up in the schema, and commit it as the first entry. Second, write the runner once, by generalizing your existing orchestrator. Third, add the schema validator to CI, so the library can't accept a broken or unbounded entry. Fourth, promote loops only as you actually reuse them. And fifth, generate the index, so the library is browsable. The discipline is the same as everywhere in this book: add a loop to the library when not having it there causes real duplication, and not before.

So let's bring the key ideas together. A loop library turns the read-only catalog into a living, executable artifact, where each loop becomes structured, version-controlled data rather than prose. Implementing one means three decisions wired together: a schema, one shape for every loop; a store, version-controlled files; and a runner, which executes any entry.

The schema should make every component an explicit field, with the prompt stored as a referenced file, and no entry may lack a stopping condition. The store is a git directory of one-file-per-loop entries, foldered by category, with shared prompts, rubrics, and skills, and a JSON Schema validated in CI. Search comes almost for free from the structured frontmatter. The runner is written once and translates an entry's fields onto the lifecycle. It's the production loop with its constants replaced by reads from the entry, so adding loops never changes the orchestration. Parameterized inputs turn the catalog's blanks into run-time arguments. And you build the library subtractively: promote one working loop, write the runner, add CI validation, then add entries only when reuse actually demands them.
