# 14. 策展、營運與擴展一個 Loop 庫

> **本章內容：** 你已經有一個可以描述同運行 Loop（迴圈）嘅庫（第 13 章）。本章講嘅，係點樣喺個庫增長、用嘅人愈來愈多嘅同時，令佢保持*優質*。我哋會講一個令團隊可以安全噉加 Loop 嘅貢獻流程、令劣質產出（slop）入唔到嚟嘅品質標準同審查規範、點樣為條目做版本管理同棄用、邊啲度量指標可以話俾你聽一個 Loop 值唔值得保留，以及當眾多自主 Loop 喺一個團隊之間運行時隨之而來嘅治理同安全顧慮。讀完之後，你就有能力將一個 Loop 庫當作共享基礎設施嚟營運，而唔只係一個個人嘅腳本資料夾。

## 一個庫係一件產品，唔係一個資料夾

幾個 Loop 放喺一個目錄入面，係一個方便。而一個俾成個團隊信任、並針對真實系統運行嘅庫，就係*基礎設施*——而基礎設施需要一個負責人、一個品質標準、一個安全改動嘅方法，以及一個量度佢有冇值返條數嘅方法。一個成功嘅庫嘅失敗模式，唔係 Loop 太少;而係劣質嘅太多，搞到冇人分得清邊個條目可以信。策展，就係防止呢樣嘢嘅工作。

本章假設你已經有第 13 章嗰套機制——schema、儲存、runner、CI 驗證器——並喺佢哋周圍加上*人*嘅一層：Loop 點樣入嚟、佢哋點樣保持優質，以及成件事點樣擴展到一個作者以上。

## 貢獻流程

你喺一個 Loop *內部*所套用嗰份紀律——提出一個改動、獨立驗證佢、只有喺通過時先接受——同樣適用於對個庫*本身*嘅改動。將一個新嘅或修改過嘅 Loop 條目，當作任何其他程式碼改動噉處理：一個 pull request、一次審查，以及一道自動關卡。

一個由一位貢獻者擴展到多位嘅流程：

1. **透過 pull request 提出。** 一位貢獻者喺一個分支上新增或編輯一個條目檔案（連同任何被引用嘅 prompt、評分準則或技能），並開出一個 PR。冇任何嘢直接合併入主庫。
2. **自動關卡先運行。** CI 運行第 13 章嗰個驗證器：schema 合規、一個存在嘅停止條件，以及一個存在嘅 prompt 檔案。一個過唔到呢啲關卡嘅 PR，永遠到唔到人類審查者嗰度——平嘅檢查先行。
3. **一位維護者對照品質標準審查。** 一個人類去檢查自動化檢查唔到嘅嘢：個目標係咪真係可被檢查、個評估器係咪獨立於生成器、啲護欄係咪老實、標注係咪正確？（品質標準喺下一節詳述。）
4. **接受之前先做試運行。** 對於任何非瑣碎嘅 Loop，審查者透過 runner 將佢針對一個安全目標運行一次，確認佢嘅行為同所描述一致——尤其係佢真係會*停*。一個從未被執行過嘅 Loop，係一個假設，唔係一個庫條目。
5. **合併將佢晉升為 `stable`。** 一經接受，個條目嘅 `maturity` 就設為 `stable`，並重新生成個索引。呢個 Loop 而家可被團隊發現同運行。

呢個原則係遞迴嘅，而且值得講白：**一個 loop 庫，本身就係由一個 loop 嚟維護**——提出、獨立驗證、只有喺通過時先接受。第 12 章嗰啲魔鬼代言人同建造者—審查者形態，套用喺庫條目身上，同套用喺嗰啲條目所作用嘅程式碼身上，係一樣咁多嘅。

## 品質標準

自動驗證證明一個條目*形式正確*。品質標準就證明佢*優質*。每一個獲准入庫嘅條目，都應該通過一份書面、可見嘅清單——個庫以之要求自己嘅標準，存放喺儲存庫入面，令貢獻者喺開 PR 之前可以自我檢查：

- **個目標係外部可被檢查嘅。** 措辭為一個機器或者一個獨立審查者可以確認嘅成果——而唔係「改善 X」。（第 08 章嘅目標規則。）
- **個評估器獨立於生成器。** 「完成」永遠唔係靠 agent 自己嘅報告;有一個真實嘅客觀關卡，並喺需要判斷之處有一個評分準則關卡。
- **停止條件係明確而分層嘅。** 一個成功出口*加上*至少一個預算上限（迭代次數同／或成本）。一個無界限嘅條目即時不過關。
- **隔離同一條人類升級路徑都有定義。** 個 Loop 係可逆嘅，並且佢知道幾時應該停低問人，而唔係靜靜雞失敗或者燒爆預算。
- **波及範圍有記錄。** 個條目講明個 Loop 可以掂啲乜、佢需要乜權限，以及幾時*唔應該*用佢。
- **標注同來源係正確嘅。** 作者身分同任何上游來源都被老實噉記錄（如第 12 章嘅目錄）。

將呢份清單做成一個 PR 範本，咁每一個貢獻都會對照同一套準則去審查。一個過唔到標準嘅 Loop 唔係永遠被拒絕——而係以 `experimental` 或 `draft` 嘅身分獲准，清楚標明，令用戶知道佢仲未贏得完全嘅信任。

## 版本管理同棄用

Loop 會變：一個 prompt 改善咗、一個評估器收緊咗、一個更好嘅做法取代咗一個舊嘅。因為每一個條目都帶住一個語意化嘅 `version` 同一個 `maturity` 欄位（第 13 章），個庫可以喺唔搞冧依賴佢嘅人嘅前提下演化。

- **為每一個有意義嘅改動做版本。** 當 prompt、評估器或停止條件以一種會改變行為嘅方式改變時，提升個條目嘅 `version`。釘住某個版本嘅呼叫者得到可重現性;追蹤最新版嘅呼叫者得到改善。
- **用成熟度階梯。** 新條目以 `draft` 或 `experimental` 進入，一經證實就晉升為 `stable`，被取代時就以 `deprecated` 終結。`maturity` 係一個第一等嘅搜尋分面，於是用戶可以選擇預設只見 `stable` 嘅 loop。
- **棄用，唔好刪除。** 當一個 loop 被取代時，將佢標為 `deprecated`，喺條目入面指向佢嘅後繼者，並保留佢一個發行週期。靜靜雞刪除一個條目，會悄悄噉搞冧任何引用咗佢嘅嘢;棄用佢就俾用戶一條遷移路徑。（呢個就係第 12 章嘅「傳播合規」紀律，套用喺個庫自己身上。）

成熟度階梯，正正就係令一個庫能夠*歡迎新 loop* 而又*唔對信任掉以輕心*嘅嘢：任何人都可以貢獻一個 `draft`，但只有經過證實嘅 loop 先帶住嗰個示意「呢個可以針對生產環境運行」嘅 `stable` 徽章。

## 度量一個 Loop 值唔值得佢嗰個位

一個庫入面嘅 Loop，係一個「佢值得運行」嘅聲稱。一旦啲 loop 定期運行，你就可以用證據去取代嗰個聲稱。因為個 runner 本身已經驅動每一個 loop，所以你只要為佢加裝一次儀表，就免費得到成個庫嘅遙測。

值得追蹤嘅每 loop 指標：

- **接受率。** 喺到達一個停止條件嘅運行之中，有幾多係以*成功*結束，而唔係撞到預算上限或者升級？一個幾乎從未到達佢成功出口嘅 loop，有一個目標或者評估器嘅問題。
- **到達成功所需嘅迭代次數同成本。** 一次成功運行平均消耗嘅預算。成本隨時間上升，係一個早期警告，提示個 loop 或者佢所針對嘅程式碼庫已經漂移。
- **升級率。** 個 loop 幾經常交俾人類。一啲升級係健康嘅;持續升級就意味住個 loop 範圍定錯咗。
- **使用量。** 個 loop 究竟幾經常被運行。一個冇人運行嘅條目係文件，唔係基礎設施——一個棄用嘅候選。

呢啲直接連返第 08 章嗰啲停止條件：嗰啲*界定*一個 loop 行為嘅出口，同時亦係你*點算*嚟判斷佢嘅事件。一個接受率高、成本穩定、升級罕見嘅 loop，就贏得咗佢個 `stable` 徽章;一個喺度撞緊預算嘅 loop，就喺度話俾你聽要修佢或者退役佢。

## 規模化嘅治理同安全

一旦一個庫被共享、而佢啲 loop 喺好多儲存庫之間自主運行，第 10 章嗰啲風險就唔再係一個工程師嘅問題，而變成一個組織嘅問題。一個出事嘅個人 loop 頂多令*你*要 `git worktree remove`;而一個有廣泛權限、出事嘅共享 loop，就可能影響到所有人。負責任噉營運一個庫，意味住要喺佢周圍加上限制。

最重要嘅控制：

- **最小權限連接器。** 一個 loop 嘅 `connectors` 應該只授予佢需要嘅存取——可能嘅話唯讀、範圍受限嘅權杖、唔好將長期生產憑證焗死喺條目入面。schema 記錄一個 loop 用*邊啲*連接器;治理就確保每一個都係最小範圍。
- **中央預算上限。** 喺逐條目嘅 `max_cost_usd` 之上，執行一個組織層級嘅上限，咁就算每一個 loop 各自都規規矩矩，眾多 loop 嘅*總和*都唔會失控。分層預算，喺整個機隊層級套用。
- **一條審計軌跡。** 記錄每一次運行——邊個條目、邊個版本、邊啲輸入、邊個觸發、佢改咗乜、佢點樣停。當一個自主 loop 掂到一個共享系統時，你必須能夠事後回答「跑咗乜，點解？」。
- **高波及範圍 loop 嘅人類審批關卡。** 掂到生產資料、公開文案或不可逆狀態嘅 loop，應該喺佢哋嘅結果落地之前要求明確嘅人類簽核——正正就係第 12 章好幾個 loop（客戶 AI 部署 loop、生產資料清理 loop）按設計就建好嘅審批界線。
- **一個清晰嘅負責人。** 有一個人為個庫負責：品質標準、棄用決定、安全姿態。冇負責人嘅共享基礎設施，會衰退成本章存在嘅目的本身——嗰種劣質嘅蔓延。

呢啲都唔係新諗法——佢哋就係第 10 章嗰啲緩解措施（界定成本、防範劣質產出、保留人類判斷），由一個單一 loop 提升到一整隊 loop。擴展一個庫，好大程度上就係令嗰啲逐 loop 嘅保障措施，喺整個組織嘅層級都站得住腳嘅工作。

## 擴展個庫本身

當個庫增長到超過幾打條目時，兩股壓力會出現：發現變得更難，而重複亦會悄悄爬入嚟。兩者都可以用你已經有嘅結構去管理。

- **倚靠形態，而非條目。** 第 12 章顯示咗 loop 會收束成一小撮形態（審計—排序—修正、生成—評分—改進，先建造—再審查等等）。當兩個條目係同一種形態、只係目標唔同嗰陣，寧可用一個*參數化*嘅條目，都唔好用兩個近乎重複嘅。個庫應該喺*參數同 prompt* 入面增長，而唔係喺複本入面。
- **針對蔓延做策展。** 定期對個庫做一次清理巡查——啱啱好，呢樣嘢本身都係一個 loop（第 12 章嘅「管家」形態）：搵出冇人運行嘅條目、重複、失效引用同過時 prompt，並棄用或合併佢哋。一個淨係識增長嘅庫，最終會變到搜尋唔到。
- **按領域聯邦化，共享 runner。** 大型組織可以容許各團隊擁有自己嘅類別資料夾（工程、設計、營運），同時共享一個 schema 同一個 runner。本地擁有權令條目保持貼題;共享嘅 runner 令執行保持一致。

終局係一個*相對於佢嘅有用程度而保持細小*嘅庫：一套經過策展、可信、參數化、被度量嘅 loop，俾成個團隊去取用——而唔係一堆不斷增長嘅一次性腳本。呢個就係本書由頭到尾所主張嗰份減法紀律，套用喺高一個層級之上：當「冇將一個 loop 放入個庫」會造成真正嘅重複嗰陣，先將佢加入，喺佢值返個位嗰段時間保留佢，並喺佢唔再值得時退役佢。

## 重點摘要

- 一個共享嘅 **Loop 庫係基礎設施，唔係一個資料夾**：佢嘅失敗模式係劣質條目太多，所以**策展**——控制乜嘢可以入嚟、乜嘢可以留低——係核心工作。
- 對個庫嘅改動，跟住**同 Loop 本身一樣嘅 loop 紀律**：透過 PR 提出、運行自動嘅 schema／停止條件關卡、對照一份書面**品質標準**審查，並喺合併到 `stable` 之前做試運行去確認個 loop 真係會停。
- 個 **品質標準** 要求一個可被檢查嘅目標、一個獨立嘅評估器、明確而分層嘅停止條件、有定義嘅隔離同人類升級、一個有記錄嘅波及範圍，以及正確嘅標注——並作為一個 PR 範本保存，令貢獻者自我檢查。
- **做版本同棄用**，而非搞冧：行為改變就提升語意化 `version`，一條 `draft → experimental → stable → deprecated` 嘅成熟度階梯並以搜尋分面暴露出嚟，以及指向後繼者而非靜靜雞刪除。
- 用每一個 loop 自己嘅停止條件本身已經產生嘅事件去**度量每一個 loop**——接受率、到達成功所需嘅迭代次數／成本、升級率同使用量——並退役嗰啲喺度亂撞或者冇人運行嘅 loop。
- **治理將第 10 章嘅緩解措施擴展到一整隊**：最小權限連接器、組織層級嘅預算上限、每一次運行嘅審計軌跡、高波及範圍 loop 嘅人類審批關卡，以及一個清晰可問責嘅負責人。
- **按形態擴展，而非按複本**：寧可用一個參數化嘅條目，都唔好用近乎重複嘅;定期做清理巡查去對抗蔓延;並可選擇噉聯邦化類別擁有權，同時共享一個 schema 同 runner——令個庫相對於佢嘅有用程度而保持細小。
