喺呢一章，我哋會打開一個循環，逐一為佢嘅各個部件命名。睇完之後，你就會認識構成一個循環嘅八個組成部分。佢哋分別係：Automation，或者叫 Trigger，即係自動化、觸發器；Worktrees，負責隔離，即係工作樹；Generator Agent，即係生成代理；Evaluator，即係評估器；State，又或者叫 Memory，即係狀態、記憶；Skills and Knowledge，即係技能同知識；Connectors，即係連接器；最後就係 Stopping Condition，即係停止條件。就每一個部件，你會學到佢嘅用途、佢喺循環裏面所做嘅嗰一份工作，亦會至少睇到一個具體例子，講解嗰份工作喺實務上係點樣達成嘅。睇完之後，你就能夠指住循環嘅每一個構建組件，講出點解每一個都各得其所。

之前我哋為你描繪過一個循環嘅圖像：一個會自行轉動嘅圓圈。佢會提示、會觀察、會驗證，然後會迭代。我哋亦都展示過呢個圓圈係點樣由一次單一嘅代理對話入面抽離出嚟，再建構成一個系統。而家，就係時候打開呢部機器，逐一為部件命名喇。

一個循環由八個組成部分裝配而成。每一個都對應到一項判斷或者一份工作；喺人手提示嘅時候，呢啲工作本來係你親自去做嘅。決定幾時開始。避免工作互相碰撞。做實際嘅編碼。檢查結果係咪夠好。記住發生過嘅事。知道專案嘅規則。同外面嘅世界溝通。以及，決定幾時停止。把每一項都搬入系統中一個明確嘅部件裏面，你就得到一個循環。

我哋會逐一去睇，按照工作流經佢哋嘅次序。就每一個組成部分，個結構都係刻意保持一致嘅。我哋會講佢係乜嘢、佢嘅用途係乜，以及至少一個具體例子。咁樣你就可以把呢八個都記喺腦海裏面，當佢哋係同一個模式嘅各種變奏：一個只做一份工作嘅部件，好等個循環唔需要由人去做嗰份工作。

我哋先由 Automation，又或者叫 Trigger，即係觸發器講起。

觸發器係決定工作喺幾時、以乜嘢方式進入循環嘅組成部分。佢就係起跑槍，一個唔使人手打個「go」字就能夠啟動一次迭代嘅機制。喺人手提示嘅時候，係你去開始每一個回合。你決定有工作要做，然後動手去做。觸發器把嗰個決定搬入系統裏面，咁個循環就可以按時間表、又或者因應某個事件而開始工作，而唔使等一個人去發起佢。呢樣就係讓循環可以喺你瞓覺嗰陣都運行緊嘅關鍵。工作會自行開始。

一個觸發器可以有幾種唔同嘅形態。佢可以係一個排程任務，譬如話一個每晚凌晨兩點觸發嘅任務，去接手下一項工作。佢亦可以係一個綁住新議題嘅 webhook，咁一有議題喺追蹤系統入面開立，個 webhook 就會觸發，個循環就會開始處理佢。又或者，佢可以係一條簡單嘅指令，好似打 slash-loop 咁，由一個人、甚至係另一個系統，打一條指令就即時啟動個循環。舉個例子，諗像有個觸發器設定成一個每晚嘅排程，每日凌晨兩點靜靜雞無人看管咁啟動個循環。所有呢啲做法嘅共通之處，就係「幾時開始」嘅決定係預先編寫好嘅，而唔係由一個人現場去做。

跟住落嚟就係 Worktrees，即係工作樹，同隔離。

呢個組成部分讓每一次代理運行都喺自己一個封閉嘅工作空間裏面進行，咁一次運行所做嘅改動就唔會踩到另一次運行緊嘅改動。點解咁緊要呢？一個循環可能會運行好多次迭代，有時仲會同時運行幾個代理。冇咗隔離，兩次運行同一時間改同一批檔案，就會破壞對方嘅工作；而一次失敗嘅運行更加可能會用一個冇得復原嘅方式，搞壞一個共用嘅工作空間。隔離可以防止啲代理互相踩踏，亦都令每一次運行都安全又可逆轉。如果一次迭代出咗錯，你只需要把佢嗰個被隔離嘅工作空間掉咗佢，其他嘢一概唔受影響。

呢方面嘅標準工具就係 git worktrees 同埋獨立分支。用 worktrees，每一次代理運行都會由同一個儲存庫簽出佢自己嘅工作目錄，咁佢嘅改動就同其他所有運行嘅改動實質分開。用獨立分支，每一次運行都會提交到佢自己嘅分支，咁佢嘅歷史就被隔離開,可以獨立咁去審閱、合併或者掉棄。諗像一下，畀第四十二次迭代有佢自己嘅目錄同自己嘅分支,嗰個代理喺度完全私密咁工作，唔受任何其他運行騷擾。隔離就係把「好多次運行」由一個負累變成一項優勢嘅關鍵。啲運行可以並行進行，亦都可以無害咁失敗。

而家講到呢部機器嘅心臟：Generator Agent，即係生成代理。

生成代理就係嗰個產出循環實際成果嘅 AI 編碼代理。啲程式碼、啲修改、啲改動。佢就係機器裏面嘅工人。呢個組成部分做嘅，正正係一個人本來要逐個回合親手去做嘅實務工作。呢度有一個重點，值得你記住：循環唔等於代理。生成代理只係循環嘅其中一個部分，負責生成成果嗰個部分。本章入面其他所有嘢之所以存在，都係為咗決定生成代理應該做乜、保護佢嘅安全、檢查佢嘅工作，以及決定要唔要再運行佢一次。

一個生成代理係由一個現成嘅 AI 編碼代理去填補嘅。佢可以係好似 Claude Code 嘅工具，作為嗰個讀取任務、再修改程式碼庫嘅代理。又或者係一個 Grok 代理，喺循環裏面扮演完全一樣嘅生成角色。諗像一下，為一次迭代去調用嗰個代理：佢讀取當前嘅任務，然後喺佢自己被隔離嘅工作空間裏面產出佢嘅修改。循環唔理邊個代理去填呢個位，只係要有樣嘢負責做生成嘅工作。代理係可以互換嘅。但個角色係固定嘅。

工作做完之後，就需要有人去評斷佢。呢個就係 Evaluator，即係評估器，有時又叫評論者或者驗證者。

評估器係嗰個拿生成代理嘅成果去對照某啲準則、再決定佢係咪夠好嘅組成部分。佢就係嗰個獨立嘅裁判。喺循環凌駕於一次純粹嘅代理運行之上嘅眾多優點當中，呢樣係最重要嘅其中之一。喺一次單一嘅代理對話入面，個代理同時係工人,亦係佢自己工作嘅唯一裁判。循環就加入咗一個外部嘅評估器，咁「完成」就係經過驗證,而唔係淨係靠估。評估器就係保護循環免於滿有信心咁產出錯誤又或者低質素成果嘅關鍵，因為喺生成者以外有樣嘢檢查過之前，冇任何嘢會被接納。

評估可以有幾種形式，而且通常會合併咁用。你可以有一個獨立嘅代理去跑測試，執行專案嘅測試套件，把失敗當成拒絕。你可以有一個程式碼審閱或者評論代理，即係第二個代理去讀啲改動，就質素、正確性同風格去審閱佢。又或者你可以有一個評分準則計分器，按一份明確嘅評分準則去為成果打分，再把個分數同一個門檻比較。諗像一個以測試為本嘅評估器，當佢係一道閘門：如果測試通過，呢次迭代就被接納;如果失敗，啲錯誤就會回饋返去，個循環再試多次。無論佢係乜嘢形式，評估器嘅工作都一樣。由外面去評斷個工作，再交返一個循環可以據之行動嘅裁決。

跟住係 State，又或者叫 Memory，即係狀態、記憶。

呢個組成部分喺任何單一代理嘅脈絡之外，把循環嘅進度保存落嚟，咁資訊就可以由一次迭代延續到下一次。一次單一嘅代理運行,淨係記得佢脈絡視窗裏面嘅嘢，而嗰份記憶一到運行結束就會即刻消失。一個循環要喺好多次運行之間做出真正、累積得到嘅進度，就需要一份比任何一次運行都更長壽嘅記憶。一份記錄,記住做咗乜、仲剩低乜、學到咗乜。狀態同記憶就係讓第四十二次迭代可以建基於第四十一次所完成嘅成果之上、而唔使由零開始嘅關鍵。

狀態通常會擺喺耐用、又可以查閱嘅地方。佢可以係一個待辦檔案,記錄剩低嘅工作，每次迭代都更新。佢可以係一張草稿紙,讓循環喺一次次運行之間記低啲發現同決定。佢可以係一塊喺好似 Linear 又或者 Notion 嘅工具裏面嘅專案板,追蹤每一個工作單位嘅狀態。又或者,佢可以淨係磁碟上嘅純檔案,記低啲中間結果。諗像一個簡單嘅待辦檔案，當中有一項喺第四十一次迭代被剔咗做「完成」，而下一項未剔嘅就準備好交畀第四十二次迭代。而正因為狀態係住喺代理之外，佢亦都係樣你自己可以讀嘅嘢，等你睇得清楚循環究竟去到邊一步。

接住就係 Skills，同 Knowledge，即係技能同知識。

呢個組成部分就係一個循環提供畀佢啲代理嘅、已成文兼可重用嘅指引。專案嘅規則、慣例同模式，寫低喺一個代理真係用得到嘅地方。佢解決緊嘅問題係咁嘅。一個唔識你專案標準嘅代理，會一次又一次咁，前後不一致咁重新發明嗰啲標準。技能同知識把嗰份指引一次過記低，咁每一次迭代都跟同一套規則，而唔使靠一個人喺每一條提示裏面都重複講「用我哋嘅命名慣例」，又或者「永遠都要寫測試」。呢樣就係一個循環點樣保持同你團隊實際工作方式一致嘅關鍵。

呢份指引通常會以代理會讀嘅檔案形式儲存。佢可以係一個技能檔案,描述代理應該跟嘅可重用程序。佢可以係一個代理指引檔案,話畀啲代理知喺呢個特定嘅儲存庫裏面點樣運作。又或者佢可以係啲成文嘅編碼標準,代理要遵從。諗像一個簡短嘅指引檔案，叫個代理跟返現有嘅模組佈局、為每一個新函式寫一個單元測試,以及優先用專案嘅日誌輔助工具、唔好用淨係 raw print。技能同知識,就係「一個靠估你慣例嘅代理」同「一個把慣例擺正喺面前嘅代理」之間嘅分別。

而家我哋去到 Connectors，即係連接器。

連接器就係一組整合,讓循環可以同外部系統溝通,即係你團隊已經喺度用緊、用嚟追蹤同交付工作嘅工具。一個循環唔係喺真空裏面運作嘅。佢要喺某處讀取工作、喺某處寫低結果，亦要話畀人知發生緊乜事。連接器就畀咗循環喺外面世界嘅一對手同一把聲。佢哋讓循環可以由一個追蹤系統拉一項任務出嚟、開立一個拉取請求,又或者貼一則更新，咁個循環就成為咗真實工作流程嘅一部分，而唔係一座孤島。

常見嘅連接器包括一個好似 GitHub 嘅原始碼控制主機,用嚟讀程式碼同開立拉取請求。亦包括一個議題追蹤系統,用嚟拉出下一個工作單位同更新佢嘅狀態。仲包括一個好似 Slack 嘅聊天平台,用嚟公布進度,又或者喺需要人手關注嗰陣升級上報。諗像一個循環同時接駁住呢三樣嘢：透過 GitHub 讀程式碼同開立拉取請求、透過議題追蹤系統拉任務同更新佢哋嘅狀態，以及把進度貼去一個聊天頻道。連接器就係令一個循環成為你現有系統嘅參與者、而唔係一個封閉實驗嘅關鍵。

最後,就係 Stopping Condition，即係停止條件。

停止條件就係嗰條規則，話畀循環知幾時要停止迭代，又或者幾時應該升級交畀人、而唔係繼續落去。一個循環會不斷重複，而冇明確出口嘅重複係好危險嘅。佢可以永無止境咁轉、燒晒你嘅預算，又產唔出任何有用嘅嘢。停止條件畀咗循環一個清晰嘅「完成」定義，又或者「放棄、求助」嘅定義，咁佢就係有意識咁終止，而唔係無了期咁運行落去。佢就係防止一個循環變成一個失控程序嘅組成部分。

一個停止條件通常係一條或者多條明確嘅規則。一條常見嘅規則就係測試通過：一旦評估器嗰個測試套件返綠燈，目標就達成，個循環就停。另一條係評分準則嘅分數去到某個門檻，譬如話一去到至少十分之九分就停。再有一條就係安全限制，好似一個最大迭代次數,又或者一個開支預算。譬如話跑足二十次迭代之後就停，又或者一旦成本預算用晒就停，並且升級交畀人、而唔係繼續落去。諗像一個合併嘅檢查，每次迭代都跑：如果測試通過，佢就停，因為目標達成;如果已經去到二十次迭代，佢就停,然後升級交畀人;如果超咗預算，佢都係停埋,一樣升級上報。第一條規則定義成功。其餘嘅就定義安全。佢哋加埋一齊，就保證個圓圈會停得返落嚟。

咁呢八個係點樣砌埋一齊呢？按次序去讀，佢哋就描繪出一個工作單位流經循環嘅路徑。觸發器啟動一次迭代。工作樹同隔離畀嗰次迭代一個安全嘅地方去運行。生成代理喺嗰度做工作，由技能同知識去引導，再透過連接器去接觸外面嘅世界。評估器評斷個結果。狀態同記憶記低發生咗乜，咁下一次迭代就可以建基於佢之上。而停止條件就決定個循環係再轉多一次，定係收工。

唔係每一個循環都需要齊晒所有組成部分。遲啲，我哋會睇一個極簡嘅循環，當中只係得佢哋其中幾個。但呢八個加埋一齊，就構成咗部件嘅完整詞彙。識得每一個係用嚟做乜，就係讓你喺設計自己嘅循環嗰陣，能夠決定邊個要納入、邊個要合併、邊個要剔走嘅關鍵。喺下一章，我哋會把呢啲部件郁起上嚟，追蹤佢哋喺循環嘅生命週期裏面係點樣互相配合。

等我哋回顧返講過嘅嘢。一個循環由八個組成部分裝配而成，而每一個都接手咗一項人喺人手提示時所做嘅判斷或者工作。觸發器決定工作喺幾時、以乜嘢方式進入循環，無論係透過一個每晚嘅排程、一個綁住新議題嘅 webhook，定係一條簡單嘅指令。工作樹同隔離令每一次代理運行都保持分開又可逆轉，咁啲運行就唔會互相踩踏，用嘅係好似 git worktrees 同獨立分支嘅工具。生成代理做嗰份產出程式碼同修改嘅實際工作，由好似 Claude Code 又或者一個 Grok 代理嘅嘢去填補;記住，循環唔等於代理，代理只係佢嘅其中一個部分。評估器由外面去檢查成果，咁「完成」就係經過驗證,而唔係靠估，無論係透過一個獨立嘅代理跑測試、一個程式碼審閱代理，定係一個評分準則計分器。狀態同記憶喺任何代理嘅脈絡之外把進度保存落嚟，咁迭代就可以累積得到，無論係喺一個待辦檔案、一張草稿紙、一塊專案板，定係磁碟上嘅檔案。技能同知識把專案嘅規則同模式記成文，咁每一次迭代都跟住佢哋,可以透過一個技能檔案、一個代理指引檔案，又或者啲成文嘅編碼標準。連接器把循環同好似 GitHub、一個議題追蹤系統,又或者 Slack 嘅外部工具整合埋一齊。而停止條件就話畀循環知幾時要停或者升級，無論係測試通過、評分準則分數去到門檻，定係撞到一個最大迭代次數又或者預算上限。佢哋八個加埋一齊，就描繪出一個工作單位由觸發到終止嘅全程。唔係每一個循環都需要齊晒八個，但識得每一個嘅用途，就係讓你能夠有意識咁去設計一個循環嘅關鍵。
