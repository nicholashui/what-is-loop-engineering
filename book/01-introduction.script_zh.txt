在这一章里，你会了解到这本书想教你什么、它是写给哪些人看的，以及你读完之后能够达到的具体学习目标。你也会第一次用最浅显的方式，认识一下什么是 Loop（迴圈），它和你亲手一句一句去 prompt 一个 AI coding agent 有什么区别，以及后面几章会逐一深入拆解的核心组件。

那我们就从"为什么会有这本书"讲起。大部分工程师接触 AI coding agent 的方式都差不多：一个对话框、一句 prompt、一个回应，接着再来一句 prompt。你始终坐在驾驶座上，一个回合一个回合地去操控这个 agent。这样是可行的，但它会给你能够完成的事情设下一个很硬的上限，因为你的注意力会变成瓶颈。每往前一步，都要你回头看输出、判断哪里不对，再打出下一句指令。

Loop Engineering（迴圈工程）就是一门专门去拆掉这个瓶颈的学问。你不再是一个回合一个回合地手动去 prompt 这个 agent，而是设计一个小小的控制系统，也就是一个 Loop，由它来帮你 prompt 这个 agent。这个 Loop 会自己发掘要做的工作、在一个隔离的工作区里运行 agent、检查结果、把问题反馈回去、记住发生过的事情，再自己决定是继续做下去还是停下来。你的工作就从打 prompt，转移到设计那个负责 prompt 的系统。

这本书会教你怎么去思考、去推敲，以及去建构这个系统。它假设你已经在用户的层面上用着 AI coding agent，然后从这一点出发，带你去设计一些你可以放心交托给它做长时间、反复工作的自主 Loop。

那么，你读完之后究竟实际上会做些什么呢？每一个目标都由一章或者几章具体去交付，所以你随时都可以把一个目标追溯回达成它的内容。

第一，你会懂得解释什么是 Loop，以及它和手动 prompt 有什么区别。这个由第三章交付，而本章稍后也会先预览一下。

第二，你会懂得把 Loop Engineering 放回它的脉络里，把它和 ReAct 模式联系起来。ReAct 就是 Reason 加 Act，也就是推理加行动。你会明白这个领域是怎样从 prompt engineering（提示工程），经过 context engineering（脉络工程），再到 agent harness engineering（代理框架工程），最后发展到 Loop Engineering，以及一个 Loop 和 ReAct 模式之间的关系。这部分在第四章讲。

第三，你会懂得讲清楚这个名词的起源，以及它的前身，也就是叫做 Ralph Loop 的技巧。你会知道这个名词从哪里来、是谁塑造了它，以及那个简单的 Ralph Loop 技巧，是怎样演变成可以用在生产环境里的 Loop Engineering。这部分在第五章。

第四，你会懂得辨认和解释每一个 Loop 组件的用途。你会懂得描述全部八个基本构件，也就是自动化和触发器、隔离、generator agent（生成代理）、evaluator（评估器）、状态和记忆、技能和知识、连接器，以及停止条件，而且每一个都举得出一个具体例子。这部分由第六章交付。

第五，你会懂得从头到尾追踪整个 Loop 的生命周期，以及它的决策点，沿着一个有序的流程，从触发工作一路走到决定是继续、是衍生一个 sub-agent（子代理），还是停下来。这就是第七章。

第六，你会懂得建构一个最简单的 Loop，再把它演化成一个可以用在生产环境里的版本。你会从一个简单的 bash 式迴圈开始，逐步把它养大成一个有隔离、有独立 evaluator、有持久化状态，以及有明确停止条件的系统。这就是第八章的实战部分。

第七，你会懂得判断什么时候应该用 Loop，并且权衡它的好处。你会认得出那些真正值得用 Loop 的工作类别，比如重构、功能实现和调试，而且你会讲得出为什么。这就是第九章。

第八，你会懂得预先估到 Loop 的风险，并且采取具体的缓解措施。你会学会怎样防范失控的成本和低质量的输出，也明白在哪些地方，人的工程判断力依然不可或缺。这就是第十章，而结尾的综合总结就在第十一章。

如果这八样事情你都做得到，你就已经从"操作一个 AI coding agent"，提升到"工程设计那个帮你操作它的系统"。

在我们走得更深入之前，我们先建立一个初步的图像。究竟什么才算是一个 Loop 呢？后面几章会很仔细地铺陈每一个概念，但从一开始就带着一个可用的心智模型，会很有帮助。有三个问题贯穿整本书，而这一节就会每一个都答得恰到好处、够你定位。

第一，什么是 Loop。一个 Loop 是一个编排系统，是少量的软件和配置，它会驱动一个 AI coding agent，经历一轮又一轮的工作周期，而不需要人在每一步去 prompt。它不是 agent 本身。agent 只是 Loop 的其中一部分。Loop 就是周围那套机制，负责决定 agent 应该做什么、运行它、判断结果，再不断重复，直到工作达到一个既定目标，或者撞到一个既定上限为止。这个名字很直接：这个系统就是在兜圈子。它产出输出、评估那个输出，再用那个评估去驱动下一次迭代，一次又一次，就好像你自己亲自监督这个 agent 一样，区别只在于：这次是由 Loop 来做监督。

第二，它和手动 prompt 有什么区别。在手动、逐个回合去 prompt 的情况下，控制系统就是人。你看每一个回应、把目标记在脑子里、判断输出够不够好，再拼出下一句 prompt。你一离开，进度就立刻停下来。而用一个 Loop，这个控制系统只设计一次，接着就自己运行。你以前在脑子里做的那些决定，比如"这个够不够好""接下来应该发生什么""我们做完了没有"，全都变成明确、写死的规则：一个检查输出的 evaluator、一个记住进度的状态文件，以及一个结束整个运行的停止条件。实践者的角色就从操作员，转移到做系统架构师。你设计这个 Loop，Loop 负责 prompt。

第三，核心组件。每一个 Loop，从最简单的那个到最精密的那个，都是由同一小组基本构件拼成的。你会在第六章里逐一深入研究，但在这里先介绍一下这些角色，好让本书其余部分读起来成为一个连贯的整体。有自动化，又叫触发器，它决定工作什么时候、用什么方式进入这个 Loop，比如一个排程、一个事件，像是新开了一张 issue，又或者是你亲手运行的一个指令。有隔离，通常是通过 worktree 来做，它让每一次 agent 运行都各自分开、可以还原，这样错误就不会污染到你的主工作区。有 generator agent，也就是那个真正产出输出的 AI coding agent。有 evaluator，它通过运行测试、审查代码，或者根据一个评分准则去打分，来核实输出。有状态，又叫记忆，它把进度跨越多次迭代地保存下来，让 Loop 记得做了些什么。有技能和知识，它提供一些可重用的指引和标准，给 agent 去遵循。有连接器，它把 Loop 和外部系统整合起来，比如版本控制、issue 跟踪工具，以及聊天平台。最后就是停止条件，也就是结束 Loop 的规则，比如测试全部通过、达到某个质量门槛，又或者撞到一个预算上限。

这个图像，暂时松松地把握住就可以了。重点很简单：一个 Loop 是由多个部件拼成的；那些部件取代了你本来要亲手做的判断；而把它们设计得好，正正就是这本书要教你的手艺。

我们也来看一下本书其余部分是怎样编排的。各个章节跟着一条刻意安排的轨迹。头几章建立"是什么"和"为什么"：第二章讲本书是写给哪些人看的、第三章给一个精准的学科定义、第四章讲它的脉络和思想根源，而第五章就讲它的起源。中间几章拆解各个部件以及它们之间怎样互动：第六章讲那八个组件，第七章讲从头到尾的生命周期。实战那一章教你怎样建构一个出来，就是第八章。结尾几章涵盖什么时候应该用 Loop、它的好处和风险，以及一个最终的综合总结，分布在第九章到第十一章。最后还有词汇表和参考资料一节，把本书补得完整。你可以一路顺着读下去，当作一门完整的课程，又或者如果你喜欢，也可以直接跳到某个特定主题。

那我们就总结一下主要概念。Loop Engineering 是一种实践，是去设计一个控制系统，也就是一个 Loop，由它自主地去 prompt 和编排 AI coding agent，而不是一个回合一个回合地手动去 prompt 它们。一个 Loop 是那个编排系统，不是 agent；agent 只是它里面的其中一个组件。它和手动 prompt 的关键区别，就在于是谁控制整个周期：用一个 Loop，人本来要亲手做的判断就会变成明确、写死的规则，这样就把你的角色从操作员转移到做系统架构师。每一个 Loop 都是由同一组核心组件拼成的：自动化和触发器、隔离、一个 generator agent、一个 evaluator、状态和记忆、技能和知识、连接器，以及一个停止条件。而这本书有八个明确的学习目标，每一个都由特定章节去交付，带你从理解一个 Loop，一路走到建构和判断一个 Loop。
