在这一章，我们假设你已经有一个可以描述和运行 loop 的库，而我们会聚焦在：怎么在这个库增长、用的人越来越多的同时，让它保持优质。我们会讲一个让团队可以安全地加 loop 的贡献流程、让劣质产出进不来的品质标准和审查规范、怎么为条目做版本管理和弃用、哪些度量指标可以告诉你一个 loop 值不值得保留，以及当众多自主 loop 在一个团队之间运行时随之而来的治理和安全顾虑。读完之后，你就有能力把一个 loop 库当作共享基础设施来运营，而不只是一个个人的脚本文件夹。

那就从一个心态转变讲起：一个库是一件产品，不是一个文件夹。几个 loop 放在一个目录里面，是一个方便。但一个让整个团队信任、并针对真实系统运行的库，就是基础设施，而基础设施需要一个负责人、一个品质标准、一个安全改动的方法，以及一个量度它有没有值回票价的方法。一个成功的库的失败模式，不是 loop 太少。而是劣质的太多，搞到没人分得清哪个条目可以信。策展，就是防止这件事的工作。

这一章假设你已经有上一章那套机制，schema、储存、runner、CI 验证器，并在它们周围加上人的一层：loop 怎么进来、它们怎么保持优质，以及整件事怎么扩展到一个作者以上。

首先，贡献流程。你在一个 loop 内部所套用那份纪律，提出一个改动、独立验证它、只有在通过时才接受，同样适用于对这个库本身的改动。把一个新的或者修改过的 loop 条目，当作任何其他程式码改动来处理：一个 pull request、一次审查，以及一道自动关卡。

这里有一个从一位贡献者扩展到多位的流程。第一，透过 pull request 提出。一位贡献者在一个分支上新增或编辑一个条目文件，并开出一个 PR。没有任何东西直接合并入主库。第二，自动关卡先运行。CI 运行那个验证器：schema 合规、一个存在的停止条件，以及一个存在的 prompt 文件。一个过不了这些关卡的 PR，永远到不了人类审查者那里。便宜的检查先行。第三，一位维护者对照品质标准审查。一个人类去检查自动化检查不了的东西：这个目标是不是真的可被检查、这个评估器是不是独立于生成器、那些护栏是不是诚实、标注是不是正确。第四，接受之前先做试运行。对于任何非琐碎的 loop，审查者透过 runner 把它针对一个安全目标运行一次，确认它的行为跟所描述一致，尤其是它真的会停。一个从未被执行过的 loop，是一个假设，不是一个库条目。第五，合并把它晋升为 stable，并重新生成那个索引。

这个原则是递归的，而且值得讲白：一个 loop 库，本身就是由一个 loop 来维护。提出、独立验证、只有在通过时才接受。魔鬼代言人和建造者审查者形态，套用在库条目身上，跟套用在那些条目所作用的程式码身上，是一样多的。

现在讲品质标准。自动验证证明一个条目形式正确。品质标准就证明它优质。每一个获准入库的条目，都应该通过一份书面、可见的清单，存放在储存库里面，让贡献者在开 PR 之前可以自我检查。这个目标是外部可被检查的，措辞为一个机器或者一个独立审查者可以确认的成果，而不是"改善 X"。这个评估器独立于生成器，于是"完成"永远不是靠 agent 自己的报告。停止条件是明确而分层的：一个成功出口加上至少一个预算上限。一个无边界的条目即时不过关。隔离和一条人类升级路径都有定义。波及范围有记录。而标注和来源是正确的。

把这份清单做成一个 PR 模板，这样每一个贡献都会对照同一套准则去审查。一个过不了标准的 loop 不是永远被拒绝。它以 experimental 或者 draft 的身份获准，清楚标明，让用户知道它还没赢得完全的信任。

接着，版本管理和弃用。Loop 会变：一个 prompt 改善了、一个评估器收紧了、一个更好的做法取代了一个旧的。因为每一个条目都带着一个语义化的 version 和一个 maturity 字段，这个库可以在不搞垮依赖它的人的前提下演化。为每一个有意义的改动做版本：当行为改变时提升这个 version，这样钉住某个版本的调用者得到可重现性，而追踪最新版的调用者得到改善。用成熟度阶梯：新条目以 draft 或者 experimental 进入，一经证实就晋升为 stable，被取代时就以 deprecated 终结。而弃用，不要删除。当一个 loop 被取代时，把它标为 deprecated、指向它的后继者，并保留它一个发行周期。悄悄地删除一个条目，会悄悄地搞垮任何引用了它的东西。成熟度阶梯，正好就是让一个库能够欢迎新 loop 而又不对信任掉以轻心的东西。

现在，度量一个 loop 值不值得它那个位置。一个库里面的 loop，是一个"它值得运行"的声称。一旦这些 loop 定期运行，你就可以用证据去取代那个声称。因为这个 runner 本身已经驱动每一个 loop，所以你只要为它加装一次仪表，就免费得到整个库的遥测。追踪接受率：在到达一个停止条件的运行之中，有多少是以成功结束，而不是撞到预算上限或者升级。追踪到达成功所需的迭代次数和成本：成本随时间上升，是一个漂移的早期警告。追踪升级率：一些升级是健康的，但持续升级就意味着这个 loop 范围定错了。追踪使用量：一个没人运行的条目是文件，不是基础设施，是一个弃用的候选。这些直接连回那些停止条件：那些界定一个 loop 行为的出口，同时也是你点算来判断它的事件。

然后就是规模化的治理和安全。一旦一个库被共享、而它的 loop 在很多储存库之间自主运行，那些风险就不再是一个工程师的问题，而变成一个组织的问题。一个出事的个人 loop 顶多让你要移除一个 worktree。而一个有广泛权限、出事的共享 loop，就可能影响到所有人。最重要的控制是以下几个。最小权限连接器：一个 loop 的连接器应该只授予它需要的存取，可能的话只读，并且不要把长期生产凭证焗死在条目里面。中央预算上限：在逐条目的成本上限之上，执行一个组织层级的上限，这样众多 loop 的总和就不会失控。一条审计轨迹：记录每一次运行，哪个条目、哪个版本、哪些输入、谁触发、它改了什么、它怎么停。高波及范围 loop 的人类审批关卡：任何碰到生产数据、公开文案或者不可逆状态的东西，都应该要求明确的人类签核。以及一个清晰的负责人：有一个人为品质标准、弃用决定和安全姿态负责。这些都不是新想法。它们就是之前那些缓解措施，从一个单一 loop 提升到一整队 loop。

最后，扩展这个库本身。当它增长到超过几打条目时，两股压力会出现：发现变得更难，而重复也会悄悄爬进来。倚靠形态，而非条目：当两个条目是同一种形态、只是目标不同的时候，宁可用一个参数化的条目，都不要用两个近乎重复的。这个库应该在参数和 prompt 里面增长，而不是在副本里面。针对蔓延做策展：定期对这个库做一次清理巡查，刚刚好，这件事本身也是一个 loop，去找出没人运行的条目、重复、失效引用和过时 prompt，并弃用或合并它们。以及按领域联邦化、共享 runner：大型组织可以容许各团队拥有自己的类别文件夹，同时共享一个 schema 和一个 runner。

终局是一个相对于它的有用程度而保持小巧的库：一套经过策展、可信、参数化、被度量的 loop，给整个团队去取用，而不是一堆不断增长的一次性脚本。这个就是本书从头到尾所主张那份减法纪律，套用在高一个层级之上。当"没有把一个 loop 放入这个库"会造成真正的重复的时候，才把它加入，在它值回那个位置那段时间保留它，并在它不再值得时退役它。

那让我们把这些关键想法收拢到一起。一个共享的 loop 库是基础设施，不是一个文件夹，而它的失败模式是劣质条目太多，所以策展是核心工作。对这个库的改动，跟着跟 loop 本身一样的 loop 纪律：透过 PR 提出、运行自动关卡、对照一份书面品质标准审查，并在合并到 stable 之前做试运行去确认这个 loop 真的会停。这个品质标准要求一个可被检查的目标、一个独立的评估器、明确而分层的停止条件、有定义的隔离和升级、一个有记录的波及范围，以及正确的标注。做版本和弃用，而非搞垮，用一条以搜索分面暴露出来的成熟度阶梯。用每一个 loop 自己的停止条件本身已经产生的事件去度量它，并退役那些在那里乱撞或者没人运行的 loop。治理把之前那些缓解措施扩展到一整队：最小权限连接器、组织层级的预算上限、一条审计轨迹、人类审批关卡，以及一个清晰可问责的负责人。而按形态扩展，而非按副本，让这个库相对于它的有用程度而保持小巧。
