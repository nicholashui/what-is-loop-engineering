在这一章里，我们要老老实实地聊一聊 Loop Engineering 的坏处。我们会看看一个 loop 在没人盯着的情况下运行时会出什么乱子，以及你可以怎样在每一种失败把你拖到亏钱之前，围着它加上限制。我们会深入讲两个风险。第一个是失控的 token 成本，也就是一个自主的 loop 悄无声息地把整个预算烧光。第二个是低质量输出（slop），也就是薄弱的验证让一个 loop 大量喷出又多又差的工作。对于每一个风险，我们都会把解决办法接回到你已经搭好的机制上，也就是前面几章讲过的护栏和 Evaluator。讲到最后，我们会说清楚一点：这些全都不会消除对工程判断力的需要。一个 loop 只是把你的判断力挪到了另一个位置，并没有让它退役。

我们先从一笔老实账开始。上一章讲了 loop 的好处。这一章就是那个必要的对照。让一个 loop 强大的所有东西——它没有你也能跑、不知疲倦地迭代、还自己提供下一个提示——恰恰就是当有东西出错时，让它变得危险的东西。一个一个回合给提示的人，在下一个回合就会注意到出了错，然后简单地停手不再打字。一个 loop 完全没有这种本能。让它在那儿自己跑，它会很有耐心地重复一个错误，就像它重复一次成功那样有耐心，而且全程都在花着真金白银。

所以这一章特意不是一场推销。Loop Engineering 值得你去学，但它不是魔法，过度吹捧只会害了你。我们要讲的两个风险，是你在实际操作里真真正正会撞到的。两者都不是叫你避开 loop 的理由。两者都是叫你要用前面几章描述过的护栏和 Evaluator 去搭建 loop 的理由。而这里有个真的挺好的消息：让一个 loop 达到生产级的同一套结构，也正是让这些风险变得可控的东西。这些解决办法不是什么新发明，它们就是你已经见过的组件，被指向了它们本来就是用来防范的那些失败模式。

第一个风险最直接，也最容易被人低估。一个自主的 loop 每一次迭代都要花钱，而一个没有上限的 loop 可以花出一个没有上限的数目。每一次调用 generator agent 都会消耗 token，而一个 loop，按定义来讲，就是一台在中间不用问你批准、就可以做很多次这类调用的机器。你搭建这个 loop 所追求的那种自主性，恰恰就是消除了那个天然刹车的东西，因为不再有一个人在那儿一个回合一个回合地看、去决定下一个回合究竟值不值得做。

这件事出错有几个典型的样式。有"卡住的 loop"，它永远都到不了目标，但也永远不知道自己到不了。它迭代、过不了 Evaluator、把那些反馈循环回去、再试一遍——如果没有东西去阻止它，就会无止境地试下去。每一圈都很便宜，但账单就是很多很多便宜的圈加在一起的总和。接着有"来回摆动的 loop"，是一个更隐蔽的版本。Agent 修好了问题 A，弄坏了 B。下一次迭代修好 B，又再弄坏了 A。这个 loop 每一轮都很有信心地看似有进展，但实际上哪儿都收敛不了，而且每一次都付足全费。最后还有"昂贵的单次尝试"，因为成本不只是关乎你迭代了多少次。一次尝试，如果往提示里塞了一大堆 context，或者一次又一次地调用一个昂贵的模型，可以在单单一圈里就爆掉整个预算。所以只限制迭代次数是不够的。

老实的说法是：这个风险是结构性的，不是偶发的。它不是一个只袭击写得差的 loop 的罕见 bug。它是任何一个缺乏明确限制的 loop 的默认行为。一个没有停止条件（Stopping Condition）的 loop，通常都不会完成。它根本就没有"完成"这个概念。

那你怎样去缓解失控的成本呢？答案就是你之前设计过的停止条件和护栏，要有纪律地去套用。失控这个风险，恰恰就是那些护栏存在的目的，所以防守之道就是刻意地去用它们，而不是把它们当成事后补救。你要设一个硬性的迭代上限，也就是限制圈数，这就保证了这个 loop 一定会终止，哪怕它永远成功不了、又永远不会彻底失败。光是这样，就已经把一个卡住的 loop，从一张开放无底的账单，变成一张有边界的账单。你还要设一个独立的成本上限，也就是一个以金额计的预算（预算），当累计花费超过某个门槛时就停下这个 loop，不管迭代了多少次。这就是去抓回那个迭代上限会漏掉的昂贵单次尝试。你真的两个上限都想要，因为每一个都补上另一个的盲点。除此之外，你要让"成功"那个停止条件是真实、可核查的，因为要停止花钱，最干净利落的办法就是真的把它完成。一个 Evaluator 可以客观确认的目标，比如测试全过了，或者达到了评分准则（评分准则）的门槛，就让这个 loop 在工作一做完那一刻就退出，而不是过了完成点还在那儿迭代。而当一个预算用完了、但又还没被接受的时候，这个 loop 应该升级上报（升级上报）给人去处理，而不是悄无声息地失败，或者更糟，被人盲目地重新启动。检测到重复的无进展——也就是那个来回摆动的 loop——并且早早升级上报，就省回你本来只用来确认这个 loop 卡住了的那笔预算。

这些限制是以数据的形式，存在于这个 loop 的设定里。实际来讲，意思就是你同时为迭代次数和花费金额设上限，你叫这个 loop 在测试一过的那一刻就停，你又接好升级上报的机制，让预算用完时会通知一个人，而当它连续三次见到同一个失败时，就早早升级上报，而不是硬着头皮继续磨。

有一个限制值得老实地讲。收得越紧的预算，就越会牺牲掉最初让这个 loop 吸引到你的那份自主性。一个设得太低的上限，会在这个 loop 真的完成那些难搞的工作之前就把它停掉。一个设得太高的上限，就打败了它自己的目的。没有一个放之四海皆准的正确数字。挑这个数字，是一个工程判断力的动作，而这恰恰就是整章都在铺垫的主题。

第二个风险更安静，但长远来讲更加蛀蚀。Slop 就是一个 loop 在迭代得快、但验证得弱的时候产生出来的东西。它是大量看起来像有进展、但其实站不住脚的输出。失控的成本会在账单上自己宣告出来，但 slop 就藏在那些看起来很合理的修改里，靠着纯粹的数量和信心过了 review，直到有人发现里面很多东西其实是微妙地错了、或者重复、或者跑题了。

Slop 是让自主性跑赢了验证的直接后果。记住"好处"那一章的论点：没有 Evaluator 的自主，只是让出错出得更快。一个 loop 是一个放大器。它会放大你指向它的任何一个流程。指它去一个强的验证流程，它就放大好的工作。指它去一个弱的，它就同样高效地放大 slop。危险的地方就在于：哪怕实质内容在退化，那个输出的形式还是很有说服力。

这件事有几种出现方式。有"很有信心但错了"的输出，就是 generator agent 报告说成功了、那个修改读起来很干净、又没有任何客观的东西去反驳它——因为从头到尾就根本没有咨询过任何客观的东西。那个只信着 agent 自己宣称"全部搞定"的最简 loop，就是典型的 slop 制造机。有"把数量当成了价值"，因为一个 loop 在同一段时间里，可以产出远远多过一个人的代码。没有一道质量关卡，那种产量就变成一种负担——更多要 review 的表面面积、更多让缺陷藏身的地方、更多看似合理的噪声盖过了信号。还有一样我们可以叫做"Goodhart 漂移"的东西，就是一个 loop 针对一个薄弱的代理指标去优化——一个肤浅的测试或者一份含糊的评分准则——然后学会了去满足那个代理指标，而不是去满足真正的意图。指标变绿灯，但实际的目标就悄无声息地退走了。

缓解 slop 的办法就是 Evaluator，而且特别是那条原则：Evaluator 必须独立于 generator agent。当验证薄弱的时候，slop 就是填补那个真空的东西，而一个强而独立的 Evaluator，就是你去消除那个真空的办法。所以你要以强而客观的测试打头阵。一道 agent 怎么讲都过不了的客观关卡——一个真实的测试套件、一个类型检查器、一次构建、一次 schema 验证——就是抵抗 slop 的脊梁。它返回一个毫不含糊的通过或者失败，是再多有信心的花言巧语都伪造不了的。测试越强、越有意义，slop 能溜过去的空间就越小。接着你加一个 critic 或者 rubric agent，去管那些测试漏掉的东西。客观检查确认了行为，但它们对于可读性、安全性，或者那个修改究竟有没有真的符合意图，就讲不了多少。一个第二个 agent，对着一份明确的、有版本的评分准则去给工作打分，就补上那个缺口。你要让评估独立于生成，而这是最最重要的一个特性。那个去裁判工作的东西，不能和那个产生工作的东西是同一个，而且也不能只信生产者一面之词。独立的评估，就是把"agent 说它搞定了"变成"一个外部的检查确认了它搞定了"的东西，而这恰恰就是 slop 和经过验证的进展之间的区别。最后，你要让验证是一道硬性的关卡，而不是一个建议。你只会在一次尝试通过了客观关卡、又越过了评分准则门槛时才接受它。一个只出警告、但又照样放未验证的工作过去的检查，根本不是防范 slop 的防线。它只是加了多点日志的 slop。

那条接受规则，就是之前讲过的那个独立的双重关卡检查，而它值得我们当它做反 slop 机制再看多一次。简单来讲，一份工作只有在两个条件同时成立时才会被接受。第一，客观测试要过，一个伪造不了的通过或者失败。第二，独立 critic 的评分要越过一个明确的门槛。只有当两者都满足时，这份工作才会被提拔。其余所有东西都丢弃。这个 agent 没办法靠自己认证自己的方式，过到任何一道关卡。

这里老实的限制就是：一个 Evaluator 的好坏，只取决于你给它的准则。一个 loop 没办法验证你从来都没指明过的质量。如果你的测试肤浅，或者你那份评分准则含糊，这个 Evaluator 就会忠忠实实地放 slop 过去，而这个 loop 又会忠忠实实地产出更多 slop。这不是一个你可以靠多加几个 agent 就自动化解决的缺陷。它是对你——对人——的一个要求：要把"好"的意思界定到够清楚，清楚到一台机器都可以去执行它。而这就把我们带到那个垫在两个风险底下的限制。

如果一章讲风险的内容，结尾去到暗示说只要设定对了就会让 loop 自动安全，那就不老实了。它不会。Loop Engineering 最深层的限制，也是最容易被人忽略的。一个 loop 并没有消除对工程判断力的需要。它只是把它挪了位置。

这一章里每一个缓解办法，最终都建立在一个没有任何 loop 可以替你做的人为决定之上。有目标。一个 loop 会追求你给它的目标，一字不差地照你那个说法去追。决定什么值得去搭建，并且把它陈述成一个清晰、可核查的结果，是人的判断。一个 loop 会用和追一个好目标一模一样的不知疲倦的劲头，去追一个挑得差的目标。有评估准则。Evaluator 会执行你界定的标准，不多不少。决定要测什么、测试要多强、那份评分准则奖励什么、以及把它的门槛设在哪里，是人的判断。Slop，讲到底，就是评估准则弱到抓不住它的症状。又有护栏。那些预算、那些迭代上限、那些隔离边界、以及那条升级上报的路径，全都是你挑的数值。这个任务值多少钱？试多少次之后一个人才应该来看一下？什么是永远都不可以进到 main 分支的？那些都是 loop 自己原创不了的判断抉择。

这样去看，这一章的两个风险其实是同一个教训，从两个方向去看。失控的成本，就是护栏体现了太少判断时发生的事。Slop，就是评估准则体现了太少判断时发生的事。在两种情况里，这个 loop 都是精准地做了人叫它做的事。缺口是在那个"叫"的过程里。

这就是为什么从操作员转做架构师，是一次判断力的挪位，而不是消除。做操作员的时候，你应用判断力一百次，一次一句提示，而且你可以在半路抓到一个问题。做架构师的时候，你应用判断力的次数少了，但杠杆大得多，而且关键是在前头，在这个 loop 开始没人盯着地跑之前。每一个决定的风险都升高了，恰恰是因为你已经不再在那儿一个回合一个回合地更正它。一个 loop 把你的判断力，浓缩成一手那几个高杠杆的抉择，然后忠忠实实地、而且大规模地去执行它们。这就是它的力量和它的风险，一句话讲完：它会精准地做你设计它去做的事，不管那是不是你真正想要的东西。

接下来该有的实际姿态，不是吹捧、也不是恐惧。为对的工作去搭建 loop，用这本书描述过的护栏和 Evaluator 去界定它们，并且把你花在设计目标、准则和护栏上面的时间，当做是真正的工程，因为那里就是你的判断力现在所在的地方。

那我们就归纳一下我们讲过的东西。Loop Engineering 很强大，但它不是魔法，而一笔老实账会直白地讲出它的风险，而不是过度吹捧。让一个 loop 达到生产级的同一套结构，就是让它的风险变得可控的东西，因为那些缓解办法都是你已经见过的组件，被指向了它们本来就是用来防范的那些失败。

第一个风险是失控的 token 成本。因为一个 loop 在中间没人刹车的情况下，做很多次自主的 agent 调用，所以一个没有上限的 loop 可以无止境地花费，无论是通过一个卡住的 loop、一个来回摆动的 loop，还是一次昂贵的单次尝试。那是任何一个没有明确限制的 loop 的默认行为，而不是一个罕见的 bug。你用停止条件和护栏去缓解成本。同时设一个硬性的迭代上限和一个独立的成本上限，让成功条件是真实、可核查的，好让这个 loop 真的可以完成，并且在预算用尽或者重复无进展时升级上报给人，而不是硬着头皮继续磨。

第二个风险是 slop。当迭代跑赢了验证的时候，一个 loop 就会产出大量看似合理、但其实低质量的输出——很有信心但错了的修改、把数量当成了价值、以及对着薄弱代理指标的漂移。一个 loop 会放大你指向它的任何一个流程，好的也行，差的也行。你用一个强而独立的 Evaluator 去缓解 slop。以 agent 伪造不了的客观测试打头阵，加一个 critic 或者 rubric agent 去管那些测试漏掉的东西，让 Evaluator 保持独立于 generator，并且让验证是一道硬性的接受关卡，而不是一个警告。还有，记住，一个 Evaluator 的好坏，只取决于你给它的准则。

最重要的是，工程判断力不会消失。它只是挪了位置。一个 loop 会忠忠实实地追求你设计的目标、执行你设计的评估准则、以及尊重你设计的护栏。它原创不了这些决定里的任何一个。失控的成本和 slop，两者都是人的判断太少的症状——在护栏里，也在准则里。所以负责任的姿态，不是吹捧、也不是恐惧。为对的工作去搭建 loop，用护栏和一个 Evaluator 去界定它们，并且把设计目标、准则和护栏当做是真正的工程，因为那里就是你的判断力现在所在的地方，在更高的风险和更大的杠杆之上。
