最後,只喺你嘅任務真係要求嗰陣,先至加埋其餘嘅元件。自動化或者觸發器,例如排程作業、webhook、又或者斜線指令;連接器,接去你嘅問題追蹤器、即時通訊、或者原始碼控制主機;同埋技能與知識,例如一個技能檔案、又或者你嘅程式編寫規範,呢啲嘢就係將一個你要親手去跑嘅 Loop,變成一個自己會跑嘅 Loop。佢哋對一個無人睇守嘅 Loop 嚟講係必需嘅,但係佢哋對正確性嚟講係毫無貢獻,所以就將佢哋押後,直到嗰個「生成同評估」嘅核心已經值得信任為止。

呢度嘅紀律係減法,唔係加法。你每加一個元件,都係一個你而家要去理解同維護嘅控制面,所以喺每一步啱啱好嘅問題係:省略呢樣嘢,會唔會喺我嘅任務上面造成一個具體嘅失敗?如果一個喺本機上面對住一個綠色測試套件跑嘅 Loop 已經做到件工,咁你就唔需要一個 webhook 觸發器或者一個即時通訊連接器。只係喺你嘅任務真係推你去到嗰個地步嘅時候,先至向住個生產例子去成長。

而家我哋深入啲講吓目標、評估準則、同埋護欄嘅模式。有三個設計決定,比任何其他嘢都更加決定一個 Loop 嘅成敗,就係你畀佢嘅目標、決定佢嘅工作幾時先算可接受嘅評估準則、同埋約束佢行為嘅護欄。啲元件淨係喉管。呢三樣嘢先至係工程判斷所在嘅地方。我哋就行一行嗰啲值得首先去攞嚟用嘅模式。

先由設計目標開始。一個 Loop 係喺無人睇守之下跑,所以佢嘅目標必須清晰又可核查,要咁樣去措辭,令到個 agent 自己嘅樂觀以外嘅嘢,都可以判斷到佢究竟達到咗未。

第一個模式,係令個目標可以由外部驗證。揀一個佢嘅完成度可以由一部機去確認嘅目標。類似「令到測試套件乾乾淨淨咁退出,冇任何被略過嘅測試」係可核查嘅。類似「改善啲程式碼」就唔係。一個評估器核查唔到嘅目標,就係一個 Loop 完成唔到嘅目標。

第二個模式,係將範圍界定喺一個連貫嘅工作單元。一個由「重構支付模組,令到所有現有測試仍然通過」去駕馭嘅 Loop,會收斂。一個瞄準「現代化成個程式碼庫」嘅 Loop,就會四圍遊蕩。窄嘅目標,先至畀到回饋路徑喺每一次疊代都有啲具體嘅嘢去頂住。

第三個模式,係將目標寫成一個結果,而唔係一個程序。描述你想要嘅最終狀態,「個套件係綠嘅,而且個新端點成功咗」,而唔係去到嗰度嘅步驟。步驟由個 agent 揀,個 Loop 就驗證個結果。

一個咁樣寫嘅目標,可以直接擺入去設定檔嘅目標欄位。舉個例,一個好嘅目標可能係:「畀 orders 端點加分頁功能,令到佢接受 page 同 limit 參數、最多回傳所要求嘅項目數量、並且令現有測試保持綠色。」呢個係可核查嘅,因為現有同新嘅測試就係「做完」嘅客觀訊號。佢亦都係有界定嘅,因為佢係一個端點、一個行為,而唔係「改善 orders API」。

跟住,設計評估準則。評估器係個 Loop 對抗「好有自信但係做錯咗」嘅輸出嘅防線。一個值得信任嘅評估器,有一個界定性嘅特徵。佢係獨立於個生成器 agent 嘅。佢永遠唔會將個 agent 自己嘅報告當成成功嘅證據。好嘅評估通常會疊兩種檢查。

第一種,係一道難以作假嘅客觀閘。測試、型別檢查、linter、一個建置步驟、又或者 schema 驗證,任何會回傳一個明確嘅「通過或者失敗」、而個 agent 講都講唔甩嘅嘢。呢個係值得信任嘅評估嘅骨幹,所以由呢度開始。

第二種,係一道評論家或者評分標準閘,用嚟補返客觀檢查漏咗嘅嘢。一個測試套件確認到行為,但係佢對於可讀性、安全性、又或者個改動究竟有冇真係符合意圖,就講得好少。一個第二嘅 agent 根據一份寫低嘅評分標準去同件工評分,就補返呢個缺口。將評分標準保持明確,並且為佢做版本控制,咁個 Loop 要達到嘅標準就睇得到、又調得到。

兩道閘組合成單一條接受規則。只有喺客觀閘通過、而且評分標準過到佢個門檻嗰陣,先至接受。用淺白嘅講法,規則係咁運作嘅。佢首先檢查客觀閘,如果真正嘅測試套件冇通過,答案就係冇,拒絕。然後佢檢查質性閘,如果評論家嘅分數喺門檻之下,答案又係冇。只有當兩道閘都滿足嗰陣,佢先至回傳係,升格呢次嘗試,然後停。呢個係一個獨立、兩道閘嘅檢查,個 agent 根本冇得自我認證。

仲有兩個模式,令一個評估器喺實務上值得信任。將客觀閘喺個 agent 工作嗰個一模一樣嘅隔離工作空間裏面去跑,咁你評分嘅就正正係佢產出嗰樣嘢。並且要審慎咁設定評分標準門檻。太低,啲求其嘢就會穿過。太高,個 Loop 就會為咗追一個佢永遠都達唔到嘅分數,而燒晒佢成個預算。一個你守得住、又會隨住你學習而調整嘅門檻,好過一個求其定嘅門檻。

最後,設計護欄。護欄就係啲停止條件,加埋啲令一個無人睇守嘅 Loop 唔好造成破壞嘅隔離同升級規則。評估決定件工好唔好,而護欄就決定個 Loop 喺有嘢停低佢之前,可以行去到幾遠。攞呢啲模式嚟用啦。

永遠設多過一個預算。為疊代設上限,又為使費設上限。單係其中一個,都有個盲點。一個平但係卡住咗嘅 Loop,可以喺佢個疊代次數裏面打轉;而一個昂貴嘅單一嘗試,就可以一鋪過爆咗個美金預算。兩個夾埋,就將兩種失控模式都箍住。

隔離每一次嘗試,並且只喺成功嗰陣先合併。可逆性本身就係一道護欄。如果一個畀人拒絕嘅嘗試係住喺一個用完即棄嘅工作空間裏面,咁一次差嘅疊代,代價就只係刪咗嗰個工作空間,而唔係一條畀人搞污糟咗嘅主分支。

並且定義一條明確嘅路徑,去升級畀返一個人類。一個 Loop 應該知道幾時要停低嚟發問。當佢用盡咗預算都仲未畀人接受、當佢喺度震盪但係冇進步、又或者當佢觸發咗一條高風險嘅引線,佢就必須交畀返一個人,而唔係靜雞雞咁失敗,又或者繼續燒 token。

呢啲規則係住喺設定檔裏面,連同一個升級鈎子一齊。啲停止條件設定疊代上限,捉住嗰啲「卡住但係平」、喺度原地打轉嘅 Loop;設定成本上限,獨立於疊代次數咁捉住一個昂貴嘅 Loop;同埋設定成功出口,即係客觀同評分標準兩道閘都過晒嗰陣。升級設定就講明,當一個預算撞到咗但係仲未畀人接受嗰陣應該點做,唔好靜雞雞咁失敗,交畀返人,可能係開一個 issue,又或者出一個訊息去即時通訊。而如果同一個失敗連續重複咗某個既定次數,就升級。

留意吓呢度每一個模式,都係綁返去解剖嗰章嘅一個元件,同埋生命週期嘅一個階段。目標塑造嗰啲喺第一個階段被發現嘅工作。評估準則就係評估器喺第四個階段嘅檢查。而護欄就係最後一個階段嗰個「繼續、孵化、定停」嘅決定。將一個 Loop 設計得好,到頭嚟,就係將呢三樣嘢設計得好。啲元件只係畀每一個決定一個落腳嘅地方。

咁我哋就將呢一章嘅重點諗法歸納埋一齊。要理解一個 Loop,最快嘅方法就係整一個最細、可以行得到嘅版本,睇住佢喺邊度爆煲,然後加結構去修正每一個弱點。

一個最小化、Ralph 風格嘅 Loop,只係一個簡單嘅循環,佢叫一個生成器 agent、捉住佢嘅輸出同錯誤、然後將佢哋回收入去下一次疊代嘅脈絡,直到一個哨兵停止條件被滿足為止。

嗰個最小化 Loop 係特登冇隔離、冇獨立評估器、亦都冇保存狀態,而正正係嗰啲缺口,就係之後嗰個生產級數 Loop 嘅動機。

生產級數嘅 Loop 保持同一個「生成、評估、回饋」嘅心跳,但係佢用一個明確嘅元件去填返每一個缺口。每一次嘗試一個用完即棄嘅工作空間,用嚟做隔離。一個獨立嘅評估器,佢喺測試上面、同埋喺一個評分標準分數上面把閘。一個保存進度嘅檔案,佢喺當機之後仍然生存,並且為下一次嘗試提供資訊。同埋明確嘅停止條件,成功、一個最大疊代次數、同埋一個預算上限,令到失控嘅 Loop 變成冇可能。

每一樣生產上嘅新增,都一對一咁對應到解剖嗰章嘅一個元件,同埋生命週期嘅一個階段。整一個 Loop,大部分時候就係將嗰啲零件變得明確、又可以被檢視。

用減法去整一個 Loop,一次一個元件。由一個生成器 agent 加一個停止條件開始。喺信任佢之前加隔離。然後加一個評估器,再加保存狀態,再加明確嘅預算護欄。而自動化、連接器、同埋技能與知識,就只喺任務真係要求嗰陣先至加。
