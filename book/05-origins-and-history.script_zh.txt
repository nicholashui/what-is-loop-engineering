在这一章里，我们会追溯 Loop Engineering，也就是循环工程，这个名字到底从何而来，又是哪些人塑造了它。我们会看看 2026 年 6 月那篇被认为让这个名字流行起来的文章，也会听听几位实战者怎样描述这种思维上的转变，还有最早出现、为后来指明方向的一个极简技巧，叫做 Ralph Loop，也就是 Ralph 循环。因为这些都是近期、有具名出处的说法，我也会讲讲我们应该多谨慎地去看待它们，好让你可以自己去查证这些细节。

我先说一点关于资料来源的事，因为在这里这真的很重要。这一章讲的是人物、日期和引述。这类近期的历史很容易出现一些出入，事后又难以核实。所以这一章里的想法，你大可以当它是扎实可靠的；但那些具体的出处，也就是那些名字和日期，你就把它们当作值得自己对照原始资料去确认的线索吧。

明白了这一点之后，我们就来看看一个聪明的 shell 小技巧，是怎样变成一门有名有姓的学问的。

很多想法在出名之前，其实早就在实战中存在了。把 coding agent，也就是编程代理，放进一个循环里去跑，这种做法在早期采用者之间其实已经慢慢传开，直到 Loop Engineering 这个标签成形，才正式流行、走进主流。

这个名字的流行，被认为源自 Addy Osmani 在 2026 年 6 月 7 日发表的一篇文章，这一点也值得你自己去查证。那篇文章的意义，不在于它发明了什么新东西，而在于它为一些早就在发生的事起了个名字。它为一种正在浮现的做法给出了清晰的标签，那就是去设计一些系统，由系统来提示和调度 AI coding agent，而不是你一个回合一个回合地亲手去提示这些 agent。一个名字是很重要的。一旦这种做法有了名字，那些零散的实验就能被认出来，原来都是同一门叫得出名字的学问的实例。而一门有名字的学问，正是人们可以去研究、可以互相交流心得、可以有意识地去改进的东西。

这个起名的时刻，就是这段简短历史的定锚日子。但真正让它站得住脚的，是那些在自己日常工作里早就感受到这种转变、又描述得很生动的实战者。

要证明 Loop Engineering 是一个真正的改变，而不只是换了个包装，最清楚的证据就来自那些工程师，他们描述了自己跟 coding agent 之间的关系是怎样改变的。有两个视角特别能捕捉到这个转变。

第一个来自 Peter Steinberger，他把这个改变形容为从“提示”转向“构建那个负责提示的东西”。正如 Peter Steinberger 所说，你不应该再去提示那些 coding agent，你应该去设计一些循环，由它们来提示你的 agent。

这个说法的价值，就在于它足够直接。他没有把 Loop Engineering 形容成手动提示之上一个可有可无的升级，而是把手动提示形容成你应该停下来不再做的事。你所设计的工作单位，已经不再是那条消息，而是那个产生消息的循环。这就是整本书的核心论点，浓缩成了两句话。

第二个视角来自 Boris Cherny，他在 Anthropic 带领 Claude Code 团队，他是从自己的工作流程里去描述同一个转变的。正如 Boris Cherny 所说，我不再提示 Claude 了。我有一些循环在跑，由它们来提示 Claude，然后想清楚要做什么。我的工作就是写循环。

让这句话这么有冲击力的，是它出自谁之口。这个人就在现代 coding agent 工具的最核心位置工作，但他形容自己的工作不是去用那个 agent，而是去写那些使用那个 agent 的循环。“我的工作就是写循环”这句话，恰恰简洁地道出了这本书反复提到的角色转变。实战者从一个操作员，变成了一个系统设计者。你不再是坐在那里逐条发出指令的那个人，你是建立了那个发出指令的系统的那个人。

把这两个声音放到一起，他们是从两个角度去描述同一个转变。Steinberger 给了我们一条处方，设计循环，不要提示。Cherny 则给了我们一个亲身实践的描述，我现在是在写循环。他们两个都指回了一个技巧，这个技巧用最原始的形式展示了这个想法，而且其实还早于这个名字本身。

在这个名字出现之前，技巧就已经存在了。Loop Engineering 最直接的祖先，就是 Ralph Loop，也叫做 Ralph Wiggum 技巧，被认为是 Geoffrey Huntley 在 2025 年提出的。

Ralph Loop 简单得让你意外。它最初的形式，其实只不过是一个基本的 shell 循环。它会不断地把新的 context 喂给一个 AI coding agent，再加上上一次运行的输出和任何错误，然后一次又一次地去跑那个 agent，直到任务完成为止。没有调度框架，没有复杂机制。只有那个核心洞见，只要你不断用 agent 上一次产生的东西去重新启动它，它就能在很多次反复当中，慢慢磨向目标，而不是在单次运行结束时就放弃。Ralph Wiggum 这个搞笑的名字，挺契合这个技巧的精神。它几乎天真到直接，但偏偏它真的行得通。

用通俗一点的话讲，这个循环做的事情是这样的。只要任务还没完成，它就跑那个 agent，把提示连同上一次的输出和上一次的错误一起，当作新的 context 交给它。它会捕捉 agent 产生的一切，存下输出和任何错误，然后再绕回去重新跑那个 agent。它会一直这样做，直到任务终于完成。

这个小小的循环，就是这本书一切的种子。它确立了最根本的那一步，把一个 agent 放进循环里去跑，再把结果喂回去，用一种你能想到的最精简的方式，别的什么都不加。

那我们是怎样从那里走到 Loop Engineering 的呢？Loop Engineering 就是当你认真看待这粒种子，再把它培育成一些达到生产级水平的东西之后所得到的成果。Ralph Loop 是概念验证。Loop Engineering 就是在它之上建立起来的工程学问。而这个演进，就补上了那些赤裸裸的循环刻意略去的部分。

第一，隔离，让每一次运行都安全而且可以还原，而不是在一个共享的工作空间上面乱涂乱改。第二，一个独立的评估者，也就是 Evaluator，让“完成”是由 agent 以外的东西来判断，而不是只假设它完成了。第三，持久化的状态，让进度是真实而且耐用的，而不是只靠什么东西碰巧在一个 context window 里残存下来。第四，明确的停止条件，让这个循环知道什么时候该收手，无论是测试通过、达到质量门槛，还是到了预算上限，而不是无止境地绕圈。

换句话说，Ralph Loop 提供了直觉，不断用反馈去重新运行那个 agent。而 Loop Engineering 就提供了结构，安全地、可验证地、耐用地去做，还要有一个明确的出口。这本书其余的部分，很大程度上，就是在讲在后面的章节里，一个组件一个组件地把这个结构补上的故事。

这就是为什么这条历史脉络这么重要。技巧最先出现，证明了这个想法行得通。实战者的声音道出了思维上的转变。而起名的那一刻，就给了整个做法一个身份。这三样加到一起，就不只是解释了 Loop Engineering 是什么，还解释了它是怎样出现的。

那我们就来总结一下要点。Loop Engineering 这个名字的流行，被认为源自 Addy Osmani 在 2026 年 6 月 7 日的文章。它的重要性在于为一个早就在浮现的做法起了个名字，让它可以当作一门学问去研究和改进。有两位实战者的视角捕捉到了这个思维转变。Peter Steinberger 力主要设计一些循环去提示你的 agent，而不是直接去提示 agent。而在 Anthropic 带领 Claude Code 的 Boris Cherny，则形容自己的工作是写那些提示 agent 的循环。他们两个都是在讲从操作员转变成系统设计者。

最直接的先行者就是 Ralph Loop，也就是 Ralph Wiggum 技巧，被认为是 Geoffrey Huntley 在 2025 年提出的。它是一个简单的循环，用新的 context 加上之前的输出和错误去重新运行一个 agent，直到任务完成。而 Loop Engineering 就把这个 Ralph Loop 演进成一门生产级的学问，加入了隔离、一个独立的评估者、持久化的状态，还有明确的停止条件。赤裸的循环是概念验证。工程就是在它周围建立起来的结构。

最后再提醒你一句。这一章里每一个有日期的事件、每一个具名的出处、还有每一句直接引述，都是取自原始材料的近期说法，所以真的很值得你自己对照原始资料去查证。
