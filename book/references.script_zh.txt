在我们结束之前，想简单聊一聊资料来源这个问题。

这本书在写作的时候，并没有实时连接互联网，也没有使用任何在线搜索工具。不过这个限制，并不会影响到那些概念本身。Loop Engineering 的定义、一个 loop 的结构、它的生命周期、各个实际的例子、有哪些风险以及该怎样应对，这些全都是根据扎实的背景知识和手头上的资料来解释的。所以这些概念你是可以信得过的。你真正需要再去核实的，是那些出处的标注。每一次本书提到某个特定人物是某个想法或者某句话的来源、指向某个特定的日期、又或者复述某个人的原话，这些细节在写作的时候，都还无法对照原始出处去查证。

所以呢，与其悄悄地把这些资料省略掉，又或者把它们当成已成定局的历史那样去讲，本书反而会把每一项都当作是作者提出的一个说法，是一些你应该自己去确认的东西。你可以把它们当成是一些指向值得查证的来源的线索，而不是已经确立的事实。如果你自己去看了原始出处之后，发现和你在这里听到的有出入，那就以出处为准。

这类说法总共有四个，让我一个一个来讲。

第一，是 Loop Engineering 这个词语本身。它之所以流行，要归功于 Addy Osmani 在 2026 年 6 月 7 日所发表的一篇帖子。重点是，这篇帖子是为一个其实已经逐渐成形的做法起了个名字，而不是从零开始发明出来的。

第二，是 Peter Steinberger 提出的一个辅助观点。他把这个转变形容为：从向 agent 下 prompt，转向去设计那些向 agent 下 prompt 的 loop。他的说法是：你不应该再向 coding agent 下 prompt 了；你应该去设计那些会向你的 agent 下 prompt 的 loop。

第三，是 Boris Cherny 一个类似的看法，他在 Anthropic 负责带领 Claude Code。他从自己日常的工作当中，形容了同样的一个转变。他的说法是：他不再向 Claude 下 prompt 了。他改为设置了一些 loop 在运行，由这些 loop 去向 Claude 下 prompt、去想应该做些什么，而他自己的工作，就是去写这些 loop。

第四，是在这一切之前就出现的技巧，也就是 Ralph Loop，有时又叫做 Ralph Wiggum technique。它在 2025 年归功于 Geoffrey Huntley。它被形容为一个简单的 loop，会不断把新的 context，连同上一次运行的输出和错误，一起喂回给这个 agent，直到任务最终完成为止。

所以这四个就是了。这些名字、日期以及引述，都已经尽可能忠实地记录了下来，不过在你依赖它们之前，请花一点时间自己去核实一下。
