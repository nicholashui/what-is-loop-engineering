## 名詞誕生：Addy Osmani，2026 年 6 月

概念喺實踐上往往早喺有名稱之前就已經存在。喺 *Loop Engineering* 呢個標籤成形
並廣為人知之前，將編程代理（coding agent）放喺循環裏面運行嘅做法，其實已經喺
早期採用者之間流傳開嚟。

「Loop Engineering」一詞嘅普及，一般歸功於 Addy Osmani 喺 2026 年 6 月 7 日
發表嘅一篇貼文 [^aac-osmani-2026] *（作者自述；請自行查證）*。嗰篇貼文嘅重要
之處，與其話係發明咗啲新嘢，不如話係佢 *命名* 咗一啲早就喺度發生緊嘅嘢：佢
為一種正在興起嘅做法賦予咗一個鮮明嘅標籤——即係設計一啲會去提示同統籌 AI
編程代理嘅系統，而唔係由人手一回合一回合咁去提示嗰啲代理。一個名稱好緊要。
一旦有咗名，零散嘅實驗就可以被認出係同一門可命名學科嘅實例——而一門有名稱
嘅學科，係人哋可以去研究、互相交流心得、並且刻意改進嘅嘢。

呢個命名時刻，就係呢段簡短歷史嘅定錨日期。不過真正令佢站得住腳嘅，係嗰啲早就
喺自己日常工作中親身感受到呢種轉變、而且可以生動描述出嚟嘅實踐者。

## 心態嘅轉變：兩把實踐者嘅聲音

最能證明 Loop Engineering 係一場真正改變——而唔單止係換個招牌——嘅證據，
來自啲工程師描述佢哋 *自己* 同編程代理之間嘅關係點樣改變咗。以下兩個觀點，
兩者喺度都以作者自述主張形式呈現，特別能夠捕捉到呢種轉變。

### Peter Steinberger：由提示走向設計循環

其中一個佐證觀點歸於 Peter Steinberger [^aac-steinberger]，佢將呢種改變
表述為由提示轉向 *打造嗰個去提示嘅東西*：

> "You shouldn't be prompting coding agents anymore. You should be designing
> loops that prompt your agents."
>
> 「中譯：你唔應該再去提示編程代理。你應該設計一啲循環，由佢哋去提示你嘅
> 代理。」

呢個表述嘅價值在於佢夠直白。佢並無將 Loop Engineering 形容為人手提示嘅一個
*可有可無嘅升級*；佢係將人手提示形容為你應該 *停止去做* 嘅嘢。你所設計嘅
工作單位唔再係嗰條訊息——而係嗰個產生訊息嘅循環。呢個就係本書整套論點濃縮
成兩句話。

### Boris Cherny：「我嘅工作就係寫循環」

第二個佐證觀點歸於 Boris Cherny，佢係 Anthropic 嘅 Claude Code 主管
[^aac-cherny]，佢由自己工作流程嘅內部描述咗同一種轉變：

> "I don't prompt Claude anymore. I have loops running that prompt Claude and
> figuring out what to do. My job is to write loops."
>
> 「中譯：我已經唔再提示 Claude。我有啲循環喺度運行，由佢哋去提示 Claude 並
> 諗下要做啲乜。我嘅工作就係寫循環。」

令呢個觀點咁觸目嘅，係 *邊個* 講出嚟：一個身處現代編程代理工具核心嘅人，將
自己嘅工作形容為唔係 *使用* 代理，而係 *寫出嗰啲使用代理嘅循環*。「我嘅工作
就係寫循環」呢句話，簡潔咁道出咗本書一再回到嘅角色轉變——實踐者由操作者
變成系統設計師。你唔再係坐喺位上逐條發出指令嗰個人；你係打造咗發出指令嗰個
系統嘅人。

兩把聲音合埋一齊，由兩個角度描述咗同一場過渡——Steinberger 以一種處方形式
（「設計循環，唔好提示」），而 Cherny 就以親身實踐嘅描述形式（「我而家寫
循環喇」）。兩者都指向一個以最原始形態展示咗呢個概念、而且早於名稱本身嘅
技巧。

## 先驅：Geoffrey Huntley 嘅 Ralph Loop

喺呢個名詞出現之前，技巧已經存在。Loop Engineering 最直接嘅祖先就係
**Ralph Loop**（Ralph 循環）——又叫做「Ralph Wiggum 技巧」——歸於
Geoffrey Huntley 喺 2025 年 [^aac-huntley-2025]。

Ralph Loop 簡單到令人卸下心防。喺佢最初嘅形態下，佢只不過係一個基本嘅 bash
循環：佢不斷將新鮮嘅脈絡，加埋上一次運行嘅輸出同錯誤，餵畀一個 AI 編程代理，
然後一次又一次咁運行嗰個代理，直到任務完成為止。冇統籌框架，冇複雜機械
裝置——淨係得個核心洞見：*如果你不斷用代理上一次產生嘅嘢去重新啟動佢，佢就
可以喺多次迭代之間慢慢磨向目標*，而唔係喺單次運行結束時就放棄。「Ralph
Wiggum」呢個玩味嘅名稱好啱呢個技巧嘅神髓：佢直白到近乎天真，但偏偏又行得通。

從概念上講，呢個赤裸裸嘅循環睇落係咁樣：

```bash
# Ralph Loop 嘅精神所在：不斷重新運行代理直到任務完成，
# 並將上一次嘅輸出同錯誤作為新鮮脈絡再餵返入去。
while ! task_is_done; do
  agent --context "$(cat prompt.md previous_output.txt errors.txt)" \
        > previous_output.txt 2> errors.txt
done
```

嗰個細細嘅循環就係本書一切嘅種子。佢以可以想像得到嘅最精簡方式，確立咗嗰個
基礎動作——*將代理放喺循環裏面運行，並將結果餵返入去*——而且冇附加任何其他
嘢。

### 由 Ralph Loop 到 Loop Engineering

當你認真對待嗰粒種子，並將佢培育成達到生產級水平嘅嘢，得出嚟嘅就係 Loop
Engineering。Ralph Loop 係概念驗證；Loop Engineering 係建基於佢之上嘅工程
學科。呢個演化加返赤裸循環刻意略去嘅部分：

- **隔離（Isolation）**，等每一次運行都安全而且可逆，而唔係喺一個共享工作空間
  上面亂塗亂改。
- **一個獨立嘅評估器（Evaluator）**，等「完成」係由代理本身以外嘅嘢去 *判斷*，
  而唔係單純假設。
- **持久化狀態（Persisted state）**，等進度係真實而且耐久，而唔係淨靠脈絡視窗
  （context window）裏面碰巧殘留低嘅嘢。
- **明確嘅停止條件（Stopping conditions）**，等循環知道幾時應該收手——無論係
  測試通過、達到品質門檻，定係觸及預算上限——而唔係無止境咁轉落去。

換句話講，Ralph Loop 提供 *直覺*（持續帶住回饋去重新運行代理），而 Loop
Engineering 就提供 *結構*（安全、可驗證、耐久咁去做，而且有明確嘅退出）。本書
其餘部分，好大程度上就係講述加上呢層結構嘅故事——喺接下嚟嘅各章節中逐個組件
咁加上去。

呢個就係點解呢條歷史線索咁重要：技巧首先出現並證明咗呢個概念行得通；實踐者
嘅聲音道出咗心態轉變；而命名時刻就賦予咗整套做法一個身份。三者合埋一齊，唔
單止解釋咗 Loop Engineering *係乜*，仲解釋埋佢 *點樣* 出現。
