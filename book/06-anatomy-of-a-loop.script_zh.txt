在这一章里，我们会把一个循环打开，逐一为它的各个部件命名。看完之后，你就会认识构成一个循环的八个组成部分。它们分别是：Automation，也就是 Trigger，即自动化、触发器；Worktrees，负责隔离，也就是工作树；Generator Agent，即生成代理；Evaluator，即评估器；State，也叫 Memory，即状态、记忆；Skills and Knowledge，即技能与知识；Connectors，即连接器；最后是 Stopping Condition，即停止条件。对于每一个部件，你都会了解到它的用途、它在循环里所承担的那份工作，也会至少看到一个具体的例子，讲解那份工作在实践中是怎样完成的。看完之后，你就能够指着循环的每一个构建组件，说出为什么每一个都各得其所。

之前我们为你描绘过一个循环的图景：一个会自行转动的圆圈。它会提示、会观察、会验证，然后会迭代。我们也展示过这个圆圈是怎样从一次单独的代理对话当中抽离出来，再构建成一个系统的。现在，就是时候打开这台机器，逐一为部件命名了。

一个循环由八个组成部分装配而成。每一个都对应着一项判断或者一份工作；在人工提示的时候，这些工作本来都是你亲自去做的。决定什么时候开始。避免工作彼此碰撞。做实际的编码。检查结果是否足够好。记住发生过的事情。知道项目的规则。和外面的世界沟通。以及，决定什么时候停止。把每一项都搬进系统中一个明确的部件里，你就得到了一个循环。

我们会逐一来看，按照工作流经它们的顺序。对于每一个组成部分，结构都是刻意保持一致的。我们会讲它是什么、它的用途是什么，以及至少一个具体的例子。这样你就可以把这八个都记在脑海里，把它们当作同一个模式的各种变奏：一个只做一份工作的部件，好让这个循环不需要由人去做那份工作。

我们先从 Automation，也就是 Trigger，即触发器讲起。

触发器是决定工作在什么时候、以什么方式进入循环的组成部分。它就是发令枪，一个不用人工敲一个“go”字就能够启动一次迭代的机制。在人工提示的时候，是你去开始每一个回合。你决定有工作要做，然后动手去做。触发器把那个决定搬进系统里，这样循环就可以按时间表、或者因应某个事件而开始工作，而不用等一个人去发起它。这正是让循环可以在你睡觉的时候也照样运行的关键。工作会自行开始。

一个触发器可以有几种不同的形态。它可以是一个排程任务，比如说一个每晚凌晨两点触发的任务，去接手下一项工作。它也可以是一个绑定新议题的 webhook，这样一有议题在追踪系统里创建，这个 webhook 就会触发，循环就会开始处理它。又或者，它可以是一条简单的指令，比如敲一个 slash-loop，由一个人、甚至是另一个系统，敲一条指令就立即启动循环。举个例子，设想有个触发器设定成一个每晚的排程，每天凌晨两点悄无声息、无人看管地启动循环。所有这些做法的共通之处，就是“什么时候开始”这个决定是预先编写好的，而不是由一个人现场去做的。

接下来就是 Worktrees，即工作树，以及隔离。

这个组成部分让每一次代理运行都在自己一个封闭的工作空间里进行，这样一次运行所做的改动就不会踩到另一次运行的改动。为什么这么重要呢？一个循环可能会运行很多次迭代，有时还会同时运行好几个代理。没有隔离，两次运行同时改同一批文件，就会破坏对方的工作；而一次失败的运行更有可能用一种无法恢复的方式，搞坏一个共用的工作空间。隔离可以防止代理之间互相踩踏，也能让每一次运行都安全又可逆。如果一次迭代出了错，你只需要把它那个被隔离的工作空间丢掉，其他一切都不受影响。

这方面的标准工具就是 git worktrees 以及独立分支。用 worktrees，每一次代理运行都会从同一个仓库检出它自己的工作目录，这样它的改动就和其他所有运行的改动实质上分开了。用独立分支，每一次运行都会提交到它自己的分支，这样它的历史就被隔离开来，可以独立地去审阅、合并或者丢弃。设想一下，让第四十二次迭代有它自己的目录和自己的分支，那个代理在里面完全私密地工作，不受任何其他运行的干扰。隔离正是把“很多次运行”从一个负担变成一项优势的关键。这些运行可以并行进行，也可以无害地失败。

现在讲到这台机器的心脏：Generator Agent，即生成代理。

生成代理就是那个产出循环实际成果的 AI 编码代理。那些代码、那些修改、那些改动。它就是机器里的工人。这个组成部分做的，正是一个人本来要逐个回合亲手去做的实际工作。这里有一个重点，值得你记住：循环不等于代理。生成代理只是循环的其中一个部分，负责生成成果的那个部分。这一章里其他所有东西之所以存在，都是为了决定生成代理应该做什么、保护它的安全、检查它的工作，以及决定要不要再运行它一次。

一个生成代理由一个现成的 AI 编码代理来充当。它可以是像 Claude Code 这样的工具，作为那个读取任务、再修改代码库的代理。又或者是一个 Grok 代理，在循环里扮演完全一样的生成角色。设想一下，为一次迭代去调用那个代理：它读取当前的任务，然后在它自己被隔离的工作空间里产出它的修改。循环不在乎哪个代理来充当这个角色，只要有东西负责做生成的工作就行。代理是可以互换的，但角色是固定的。

工作做完之后，就需要有人去评判它。这就是 Evaluator，即评估器，有时也叫评论者或者验证者。

评估器是那个拿生成代理的成果去对照某些准则、再决定它是否足够好的组成部分。它就是那个独立的裁判。在循环胜过一次纯粹的代理运行的众多优点当中，这是最重要的其中之一。在一次单独的代理对话里，代理同时是工人，也是它自己工作的唯一裁判。循环则加入了一个外部的评估器，这样“完成”就是经过验证的，而不只是靠猜。评估器正是保护循环免于自信满满地产出错误或者低质量成果的关键，因为在生成者之外有东西检查过之前，任何东西都不会被接纳。

评估可以有几种形式，而且通常会合并起来用。你可以有一个独立的代理去跑测试，执行项目的测试套件，把失败当成拒绝。你可以有一个代码审阅或者评论代理，也就是第二个代理去读那些改动，就质量、正确性和风格去审阅它。又或者你可以有一个评分准则计分器，按一份明确的评分准则去为成果打分，再把分数和一个门槛比较。设想一个以测试为本的评估器，把它当成一道闸门：如果测试通过，这次迭代就被接纳；如果失败，那些错误就会反馈回去，循环再试一次。无论它是什么形式，评估器的工作都一样。从外面去评判工作，再交回一个循环可以据此行动的裁决。

接下来是 State，也叫 Memory，即状态、记忆。

这个组成部分在任何单一代理的语境之外，把循环的进度保存下来，这样信息就可以从一次迭代延续到下一次。一次单独的代理运行，只记得它上下文窗口里的东西，而那份记忆一到运行结束就立刻消失。一个循环要在很多次运行之间做出真正的、累积得到的进展，就需要一份比任何一次运行都更长寿的记忆。一份记录，记住做了什么、还剩下什么、学到了什么。状态和记忆正是让第四十二次迭代可以建立在第四十一次所完成的成果之上、而不用从零开始的关键。

状态通常会放在耐用、又可以查阅的地方。它可以是一个待办文件，记录剩下的工作，每次迭代都更新。它可以是一张草稿纸，让循环在一次次运行之间记下一些发现和决定。它可以是一块在像 Linear 或者 Notion 这样的工具里的项目板，追踪每一个工作单元的状态。又或者，它可以只是磁盘上的纯文件，记下一些中间结果。设想一个简单的待办文件，当中有一项在第四十一次迭代被勾掉为“完成”，而下一项还没勾的就准备好交给第四十二次迭代。而正因为状态住在代理之外，它也是你自己可以读的东西，让你看得清楚循环究竟走到哪一步了。

接着就是 Skills，和 Knowledge，即技能与知识。

这个组成部分就是一个循环提供给它的代理的、已成文又可重用的指引。项目的规则、惯例和模式，写在一个代理真正用得到的地方。它解决的问题是这样的。一个不熟悉你项目标准的代理，会一次又一次地、前后不一致地重新发明那些标准。技能和知识把那份指引一次性记下来，这样每一次迭代都遵循同一套规则，而不用靠一个人在每一条提示里都重复说“用我们的命名惯例”，或者“永远都要写测试”。这正是一个循环怎样保持与你团队实际工作方式一致的关键。

这份指引通常会以代理会读的文件形式存储。它可以是一个技能文件，描述代理应该遵循的可重用流程。它可以是一个代理指引文件，告诉代理在这个特定的仓库里怎样运作。又或者它可以是一些成文的编码标准，代理要遵从。设想一个简短的指引文件，让代理沿用现有的模块布局、为每一个新函数写一个单元测试，以及优先用项目的日志辅助工具、不要只用 raw print。技能和知识，正是“一个靠猜你惯例的代理”和“一个把惯例摆在面前的代理”之间的区别。

现在我们来到 Connectors，即连接器。

连接器就是一组集成，让循环可以和外部系统沟通，也就是你团队已经在用、用来追踪和交付工作的那些工具。一个循环不是在真空里运作的。它要在某处读取工作、在某处写下结果，也要告诉别人发生了什么事。连接器给了循环在外面世界的一双手和一把声音。它们让循环可以从一个追踪系统拉一项任务出来、创建一个拉取请求，又或者贴一则更新，这样循环就成为了真实工作流程的一部分，而不是一座孤岛。

常见的连接器包括一个像 GitHub 这样的源代码控制主机，用来读代码和创建拉取请求。也包括一个议题追踪系统，用来拉出下一个工作单元和更新它的状态。还包括一个像 Slack 这样的聊天平台，用来公布进度，或者在需要人工关注的时候升级上报。设想一个循环同时接驳着这三样东西：通过 GitHub 读代码和创建拉取请求、通过议题追踪系统拉任务和更新它们的状态，以及把进度贴到一个聊天频道。连接器正是让一个循环成为你现有系统的参与者、而不是一个封闭实验的关键。

最后，就是 Stopping Condition，即停止条件。

停止条件就是那条规则，告诉循环什么时候要停止迭代，或者什么时候应该升级交给人、而不是继续下去。一个循环会不断重复，而没有明确出口的重复是很危险的。它可以永无止境地转、烧光你的预算，又产不出任何有用的东西。停止条件给了循环一个清晰的“完成”定义，或者“放弃、求助”的定义，这样它就是有意识地终止，而不是无止境地运行下去。它正是防止一个循环变成一个失控程序的组成部分。

一个停止条件通常是一条或者多条明确的规则。一条常见的规则就是测试通过：一旦评估器那个测试套件亮起绿灯，目标就达成，循环就停。另一条是评分准则的分数达到某个门槛，比如说一达到至少十分之九分就停。再有一条就是安全限制，比如一个最大迭代次数，或者一个开支预算。比如说跑满二十次迭代之后就停，或者一旦成本预算用光就停，并且升级交给人、而不是继续下去。设想一个合并的检查，每次迭代都跑：如果测试通过，它就停，因为目标达成了；如果已经到了二十次迭代，它就停，然后升级交给人；如果超了预算，它也停，一样升级上报。第一条规则定义成功。其余的就定义安全。它们加在一起，就保证这个圆圈会停得下来。

那这八个是怎样拼在一起的呢？按顺序去读，它们就描绘出一个工作单元流经循环的路径。触发器启动一次迭代。工作树和隔离给那次迭代一个安全的地方去运行。生成代理在那里做工作，由技能和知识来引导，再通过连接器去接触外面的世界。评估器评判结果。状态和记忆记下发生了什么，这样下一次迭代就可以建立在它之上。而停止条件就决定这个循环是再转一次，还是收工。

不是每一个循环都需要凑齐所有组成部分。后面，我们会看一个极简的循环，当中只有它们其中几个。但这八个加在一起，就构成了部件的完整词汇表。懂得每一个是用来做什么的，正是让你在设计自己的循环的时候，能够决定哪个要纳入、哪个要合并、哪个要剔除的关键。在下一章，我们会把这些部件动起来，追踪它们在循环的生命周期里是怎样互相配合的。

让我们回顾一下讲过的东西。一个循环由八个组成部分装配而成，而每一个都接手了一项人在手工提示时所做的判断或者工作。触发器决定工作在什么时候、以什么方式进入循环，无论是通过一个每晚的排程、一个绑定新议题的 webhook，还是一条简单的指令。工作树和隔离让每一次代理运行都保持分开又可逆，这样这些运行就不会互相踩踏，用的是像 git worktrees 和独立分支这样的工具。生成代理做那份产出代码和修改的实际工作，由像 Claude Code 或者一个 Grok 代理这样的东西来充当；记住，循环不等于代理，代理只是它的其中一个部分。评估器从外面去检查成果，这样“完成”就是经过验证的，而不是靠猜，无论是通过一个独立的代理跑测试、一个代码审阅代理，还是一个评分准则计分器。状态和记忆在任何代理的语境之外把进度保存下来，这样迭代就可以累积得到，无论是在一个待办文件、一张草稿纸、一块项目板，还是磁盘上的文件里。技能和知识把项目的规则和模式记成文，这样每一次迭代都遵循它们，可以通过一个技能文件、一个代理指引文件，又或者一些成文的编码标准。连接器把循环和像 GitHub、一个议题追踪系统，或者 Slack 这样的外部工具集成在一起。而停止条件就告诉循环什么时候要停或者升级，无论是测试通过、评分准则分数达到门槛，还是撞到一个最大迭代次数或者预算上限。它们八个加在一起，就描绘出一个工作单元从触发到终止的全程。不是每一个循环都需要凑齐八个，但懂得每一个的用途，正是让你能够有意识地去设计一个循环的关键。
