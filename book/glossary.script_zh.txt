在我们做总结之前，我想带大家从头到尾梳理一下这本书里用到的几个关键词，逐个解释它们的意思。你可以把这一段当作一份口述的参考，在所有概念慢慢拼到一起的时候，把它们记在心里。

我们先从最核心的想法讲起。Loop，也就是循环，是由 loop engineer，也就是循环工程师，所建立的一套统筹系统。它会发掘或者触发工作，在隔离的环境里启动代理，验证它们的输出，在多次运行之间保存状态，然后再决定是继续、启动子代理，还是停下来。

而 Loop Engineering，也就是循环工程，是一种实践，讲的是设计一套控制系统，去自动地向 AI 编码代理发出指令、观察、验证并迭代，而不是靠人一个回合一个回合地去逐次提示它。它处在一条脉络的最顶端，这条脉络从 Prompt Engineering 开始，经过 Context Engineering，再到 Agent Harness Engineering。

这条脉络值得我们顺着每个阶段逐一来讲。Prompt Engineering，也就是提示工程，是第一个阶段。它的实践就是雕琢一条单一指令的措辞，包括用词、例子、输出格式、角色设定，全都是为了引导模型，在一个提示对一个回应之间，给出一个好的答案。

Context Engineering，也就是情境工程，是第二个阶段。它的实践就是有意识地去组织放入模型情境窗口的信息，包括相关的文件、检索回来的文档、之前的对话、例子和指令，让模型能够根据合适的材料去推理。简单说，它是在 Prompt Engineering 精挑细选的用词之上，再加入合适的信息。

Agent Harness Engineering，也就是代理框架工程，是第三个阶段。它的实践就是为单独一次代理运行，建立一个可靠的执行环境，这个环境就叫做 Harness。它在一个工作会话里加入了行动和反馈，意思就是工具、观察到的结果，以及多步骤的控制，全都建立在精心整理过的情境之上。

这就带到我们要讲的 Harness 本身。Harness，也就是框架，是为单独一次代理运行而设的可靠执行环境。它会给代理供应工具，把工具的结果反馈回代理的情境里，管理一个多步骤工作会话的来来回回，并且让那次运行保持稳定。Loop 的统筹层，就是建立在 Harness 之上的。

接下来，讲讲代理。AI Coding Agent，也就是 AI 编码代理，是一套以大型语言模型为基础的系统，它可以针对一个软件任务去推理，并且采取行动，包括执行命令、读取和编辑文件、检查结果，而不只是产生文字。在一个 Loop 里，AI 编码代理通常会担任 Generator Agent 的角色。

Generator Agent，也就是生成代理，是 Loop 里负责产生实际输出的部件，比如代码、修改，或者其他成品。它通常是一个 AI 编码代理，例如 Claude Code 或者一个 Grok 代理。

Sub-agent，也就是子代理，是 Loop 启动出来的一个次级代理，用来处理工作里一个聚焦的部分，比如一个由主运行分支出来的专门任务。启动子代理，是 Loop 在决策点的三个结果之一，另外两个就是继续和停下来。

接着，讲讲一个 Loop 实际上是怎么运行的。Iteration，也就是迭代，是 Loop 的一次完整流程，包括发掘或者接收工作、运行一个代理、评估输出、保存状态，然后再决定下一步做什么。一个 Loop 是通过很多次迭代去推进一个目标，而不是靠单独一次运行。

Stopping Condition，也就是停止条件，是一条会让 Loop 终止的规则。例子包括测试通过、Rubric 的分数达到某个门槛，以及到达最大迭代次数，或者用尽了预算。

Token Budget，也就是代币预算，是对 Loop 可以消耗的代币数量，也就是成本，所设的一个上限。紧的预算配合停止条件，可以帮助降低代币成本失控增长的风险。

现在我们逐个看看组成一个 Loop 的部件。Automation，也叫做 Trigger，也就是自动化或者触发器，是决定工作何时、以何种方式进入 Loop 的部件。例子包括一个 cron 排程、一个在有新议题时触发的 webhook，以及一个按需要调用的斜线命令。

Evaluator，也就是评估器，有时又叫做 Critic 评论者或者 Verifier 验证器，是在代理外部独立地检查 Generator Agent 输出的部件。它可能是一套测试、一个做代码审查或者评论的代理，又或者是一个 Rubric 评分器。它就是让一个 Loop 能够判断工作究竟是不是真的好，而不只是靠代理自己的评估。

Rubric，也就是评分准则，是一套明确、有评分的评估标准，用来判断 Generator Agent 的输出。一个 Evaluator 可以用 rubric 去打分，而 rubric 分数的门槛又可以用来做停止条件。

State，也叫做 Memory，也就是状态或者记忆，是在多次迭代之间保存进度的部件，这样就算任何一次单独运行结束了，工作也不会流失。例子包括一个 TODO file、一个草稿区、一块项目看板，以及磁盘上的文件。

Worktrees，也叫做 Isolation，也就是工作树或者隔离，是让每次代理运行都保持隔离、并且可以还原的部件。例子包括 git worktrees 和分开的分支，这样就让代理的工作可以进行，而不会干扰到主线的开发。

Connectors，也就是连接器，是把 Loop 和外部系统整合到一起的部件。例子包括源码控制平台、议题追踪系统，以及聊天平台。

Skills，也叫做 Knowledge，也就是技能或者知识，是向代理提供可重用指引和标准的部件。例子包括一个技能文件、一个代理指引文件，以及编码标准。

接下来讲讲一些基础的模式和前身。ReAct，这个词是 Reason 加 Act，也就是推理加行动，是一种把推理步骤和行动交替进行的代理模式。代理会想想下一步做什么，通过一件工具采取一个具体行动，观察结果，再次推理，就这样一直到任务完成为止。它是 Loop 的概念祖先，因为 Loop 就是把这个推理加行动的循环，从单独一个代理对话里抽出来，放进一套外部、可控制的系统。

Ralph Loop，也叫做 Ralph Wiggum 技巧，是 Loop Engineering 由它演化出来的前身技术。它是一个简单的 shell 循环，会不断地给代理喂入新鲜的情境，再加上前一次运行的输出和错误，然后重复运行，直到任务完成为止。它展示了在一个循环里运行代理、再把结果喂回去这个核心想法，但是没有隔离、没有一个独立的评估器，也没有保存状态。

Manual，也就是人工提示，或者说逐个回合提示，是和 AI 编码代理合作的传统方式，就是由一个人发出一条指令，读读结果，再发出下一条指令，一步一步地监督每一步。Loop Engineering 就是用一套会自己去提示和迭代代理的系统，去取代这种亲力亲为的监督。

还有几个关于质量和风险的词。Slop，也就是劣质产物，是指当代理在没有强力验证的情况下运行时，所产生的大量低质量输出。它是 Loop 的一个主要风险，可以靠测试和评论代理去把关，控制 Loop 接受什么，从而降低这个风险。

最后，讲讲一些关于这本书本身，以及它是怎么搭建起来的词。Book，也就是本书，是整个教育成品，由有次序排列的 Markdown 章节文件，加上一些辅助文件组成，包括目录、这份词汇表，以及参考资料。

Chapter，也就是章节，是一个单一的顶层 Markdown 文件，涵盖本书其中一个主要的主题范畴。

Reader，也就是读者，是本书的目标学习者，是一位软件工程师或者技术从业者，在使用者层面已经熟悉 AI 编码代理，但是才刚刚开始接触怎么自动地去统筹它们。

Worked Example，也就是实作范例，是一个具体、从头到尾的示范，无论是代码、配置，还是一段带注解的逐步讲解，都是用来示范一个概念在实践里是怎么运作的。本书会提供一个最精简的 Ralph 式范例，以及一个生产级的范例。

至于 Author-Asserted Claim，也就是作者自行断定的声明，是一句事实陈述，可能是一件有日期的事件、一个具名的出处，或者一句直接引语，由所提供的来源材料抽取出来。本书会坦白地把它呈现出来，并且注明它需要读者自行独立核实，因为在撰写的环境里，它没有办法对着即时的来源去查证。这类声明会带有一个行内标记，并且全部收集在参考资料那个文件里。

这就是全套关键词了。当你一步步去理解这些概念的时候，记得把它们放在手边，这样本书的每一个部分，就会慢慢地拼得清楚一些。
