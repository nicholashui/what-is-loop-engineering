在这一章里，我们会看看为什么 Loop Engineering 值得你下这一番功夫，以及究竟什么时候才应该用它。我想把两个最重要的好处讲清楚。第一个好处，是 agent 可以自己交出有意义的成果，不需要你不停地给提示去推它。第二个好处，就是你自己角色的转变，从一个一步一步去操作的操作员，变成整个系统的架构师。讲完这两点之后，我们会逐一拆解哪几类工作最适合用 loop。我们会看重构、功能实现，还有调试，并解释每一类工作究竟有什么特质，让它们天生就适合用 loop。看到最后，你就有能力面对任何一个任务，去判断究竟搭一个 loop 值不值得回本。

那我们就从一个很公道的问题开始。上一章已经把一个运作中的 loop 放在了你手上。在你再搭一个之前，问一句是很合理的：所有那些机制究竟为我带来了什么？把一个 agent 包在 worktree、evaluator、持久化的状态，还有停止条件里面，是实实在在的工夫。如果回报只是同样的结果、只是自动化了，那就很少会划算。

但回报其实比这更大。一个 loop 同时改变两样东西。它改变了 agent 在没有你的情况下能完成什么，也改变了你自己把注意力放在哪里。这两样就是这一章要展开讲的两个好处。它们也是下半部分的切入点，因为一旦你明白了 loop 为什么有帮助，哪些任务才适合用它这个问题，就会开始自己有答案。

我们先讲第一个好处：不需要持续给提示，也可以自主推进。

人工地一步一步给提示，会把工作的节奏绑死在你有没有空上。Agent 做了点东西，你看一下，你打下一句指示，你等着，然后你又再看一下。你一停手打字，进度就立刻停下来。Agent 本身完全有能力，但你一停，它就闲置在那里，于是你的注意力就变成了一个本身很有能力的工作者身上的瓶颈。

一个 loop 就拆掉了这种绑定。因为 loop 自己会供应下一个提示，把上一次迭代的输出和 evaluator 的裁决叠回到下一次尝试里面，于是工作可以在没有人待在内层循环的情况下继续下去。而实际的结果，就是每一个 loop engineer 最先注意到的那样：你在做第二件事的时候，有意义的工作也在被完成。Loop 可以在你吃午饭的时候、在一个会议期间，或者通宵地去慢慢打磨一个很长的任务。这就是那个常常被人提起的承诺，就是 agent 可以在你睡觉的时候都有真正的进展。

这件事为什么不只是方便这么简单，是值得讲清楚的。首先，产出不再跟着你敲键盘的次数走。一个人工的工作阶段，每一句人工提示才得到一个 agent 回合，但一个 loop 在自己的时钟之下，预算允许多少次，就可以跑多少次迭代。十次有耐性的打磨——你亲手做是永远都坐不住的——可以在完全没有人看着的情况下发生。第二，这些工作是真真正正向前推进，不只是试了一下。这就是 loop 和一个天真做法之间的关键区别，那个天真做法就是你只是把 agent 放在一个 while 循环里面跑。因为每一次迭代都会经过一个独立的 evaluator 把关，所以 loop 不是只在通宵生产输出。它只会保留通过了验证的输出。所以你回来的时候，看到的是已经被审核过的、有进展的成果，而不是一堆没验证过的乱猜。最后，你拿回了那些等候的时间。人工给提示那种所有的死时间——看、再给提示、等下一个回合——正正就是 loop 吸收掉的时间。你的参与移到了两头：设定任务和检视结果，而不再是中间每一个回合都要你。

正正是那个独立验证，让自主变得负责任，而不是鲁莽。没有 evaluator 的自主，只会让出错出得更快。有 evaluator 的自主，才能产生经过审核的进展。这个区别，以及搞错了会有的风险，就是下一章的主题。

现在讲第二个好处：从操作员转做架构师。这一个是关于你的角色，也是两者之中比较深层的那个。当你人工给提示去推 agent 的时候，你就是它的操作员。你驾驶着每一步，你供应每一句下一个指示，你把计划握在自己的脑子里。你的技艺是一次一句提示地表达出来，而结果的质量，就取决于你一个回合接一个回合地保持投入。

搭一个 loop，就完全改变了这份工作。你不再操作那个 agent，你开始设计那个去操作它的系统。你的技艺不再在下一句提示上表达。它是一次过、在前头、在 loop 的设计里面表达出来。你要决定目标是什么，并且把话说到位，让除了 agent 自己的乐观之外，有另一样东西可以确认目标已经达成。你要决定 evaluator 检查什么，以及怎样保持它独立于 generator agent，让它不那么容易被说服。你还要决定那些护栏是什么：那些预算、那种隔离、那条升级交给人去处理的路径，去界定这个 loop 允许做到什么。

这就是从操作员转做架构师的转变，是你工作的抽象层次上一次真正的升职。做操作员的时候，你做一百个细小的、即时的决定。做架构师的时候，你做那几个有杠杆作用的决定——目标、准则、护栏——然后你就放手交给系统，在每一次迭代之中一致地应用它们，比起一个人亲手把着方向盘去做，更加有耐性、更加划一。工程判断力没有消失。它移到了更早、更高的位置，去到控制系统的设计里面，而不再在每一句提示的内容里面。下一章会回来讲这一点，因为这个转变不是去掉了对判断力的需要，而是把这份判断力的风险提高了。

这两个好处互相强化。自主是用来释放你的注意力，而架构师的角色就是你把那份释放出来的注意力放下去的地方。两者加在一起，就描绘出 Loop Engineering 的核心承诺：一个小而设计得好的控制系统，可以管理 agent 的工作，让你可以去想系统，而不需要去一个回合一个回合地把方向盘。

还有第三个好处，是从前面两个衍生出来的。对于合适的任务来说，一个 loop 不只省力，还会做出一个比人工给提示更好的结果。原因就是耐性。很多工程任务都不是一步天才就解得了的。它们是靠很多细小的、经过验证的打磨，慢慢收敛出来的。一个亲手给提示的人，在任务还没跑完那些迭代之前，老早就没了耐性。烦气上来，注意力飘走，“够好了”就早早出现。

一个 loop 就没有这种限制，除了它的预算之外。它会跑测试套件、读那个失败、调整，然后在第十次迭代，用和第一次一模一样的严谨去再试。当一个任务的质量会随着迭代而改善，而这个 loop 又有一个分得清改善和退步的 evaluator 的时候，那种不知疲倦、划一的坚持，本身就是更好结果的一个来源。而这正正就是为什么我们即将要讲的那几类工作，全部都是带有迭代性质的。

那么，什么时候才应该用 loop 呢？Loop 不是每一个任务的合适工具。一个一次性的问题、一个单一又理解得很透彻的修改，或者任何你没办法用机器查得到的方式去描述“完成”的东西，通常都是直接给提示去推那个 agent 会更好。Loop 的最佳位置有一个认得出来的形状，而这个形状正正是由我们前面讲的好处直接引申出来的。

第一，这个任务又长又带迭代性。它是靠很多细步去收敛，而不是一步，所以自主、有耐性的迭代才回本。第二，“完成”有一个清晰的 verification signal。有一个客观的检查，好像一个测试套件、一次构建，或者一个类型检查器，可以让 evaluator 用来为每一次尝试把关，让自主保持负责任而不是鲁莽。第三，这份工作有足够的价值，去证明值得围着它设计一个系统。那个从操作员转做架构师的投资，在任务又大又会重复出现的时候才划算，而不是在一句提示就搞定的时候。

有三类工作特别适合这个形状。每一类都是一个长、带迭代的任务，而且有一个强而现成的 verification signal，这正正就是它们每一个都适合用 loop 的原因。

我们从重构讲起。重构的意思是在不改变行为的前提下，去改变代码的结构，而这个定义就免费地给了你一个 verification signal：现有的测试套件就是那个判准。如果那些测试之前能过，现在又仍然能过，那么行为就保留下来了。一个 loop 可以很用力地靠着这个信号，用细步去重构，每改完一步就再跑一次套件，只接受那些让套件保持绿色的改动。

重构适合用 loop 有几个原因。它的 verification signal 异常地干净。和新的工作不同，你问的不是“这个东西对不对”，你问的是“这个东西还对不对”，而那些本身已经存在的测试就客观地回答得了你这个问题，不需要再写一份新的评分准则。它也天生就带迭代性而且很枯燥。跨模块之间改名、抽出函数、解开一个依赖关系——这些都是千刀万剐、又小又机械的修改累积出来的。那种枯燥，正正就是磨蚀一个人工操作员的东西，也正正是一个 loop 完全不在乎的东西。而隔离就让它安全。因为每一次尝试都在一个用完即弃的 worktree 里面跑，一个不小心搞坏了行为的重构，会被失败的测试抓到，然后被丢弃，所以它永远都不会进到 main。

接着是功能实现。实现一个功能，无论是一个新的端点、一个新的组件，还是一个新的命令，当这个功能可以表达成一个可以审核的结果的时候，就适合用 loop。这里的 verification signal，通常是一组在前头写好的验收测试。你把你想要的行为描述成测试，你让这个 loop 去让这些测试通过而又不弄坏现有的套件，然后你放手交给它迭代直到绿色为止。

功能实现适合用 loop，是因为它有一个界定得了的 verification signal。一个这样描述的功能——“这个端点成功时返回 201，输入无效时返回 422，而且所有现有测试仍然通过”——就给了 evaluator 一个毫不含糊的目标。它也天生带迭代性。真实的功能很少第一次尝试就对。它们是靠“写测试—修改—修正”的循环一层层建立起来的，而那个循环就是这个 loop。把它自动化是直接合身，而不是硬套。而且范围可以界定得了。一个单一而连贯的功能，是一个大小刚刚好的工作单位，窄到让反馈路径每一次迭代都有具体的东西可以推，又有足够的价值去证明值得设定一番。

这里有一个提醒，是指向下一章的。这个功能越是开放没边际，这个 verification signal 就越弱，而这个 loop 就越有风险做出“有量无质”的东西。有清晰验收准则的功能，才是强烈合身的那些。模糊、探索性的那些，就是工程判断力要继续留在 loop 里面的地方。

最后，调试。调试可能是所有之中最天生合身的那个，因为一个 bug 通常会自己宣告它的 verification signal：那个失败的测试、那个重现案例，或者是那个错误本身。目标自己就写好了：让这个失败的案例通过，而又不弄坏任何其他东西。而这个 loop 就可以直接针对着这个信号去迭代。

调试适合用 loop，是因为这个 verification signal 本身已经内建在那里。一个可重现的 bug，按定义来讲，就是一个目前失败、但应该要通过的检查。Evaluator 的工作是由这个 bug 自己交给它的，不需要设计任何评分准则。它也本质上就是一个“猜完再查”的循环。诊断一个缺陷，意思是建立一个假设、改一点东西、再跑一次去看看那个症状消失了没有，一次又一次地迭代。这就是套用在一个缺陷之上的“推理—行动”循环，而它和 loop 的生命周期几乎完美地对应。而 evaluator 就守住不让有退步。一个修正，如果解决了这个 bug 但弄坏了其他三个测试，会自动被拒绝，因为整个套件都为接受把关。这个 loop 在结构上就被阻止了去用一个 bug 换另一个 bug。

在这三类之中，同一个模式都成立。一个长、带迭代、又有清晰 verification signal 的任务，就是那个典型的、loop 形状的问题。当你发觉自己正面对着这样的工作，而这个任务又有足够的价值去证明值得围着它设计一个小小的控制系统的时候，那就是你应该伸手去拿这个 loop、而不是去那个提示框的信号。

那我们就把这一章的重点归纳到一起。一个 loop 的回报，不是“同样的结果、自动化了”。它同时改变了 agent 在没有你的情况下能完成什么，以及你把自己的注意力放在哪里。

第一个好处是自主推进。因为这个 loop 在供应它自己的下一个提示，工作在没有人待在内层循环的情况下继续下去，在没有人看着之下也推进着有意义的工作。这就是“在你睡觉的时候”的承诺。而一个独立的 evaluator，就是让那种自主变得负责任的东西，因为这个 loop 只保留经过验证的进展，而不是一堆没审核过的乱猜。

第二个好处是从操作员转做架构师。搭一个 loop，把你从一个一个回合一个回合驾驶 agent 的操作员，升做设计那个去驾驶它的系统的架构师。你的判断力移到了更早、更高的位置，去到目标、评估准则和护栏里面，然后再在每一次迭代之中一致地应用。

第三个好处是在长任务上有更好的结果。一个 loop 用一种划一、不知疲倦的耐性去迭代，是一个人工操作员根本撑不住的那种。所以对于那些靠很多次经过验证的打磨去改善的任务，它可以做出一个更好的结果，而不只是一个更便宜的结果。

还有，记住，一个任务是 loop 形状的，当它又长又带迭代、当它有一个 evaluator 可以把关的清晰 verification signal、以及当它有足够的价值去证明值得围着它设计一个控制系统的时候。重构、功能实现和调试，全部都适合这个形状。重构适合，是因为现有的测试套件就是一个免费的、保留行为的判准。功能实现适合，是因为验收测试，为一个天生带迭代的构建，给了一个可以审核的目标。而调试适合，是因为一个可重现的 bug，自己就供应了一个“先失败、后通过”的 verification signal。
