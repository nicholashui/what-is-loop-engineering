在这一章，你会学会怎么建造一个属于自己的 loop 库。它是一个经过策展、纳入版本控制的可重用 loop 定义集合，给你的团队去浏览、复制和运行。我们会定义一个单一的 loop 条目 schema，让每一个 loop 都用同一种方式去描述。我们会铺排一个储存它们的储存库结构。我们会展示怎么让这个库可被搜索。而最重要的是，我们会展示怎么用"建造"那一章的机制，把一个静态的库条目变成一个运行中的 loop。读完之后，你就有能力把上一章那五十个形态，或者你自己的，变成一个可执行的库。

那就从分辨"目录"和"库"讲起。上一章是一个目录，一次只读的导览，看其他人写的 loop。而一个 loop 库是更强的东西。它是你自己储存库里面一件活生生的产物，其中每一个条目都是结构化数据，而不是散文，于是它可以被搜索、被验证、被版本化，并且——最关键那一步——被执行。这个分别，跟本书一直反反复复讲的那个一样。散文描述一个 loop。而一个库就定义它，精确到足以让一台机器运行得起来。

实作一个库，意味着要做三个决定，然后把它们接驳到一起。第一，一个 schema，一种一致的形状去描述每一个 loop，让这些条目可以互相比较、又可被机器读取。第二，一个储存，这些条目住在哪里，以及它们怎么被组织、被版本化、被审查。第三，一个 runner，一个库条目怎么变成一个由八个组件搭成的运行中 loop。

让我们按这个次序逐个讲，先从 schema 开始。一个库的基础，就是一份大家一致同意的描述，讲清楚一个 loop 是什么。请留意，上一章每一个条目都隐含地答了同样的问题：谁写、它做什么、它怎么运作、它什么时候停。把这些问题变成明确的字段，你就有了一个 schema。一个好的 schema 会映照回八个组件，让一个库条目跟一个可运行的 loop 一对一地对应。

实际的形式，就是一个 Markdown 文件加 YAML frontmatter。它让人看得明白，又让机器解析得到，而这个正好就是一个库条目所需要的东西。这个 frontmatter 带着身份和发现的字段，例如 id、name、author 和 category，然后就是 loop 契约本身：触发器、生成器、评估器、隔离策略、状态文件、停止条件、连接器和技能。这些正好就是八个组件。所以一个库条目，就是一份完整、声明式的 loop 描述。关于怎么运行它的任何东西，都不会住在这个文件之外。

有两个设计原则让这件事行得通。第一，每一个组件都是一个字段，于是这个条目就是那份契约。第二，prompt 是一个引用，而不是内嵌文字。实际的 agent 指令住在一个独立的 prompt 文件里，这样可以让 prompt 跨 loop 重用，并且可以独立地编辑。

不是每一个 loop 都需要每一个字段，但有一小撮核心字段应该是强制的，这样这个库才永远不会含有一个运行不了的条目。id、name、author 和 category 是必需的，用来做身份和发现。purpose 是必需的，作为一句话、可被检查的意图。生成器的 prompt 文件是必需的，因为没有生成器就没有 loop。而停止条件是必需的，因为一个无边界的 loop 是一个缺陷，不是一个条目。评估器就强烈建议要有，因为没有它，"完成"就只是 agent 自己说的。最重要的一条规则就是：没有任何条目可以缺少停止条件。这一条约束，正好就是让一个由众多自主 loop 组成的库，不会变成一个由众多失控进程组成的库的关键。

现在讲储存。定了 schema 之后，储存就很直接：一个在版本控制之下、由条目文件组成的目录。把纯文件放入 git，你就免费得到历史、审查、分支和 diff，而这些特性，正好就是让 loop 本身值得信任的同一批特性。这个结构是一个 loop 一个文件，按类别分文件夹，映照回上一章那五大类。prompt、评分准则和技能都是共享、一等公民的文件，于是同一个评分准则可以支撑很多个 loop。而一个可被机器检查的 schema 就住在储存库里面，于是 CI 可以在每一次改动时验证每一个条目。这个就是这个库把一个评估器套用在自己之上。

这个库应该以它所宣扬的标准去要求自己。一个在合并前对照 schema 去验证每一个条目的检查，是最低门槛。它执行三条不可妥协的东西：一个有效的形状、一个存在的停止条件，以及一个存在的 prompt 文件。把它放入 CI，这个库就永远不可以合并一个运行不了、或者会永远运行下去的条目。

接着，让它可被搜索。一个库，只有当人们可以快速找到合用那个 loop 的时候，才无愧于它的名字。因为每一个条目都带着结构化的 frontmatter，所以搜索大致上就是读取那些元数据、再把它投射成可浏览的东西的问题。两个低成本的机制，几乎涵盖了所有需要。有一个生成的索引：一个小巧的脚本读取每一个条目的 frontmatter，写出一个分组、有链接的 README，就像上一章那个目录一样。还有分面、基于标签的筛选：因为 category、tags 和 maturity 都是结构化字段，你不用一个数据库都可以筛选。那些元数据就是这个搜索索引。

现在讲那一步——分隔一个真正的库和一个花哨笔记文件夹的地方——把一个库条目变成一个运行中的 loop。一个库条目是一份声明。运行它，就意味着把那份声明交给一个 runner，由它去组装八个组件、并驱动生命周期。它就是"建造"那一章那个一模一样的生产级 loop，只是现在由条目去参数化，而不是写死。

runner 的工作是一次忠实的翻译，逐个字段来。触发器决定什么时候开始。隔离策略每次尝试在一个用完即弃的分支上建立一个 git worktree。生成器的 prompt 文件和输入去渲染 prompt 并调用 agent。评估器运行客观关卡，然后评论者或者评分准则关卡。状态文件持久化进度，并把它循环进下一个 prompt。而停止条件就决定是要继续、合并并停止，还是升级。每一个都直接对应到生命周期的一个阶段。

重点不是这个特定的脚本。重点是那份分离。runner 只写一次，并包含了所有的编排。库条目就只包含声明。加多一百个 loop，runner 都不用改。这个就是 schema 带来的回报。一个 loop 变成了数据，而运行任何一个 loop，都是同一条程式码路径。

这里也是参数化 loop 出场的地方。很多目录 loop 都有空格，例如"N 次成功之后停"、"repository task"，或者"quality threshold"。一个库让这些空格变成一等公民的输入，在运行时供应。于是你可以把同一个条目针对不同目标运行，不用改动条目本身。一个参数化得当的条目，可以取代成打近乎重复的副本。这个就是一个库怎么在增长之中保持小巧的方法。你加的是参数，而不是分叉。

那么，你怎么去到你第一个库呢？你不用事前就建好整个库，就像你在"建造"那一章也不是事前建好整个 loop 一样。用同样的减法方式去培育它。第一，晋升一个行得通的 loop：拿一个你已经运行着的 loop，用 schema 写出来，并把它作为第一个条目提交。第二，runner 只写一次，方法是把你现有的编排器推广。第三，把 schema 验证器加入 CI，这样这个库就不可以接受一个破损或者无边界的条目。第四，只有在你真正重用之际才晋升 loop。第五，生成索引，让这个库可被浏览。纪律跟本书其他地方一样：当"没有把一个 loop 放入这个库"会造成真正的重复的时候，才把它加入，不要早过这个时候。

那让我们把这些关键想法收拢到一起。一个 loop 库把那个只读目录，变成一件活生生、可执行的产物，其中每一个 loop 都变成结构化、纳入版本控制的数据，而不是散文。实作一个，意味着要把三个决定接驳到一起：一个 schema，每一个 loop 一种形状；一个储存，纳入版本控制的文件；以及一个 runner，执行任何条目。

这个 schema 应该让每一个组件都成为一个明确的字段，并把 prompt 以引用文件的形式储存，而且没有任何条目可以缺少停止条件。这个储存是一个 git 目录，由"一个 loop 一个文件"、按类别分文件夹的条目组成，配上共享的 prompt、评分准则和技能，以及一个在 CI 中验证的 JSON Schema。搜索几乎免费地由结构化的 frontmatter 得来。这个 runner 只写一次，把一个条目的字段翻译到生命周期上。它就是那个生产级 loop，只是把它的常数换成由条目读取，于是加 loop 永远不会改动编排。参数化的输入把目录那些空格变成运行时的参数。而你就用减法去建这个库：晋升一个行得通的 loop、写这个 runner、加入 CI 验证，然后只有在重用真正有需要的时候才加条目。
