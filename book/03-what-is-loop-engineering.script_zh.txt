在这一章里，你会学到 Loop Engineering（循环工程）以及它所构建出来的 Loop（循环）的精确定义。你会清楚地看到，这种做法和你一个回合一个回合、用手动方式去让一个 AI coding agent（AI 编程代理）帮你干活，到底有什么不同；你也会建立起一种牢固的直觉，明白 Loop 究竟是什么，以及它为什么值得被称为 Loop。这一章会引出本书第一个、也是最基础的学习目标：就是能够讲清楚 Loop 是什么，以及它和手动提示有什么区别。

我们先从一个精确的定义讲起。Loop Engineering 是一种做法，就是设计一个控制系统，由它来自动地提示、观察、验证和迭代 AI coding agent，而不是由你一个回合一个回合地用手动方式去提示它们。

这一句话承载了很多含义，所以值得一部分一部分拆开来讲。

首先，是一个控制系统。你所构建的，是一个系统，而不是一句聪明的提示。它是一小块软件和配置，它的职责就是让一个 agent 持续、高效地朝着目标去工作。

接着，它会提示、观察、验证和迭代。这四个动词就构成了整个循环。系统向一个 agent 发出指令，这就是提示。它观察 agent 产生出来的东西，这就是观察。它检查那些输出究竟是不是真的好，这就是验证。然后它用这个结果去推动下一次尝试，这就是迭代。

再往下就是 AI coding agent。这个系统会统筹一个或者多个 AI coding agent，也就是你今天已经会用手去提示的那种助手。

最后，是自动地运作。一旦你设计好了，这个系统就会自己跑这个循环，不需要有人去一句一句地写指令。

这种做法所构建出来的产物，也有它自己的名字。一个 Loop，就是由一个 loop engineer（循环工程师）所搭建的统筹系统。它会发掘工作、生成 agent、验证它们的输出、保存状态，并且决定要继续还是停下。在整本书里，Loop Engineering 指的是这套学问，也就是一种思考和设计的方式；而 Loop 就是你所搭建出来的那个东西。把这两者分清楚，本书其余部分就能保持清晰：你做的是 Loop Engineering，而你最终得到的就是一个 Loop。

这里有一个澄清，可以帮你避开一个常见的混淆。Loop 并不是 agent。那个 AI coding agent，也就是本书所说的 Generator Agent（生成代理），也就是真正去产生代码和改动的那个部件，它只不过是 Loop 的其中一部分。Loop 是周围的机制，由它来决定 agent 应该做什么、去运行它、去评判结果，再不断重复。当本书说"Loop 决定"或者"Loop 验证"的时候，意思是指那套周围的机制，而不是里面那个 agent。

现在我们来看一下手动提示和 Loop 之间的对比。要理解 Loop Engineering，最快的方法就是把它直接和它所取代的东西放在一起比较。

想一想手动地、一个回合一个回合地提示是怎么运作的。当你用手去提示一个 AI coding agent 的时候，你自己就是那个控制系统。想一想在一次工作过程中你实际上在做些什么。你把目标记在脑子里，你知道"做完"是什么样子。你打一段提示，描述下一步要做什么。你看 agent 的输出，再判断它是不是够好。如果是错的，你就写一段更正，再提示一次。你会记住已经试过什么，这样才不会绕圈子。然后你会决定什么时候这件工作算做完，又或者什么时候应该放弃。

这些步骤每一个都靠你的注意力去推进。Agent 负责生成；而所有判断都由你来提供。对于短期的、探索性的工作来说，这种方式很强大，但它有一个很硬的天花板：一旦你走开，进度就会立刻停下。你是在最字面的意义上，在那儿看着这个 agent，每一个回合都得在场，因为每一个回合都需要一个只有你才能做的决定。

现在来看一下一个 Loop 是怎么运作的。Loop Engineering 就是把你在脑子里做的那一串事情，一样一样地变成明确的、写下来的形式，好让一个系统可以替你去做。那些判断不会消失；它们只是从你的注意力里面，搬到了 Loop 的设计里面。

所以，以前你把目标记在脑子里，现在 Loop 有一份写下来的目标和评估标准。以前你一句一句地打下一段提示，现在 Loop 有一个自动触发并提示的步骤。以前你判断输出是不是够好，现在 Loop 有一个 Evaluator（评估器），它可能是测试、是一个 critic agent（评审代理），又或者是一份评分标准。以前你记住试过什么，现在 Loop 有保存在磁盘上的状态和记忆。以前你决定什么时候停下，现在 Loop 有一个明确的停止条件。

这就是问题的核心，也值得用最直白的方式说出来：你不再当那个看护的人，而是去搭建那个会提示、观察、验证和迭代的循环。你的角色从操作员，也就是那个打下一句指令的人，转变成系统架构师，也就是那个设计"会打指令的系统"的人。你只是在最开始、在设计的时候花一次判断，而不是持续地、一个回合接一个回合地去花。

区别不在于一个 Loop 比你小心地提示更聪明。区别在于由谁去跑这个循环。一个 Loop 不需要你也能跑起来，也就是说它可以在你睡觉的时候跑，可以跑上几百次迭代，也可以去跑那些又长又磨人、又重复、闷到你不想用手去看护的工作。

我们来建立这种直觉。为什么叫它 Loop？这个名字不是一个比喻；它是对这个机制的描述。这个系统会循环，它会重复一个循环，而正是这种重复，才让它如此有价值。

把这个循环想象成一个一次又一次转动的圆圈。每转一次，Loop 就提示这个 agent，于是 agent 行动。然后它观察，去读这个输出。然后它验证，去问一下这个结果是不是好。然后它做一个决定。如果这件工作还不够好，它就把结果反馈回去，包括任何错误或者没通过的检查，再转一圈。如果这件工作达到了目标，又或者撞到了某个上限，它就停下。

这正是你在一次手动过程中，在自己脑海里跑着的那个循环。你提示、你阅读、你判断、你更正，然后不断重复，直到你满意为止。Loop Engineering 只不过是把这个循环从你的脑海里提取出来，再变成一个可以自己去跑的系统。一旦你清楚地看到这个循环，整套学问就会自然而然地跟上来。后面各章所讲的一切，都是关于怎样把这个圆圈的每一部分都设计好：工作怎样进来、agent 怎样在里面安全地运行、验证那一步怎样决定什么才算够好、进度怎样在回合之间被记住，以及 Loop 怎样知道什么时候应该停下。

记住这个画面。一个 Loop 就是一个会自己转动的圆圈，而 Loop Engineering 就是设计一个值得放手让它转动的圆圈的手艺。

我们来回顾一下重点。Loop Engineering 是一种做法，就是设计一个控制系统，由它来自动地提示、观察、验证和迭代 AI coding agent，而不是由你一个回合一个回合地用手动方式去提示它们。

而 Loop 就是这种做法所构建出来的东西：那个统筹系统会发掘工作、生成 agent、验证输出、保存状态，并且决定要继续还是停下。Loop 并不是 agent。Generator Agent 只是它里面的其中一个部件。

和手动提示最关键的对比，就在于由谁去跑这个循环。在手动提示里，你就是那个控制系统，一个回合一个回合地提供所有判断。在一个 Loop 里，那些判断会变成明确的、写下来的部件：一个目标、一个 Evaluator、保存下来的状态，以及一个停止条件，好让这个系统不需要你也能跑起这个循环。

说白了：你不再当那个看护的人，而是去搭建那个会提示、观察、验证和迭代的循环。你的角色从操作员转变成系统架构师。

而这个名字是字面上的意思。一个 Loop 真的会循环，它会重复一个"提示、观察、验证、迭代"的循环，直到这件工作达到它的目标，或者撞到某个上限为止。那个不断重复的圆圈，就是你要带着走完本书其余部分的心智模型。
