In this chapter, we're going to open up a Loop and name its parts. By the time we're done, you'll know the eight components that make up a Loop. They are the Automation, or Trigger; the Worktrees, which handle isolation; the Generator Agent; the Evaluator; State, or Memory; Skills and Knowledge; Connectors; and finally the Stopping Condition. For each one, you'll learn its purpose, the single job it does inside the Loop, and you'll see at least one concrete example of how that job gets filled in practice. By the end, you'll be able to point to every building block of a Loop and explain why each one earns its place.

Earlier, we gave you the image of a Loop as a circle that turns on its own. It prompts, it observes, it verifies, and it iterates. And we showed how that circle was lifted out of a single agent conversation and built into a system. Now it's time to open the machine and name the parts.

A Loop is assembled from eight components. Each one corresponds to a judgment or a job that, in manual prompting, you would do yourself. Deciding when to start. Keeping work from colliding. Doing the actual coding. Checking whether the result is good. Remembering what has happened. Knowing the project's rules. Talking to the outside world. And deciding when to stop. Move each of those into an explicit part of a system, and you have a Loop.

We'll take them one at a time, in the order that work flows through them. For every component, the structure is deliberately identical. We'll cover what it is, what its purpose is, and at least one concrete example. That way you can hold all eight in your head as variations on a single pattern: a part that does one job, so the Loop doesn't need a human to do it.

Let's start with the Automation, or Trigger.

The Trigger is the component that decides when and how work enters the Loop. It's the starting gun, the mechanism that kicks off an iteration without a human typing the word go. In manual prompting, you start every turn. You decide there's work to do, and you begin it. The Trigger moves that decision into the system, so the Loop can begin work on a schedule, or in response to an event, rather than waiting for a person to initiate it. This is what lets a Loop run while you sleep. The work starts itself.

A Trigger can take a few different shapes. It might be a scheduled job, for instance a task that fires every night at two in the morning to pick up the next item of work. It might be a webhook tied to a new issue, so the moment an issue is filed in the tracker, the webhook fires and the Loop begins working on it. Or it might be a simple command, something like typing slash-loop, where a human, or even another system, types a single command to launch the Loop on demand. Picture, for example, a trigger set up as a nightly schedule that quietly starts the Loop unattended every day at two a.m. The common thread across all of these is that the decision to begin is encoded ahead of time, rather than performed live by a human.

Next comes Worktrees, and isolation.

This component keeps each agent run in its own sealed-off workspace, so the changes one run makes can't trample the changes another run is making. Here's why that matters. A Loop may run many iterations, and sometimes several agents at once. Without isolation, two runs editing the same files at the same time would corrupt each other's work, and a bad run could damage a shared workspace in a way you can't undo. Isolation prevents agents from stepping on each other, and it keeps every run safe and reversible. If an iteration goes wrong, you simply discard its isolated workspace, and nothing else is affected.

The standard tools for this are git worktrees and separate branches. With worktrees, each agent run gets its own working directory checked out from the same repository, so its edits are physically separate from every other run's edits. With separate branches, each run commits to its own branch, so its history is isolated and can be reviewed, merged, or thrown away independently. Imagine giving iteration forty-two its very own directory and its own branch, where its agent works in complete privacy, untouched by any other run. Isolation is what turns many runs from a liability into a feature. Runs can proceed in parallel, and they can fail harmlessly.

Now the heart of the machine: the Generator Agent.

The Generator Agent is the AI coding agent that produces the actual output of the Loop. The code, the edits, the changes. It's the worker inside the machine. This is the component that does the hands-on work a human would otherwise do, turn by turn. And here's a point worth holding onto: the Loop is not the agent. The Generator Agent is just one part of the Loop, the part responsible for generating output. Everything else in this chapter exists to decide what the Generator Agent should do, to keep it safe, to check its work, and to decide whether to run it again.

A Generator Agent is filled by an off-the-shelf AI coding agent. It might be a tool like Claude Code, running as the agent that reads the task and edits the codebase. Or it might be a Grok agent, playing exactly the same generating role within the Loop. Picture invoking that agent for a single iteration: it reads the current task, then produces its edits inside its own isolated workspace. The Loop doesn't care which agent fills this slot, only that something does the generating. The agent is interchangeable. The role is fixed.

After the work is done, it needs to be judged. That's the Evaluator, sometimes called the critic or the verifier.

The Evaluator is the component that checks the Generator Agent's output against some criteria and decides whether it's good enough. It's the independent judge. This is one of the most important things a Loop adds over a bare agent run. In a single agent conversation, the agent is both the worker and the only judge of its own work. A Loop adds an external Evaluator, so that done is verified rather than merely assumed. The Evaluator is what protects a Loop from confidently producing wrong or low-quality output, because nothing gets accepted until something other than the generator has checked it.

Evaluation can take several forms, and often they're used in combination. You might have a separate agent that runs the tests, executing the project's test suite and treating a failure as a rejection. You might have a code-review or critic agent, a second agent that reads the changes and reviews them for quality, correctness, and style. Or you might have a rubric scorer, which scores the output against an explicit rubric and compares that score to a threshold. Think of a test-based Evaluator as a gate: if the tests pass, the iteration is accepted; if they fail, the errors get fed back and the Loop tries again. Whatever its form, the Evaluator's job is the same. Judge the work from the outside, and hand back a verdict the Loop can act on.

Next is State, or Memory.

This component persists the Loop's progress outside any single agent's context, so information survives from one iteration to the next. A single agent run only remembers what's in its context window, and that memory vanishes the moment the run ends. For a Loop to make real, cumulative progress across many runs, it needs memory that outlives any one run. A record of what has been done, what's left, and what was learned. State and Memory is what lets iteration forty-two build on what iteration forty-one accomplished, instead of starting from scratch.

State is typically kept in durable, inspectable places. It might be a to-do file that records the remaining work and gets updated each iteration. It might be a scratchpad where the Loop jots down findings and decisions between runs. It might be a project board in a tool like Linear or Notion that tracks the status of each unit of work. Or it might just be plain files on disk that capture intermediate results. Picture a simple to-do file where one item is checked off as finished in iteration forty-one, while the next unchecked item is teed up for iteration forty-two. And because state lives outside the agent, it's also something you can read yourself, to see exactly where the Loop stands.

Then there's Skills, and Knowledge.

This component is the codified, reusable guidance a Loop supplies to its agents. The project's rules, conventions, and patterns, written down where an agent can actually use them. Here's the problem it solves. An agent that doesn't know your project's standards will reinvent them inconsistently, run after run. Skills and Knowledge captures that guidance once, so every iteration follows the same rules, instead of relying on a human to repeat use our naming convention, or always write tests, in every single prompt. It's how a Loop stays aligned with how your team actually works.

This guidance is usually stored as files the agent reads. It might be a skills file describing reusable procedures the agent should follow. It might be an agent-guidance file that tells agents how to operate in this particular repository. Or it might be written coding standards the agent has to adhere to. Imagine a short guidance file that tells the agent to follow the existing module layout, to write a unit test for every new function, and to prefer the project's logging helper over a raw print statement. Skills and Knowledge is the difference between an agent that guesses your conventions and one that has them right in front of it.

Now we reach the Connectors.

Connectors are the set of integrations that let the Loop talk to external systems, the tools your team already uses to track and ship work. A Loop doesn't operate in a vacuum. It has to read work from somewhere, write results somewhere, and tell people what's happening. Connectors give the Loop hands and a voice in the outside world. They let it pull a task from a tracker, open a pull request, or post an update, so the Loop becomes part of the real workflow rather than an island.

Common connectors include a source-control host like GitHub, used to read code and open pull requests. They include an issue tracker, used to pull the next unit of work and update its status. And they include a chat platform like Slack, used to announce progress or to escalate when human attention is needed. Picture a Loop wired into all three at once, reading code and opening pull requests through GitHub, pulling tasks and updating their status through the issue tracker, and posting progress to a chat channel. Connectors are what make a Loop a participant in your existing systems, rather than a sealed experiment.

And finally, the Stopping Condition.

The Stopping Condition is the rule that tells the Loop when to stop iterating, or when to escalate to a human instead of continuing. A Loop repeats, and repetition without a defined exit is dangerous. It can spin forever, burn through your budget, and produce nothing useful. The Stopping Condition gives the Loop a clear definition of finished, or of give up and ask for help, so it terminates deliberately rather than running indefinitely. It's the component that keeps a Loop from becoming a runaway process.

A stopping condition is usually one or more explicit rules. One common rule is that the tests pass: once the Evaluator's test suite comes back green, the goal is met and the Loop stops. Another is that a rubric score reaches a threshold, for example, stopping once the score is at least nine out of ten. And another is a safety limit, like a maximum number of iterations or a spending budget. For instance, stopping after twenty iterations, or once the cost budget is exhausted, and escalating to a human rather than continuing. Picture a combined check that runs each iteration: if the tests pass, it stops because the goal is met; if it has hit twenty iterations, it stops and escalates to a human; and if it's over budget, it stops and escalates as well. The first rule defines success. The others define safety. Together, they guarantee the circle stops turning.

So how do all eight fit together? Read in order, they trace the path of a single unit of work through the Loop. The Trigger starts an iteration. Worktrees and isolation give that iteration a safe place to run. The Generator Agent does the work there, guided by Skills and Knowledge, and reaching the outside world through Connectors. The Evaluator judges the result. State and Memory records what happened, so the next iteration can build on it. And the Stopping Condition decides whether the Loop goes around again, or finishes.

You don't need every component in every Loop. Later on, we'll look at a minimal Loop that has only a few of them. But the eight together form the complete vocabulary of parts. Knowing what each one is for is what lets you decide which to include, which to combine, and which to leave out when you design a Loop of your own.
