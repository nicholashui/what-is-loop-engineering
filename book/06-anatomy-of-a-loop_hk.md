# 06. 循環的解剖：各個組成部分

> **本章內容：** 你會認識構成一個循環（Loop）的八個組成部分——
> *自動化／觸發器（Automation/Trigger）*、*工作樹／隔離（Worktrees/Isolation）*、*生成代理（Generator Agent）*、
> *評估器（Evaluator）*、*狀態／記憶（State/Memory）*、*技能／知識（Skills/Knowledge）*、*連接器（Connectors）*，以及
> *停止條件（Stopping Condition）*。就每一個部分，你會了解它的用途——它在循環裏面所做的
> 那一份工作——並至少看到一個具體例子，說明那份工作在實務上是怎樣達成的。讀完本章之後，你就能夠辨認出
> 循環的每一個構建組件，並解釋為何每一個都各得其所。

## 機器的各個部件

第 03 章為你描繪了一個循環的圖像：一個會自行轉動的圓圈——
提示、觀察、驗證、迭代——而第 04 章則展示了這個圓圈是怎樣從單一代理對話中
抽離出來，並建構成一個系統。本章會打開這部機器，逐一為各個部件命名。

一個循環由**八個組成部分**裝配而成。每一個都對應到一項判斷或一份工作；在人手提示的時候，
這些工作本來是你親自去做的：決定何時開始、避免工作互相碰撞、做實際的編碼、檢查結果
是否夠好、記住已經發生過的事、知道專案的規則、與外部世界溝通，以及
決定何時停止。把每一項都搬進系統中一個明確的部件裏，你就得到一個循環。

我們會逐一去看，按照工作流經它們的次序。就每一個
組成部分，結構都刻意保持一致——**它是甚麼**、**它的用途**，以及**至少一個具體例子**——
這樣你就可以把這八個都記在腦海裏，視之為同一個模式的各種變奏：*一個只做一份工作的部件，
好讓循環不需要由人去做那份工作。* 這裏用到的術語跟
[詞彙表](glossary_hk.md)完全一致，而當中有幾個——生成代理、
評估器、停止條件——你在較早的章節已經見過。

## 自動化／觸發器（Automation / Trigger）

**它是甚麼。** 自動化／觸發器是決定工作*在何時、以何種方式*進入循環的組成部分。
它就是起跑槍：一種無需人手打「go」就能啟動一次迭代的機制。

**它的用途。** 在人手提示的時候，是*你*去開始每一個回合——你決定有工作要做，然後動手去做。
觸發器把那個決定搬進系統裏，這樣循環就可以按時間表、或者因應某個事件而開始工作，
而不必等一個人去發起它。這正是讓循環可以在你睡覺時運行的關鍵：
工作會自行開始。

**具體例子。** 一個觸發器可以是：

- 一個 **cron 時間表**——例如，一個每晚凌晨 2 點觸發的任務，去
  接手下一項工作；
- 一個 **針對新議題的 webhook**——當議題在追蹤系統裏被開立的那一刻，
  webhook 就觸發，循環便開始處理它；
- 一個 **斜線式指令**，例如 `/loop`——由一個人（或另一個系統）打出
  單一指令，按需要啟動循環。

```yaml
# 例子：定義為每晚時間表的觸發器。
trigger:
  type: cron
  schedule: "0 2 * * *"   # 每天 02:00 — 無人看管下啟動循環
```

這三者的共通點是：*開始的決定是被編碼下來的*，而非由人即場執行。

## 工作樹／隔離（Worktrees / Isolation）

**它是甚麼。** 工作樹／隔離這個組成部分把每一次代理運行都放進它自己封閉的工作空間，
這樣某一次運行所做的改動就不會踐踏另一次運行正在做的改動。

**它的用途。** 一個循環可能會運行很多次迭代——有時甚至同時有幾個代理在跑。
沒有隔離的話，兩次運行同時編輯相同的檔案，就會破壞彼此的工作，
而一次出錯的運行更可能對共用的工作空間造成無法挽回的損害。
隔離防止代理之間互相干擾，並讓每一次運行都保持*安全而且可還原*：
若一次迭代出了問題，你只要丟棄它那個被隔離的工作空間，其他一切都不受影響。

**具體例子。** 處理這件事的標準工具是：

- **git worktrees**——每一次代理運行都從同一個版本庫取得它自己的工作目錄，
  所以它的編輯跟其他每一次運行的編輯在實體上是分開的；
- **獨立分支**——每一次運行都提交到它自己的分支，所以它的歷史是
  隔離的，可以獨立地被審閱、合併或丟棄。

```bash
# 例子：給每一次迭代它自己被隔離的 git worktree + 分支。
git worktree add ../run-42 -b loop/iteration-42   # 獨立的目錄「同埋」分支
# 第 42 次迭代的代理只在 ../run-42 裏面工作，不會被其他運行碰到
```

隔離正是把「多次運行」由一項負擔變成一項優點的關鍵：各次運行可以
並行進行，而且出錯時不會造成傷害。

## 生成代理（Generator Agent）

**它是甚麼。** **生成代理**是產出循環實際輸出的那個 AI 編碼代理——
代碼、編輯、改動。它就是機器裏面的工人。

**它的用途。** 這正是負責去做那些原本由人逐個回合去做的親力親為工作的組成部分。
正如第 03 章所確立的，循環*並不是*那個代理；生成代理只是循環的*其中一個部件*——
即是負責生成輸出的那個部件。本章其餘的一切之所以存在，都是為了決定生成代理
應該做甚麼、保障它的安全、檢查它的工作，以及決定
要不要再運行它一次。

**具體例子。** 生成代理由一個現成的 AI 編碼代理擔當，例如：

- **Claude Code**——作為那個讀取任務並編輯
  代碼庫的代理運行；
- **一個 Grok 代理**——在循環裏面擔當同樣的生成角色。

```bash
# 例子：為其中一次迭代調用生成代理。
claude-code --task "$(cat current-task.md)" --workdir ../run-42
# 代理讀取任務，並在它被隔離的 worktree 裏面產出編輯
```

循環並不在意是*哪一個*代理填補這個位置——只在意有東西去做
生成的工作。代理是可以互換的；角色則是固定的。

## 評估器／批評者／驗證者（Evaluator / Critic / Verifier）

**它是甚麼。** **評估器**——又稱為批評者（critic）或驗證者（verifier）——是
那個按照某些準則去檢查生成代理輸出、並決定它是否夠好的組成部分。它就是那位獨立的裁判。

**它的用途。** 這是循環相對於一次裸跑代理運行所作出的其中一項最重要的增補。
在單一代理對話裏，代理同時是工人*又是*評斷自己工作的唯一裁判；
循環則加入一個**外部的**評估器，這樣「完成」就是被*驗證*出來，而非僅僅被*假定*。
評估器正是用來保護循環，避免它信心十足地產出錯誤或低質的輸出，
因為在生成器以外有東西檢查過之前，沒有任何結果會被接受。

**具體例子。** 評估可以採取幾種形式，往往會組合使用：

- **一個專門去跑測試的獨立代理**——執行專案的測試套件，
  並把失敗當作不予接受；
- **一個代碼審閱／批評代理**——第二個代理去讀取 diff，
  就質素、正確性與風格作出審閱；
- **一個評分準則計分器**——按照一套明確的評分準則去為輸出計分，並把
  分數跟某個門檻作比較。

```bash
# 例子：以測試為本的評估器把關，決定是否接受生成器的輸出。
if npm test; then
  echo "PASS — 接受這一次迭代"
else
  echo "FAIL — 把錯誤回饋並繼續迭代"
fi
```

無論採取哪一種形式，評估器的工作都一樣：*從外部去評斷工作*，
然後交回一個循環可以據之行動的裁決。

## 狀態／記憶（State / Memory）

**它是甚麼。** 狀態／記憶這個組成部分把循環的進度持久化儲存在任何單一代理的脈絡*之外*，
這樣資訊就能從一次迭代延續到下一次。

**它的用途。** 一次代理運行只記得它脈絡視窗裏面的東西，
而那份記憶會在運行結束時消失。一個循環要在許多次運行之間取得真實、累積的
進展，就需要*比任何一次運行都活得更久*的記憶——一份記錄，
記下已經做了甚麼、還剩下甚麼，以及學到了甚麼。狀態／記憶正是讓
第 42 次迭代可以在第 41 次迭代所完成的基礎上繼續，而不必從頭開始的關鍵。

**具體例子。** 狀態通常存放在耐久、可檢視的地方：

- 一個 **TODO 檔案**（例如 `TODO.md`），記下尚餘的工作，並
  在每一次迭代時更新；
- 一個 **草稿簿（scratchpad）**，讓循環在各次運行之間記下發現與決定；
- 一個在 Linear 或 Notion 之類工具裏的 **專案看板**，追蹤每一個
  工作單元的狀態；
- 純粹的**磁碟上的檔案**，用來保存中間結果。

```markdown
<!-- 例子：以 TODO.md 作為持久化狀態，在各次迭代之間更新 -->
# TODO
- [x] 為解析器加入輸入驗證   <- 在第 41 次迭代完成
- [ ] 處理空檔案的邊界情況           <- 第 42 次迭代下一步要做
- [ ] 更新文件
```

由於狀態存活在代理之外，它同時也是*你*可以讀取，去確切了解
循環現時去到哪個階段的東西。

## 技能／知識（Skills / Knowledge）

**它是甚麼。** 技能／知識這個組成部分，是循環提供給它的代理、經過編碼整理、可重用的指引——
即是專案的規則、慣例與模式，寫在代理能夠使用得到的地方。

**它的用途。** 一個不知道你專案標準的代理，會一次又一次地不一致地重新發明那些標準。
技能／知識把那份指引*一次過*捕捉下來，這樣每一次迭代都跟從同一套規則，
而不必靠一個人在每次提示裏重複講「用我哋嘅命名慣例」或者「永遠都要寫測試」。
這正是循環跟*你*團隊實際運作方式保持一致的方法。

**具體例子。** 這份指引通常以代理會讀取的檔案形式儲存：

- 一個 **技能檔案**（例如 `SKILL.md`），描述代理應該跟從的
  可重用程序；
- 一個 **代理指引檔案**（例如 `AGENTS.md`），告訴代理在這個版本庫裏
  應該怎樣運作；
- 代理必須遵守的書面**編碼標準**。

```markdown
<!-- 例子：AGENTS.md — 每一次迭代都會吸收的編碼化知識 -->
# 代理指引
- 跟從 `src/` 裏面現有的模組佈局。
- 每一個新函式都需要一個單元測試。
- 優先使用專案的日誌輔助工具，而非 `print`。
```

技能／知識正是「一個靠猜測你慣例的代理」與「一個把你慣例擺在眼前的代理」之間的分別。

## 連接器（Connectors）

**它是甚麼。** 連接器這個組成部分，是讓循環能夠跟外部系統溝通的一組整合——
即是你團隊已經在用、用來追蹤與交付工作的工具。

**它的用途。** 循環並非在真空中運作；它必須從某處讀取工作、
把結果寫到某處，並告訴大家正在發生甚麼事。連接器
給予循環在外部世界裏的雙手和聲音：它們讓循環從追蹤系統拉取任務、
開立拉取請求（pull request），或者張貼一則更新，這樣循環就成為真實
工作流程的一部分，而非一座孤島。

**具體例子。** 常見的連接器包括：

- 一個 **原始碼版本控制主機**，例如 GitHub——用來讀取代碼並開立拉取請求；
- 一個 **議題追蹤系統**——用來拉取下一個工作單元並更新其狀態；
- 一個 **聊天平台**，例如 Slack——用來公布進度，或在需要人手
  關注時升級上報。

```yaml
# 例子：把循環接駁到現有工具的連接器。
connectors:
  source_control: github      # 讀取代碼、開立拉取請求
  issue_tracker: github-issues # 拉取任務、更新狀態
  chat: slack                  # 張貼進度、升級上報給人
```

連接器正是令循環成為你現有系統中一名參與者，而非一場封閉
實驗的關鍵。

## 停止條件（Stopping Condition）

**它是甚麼。** **停止條件**是那條告訴循環何時停止迭代——
或者何時改為升級上報給人而非繼續下去——的規則。

**它的用途。** 循環會不斷重複，而沒有既定出口的重複是危險的：
它可以永遠空轉、燒光預算，而且甚麼有用的東西都產不出來。停止條件
給予循環一個清晰的「完成」（或「放棄並求助」）定義，這樣它就會刻意地
終止，而非無休止地運行。它正是令循環不致變成一個失控進程的組成部分。

**具體例子。** 一個停止條件通常是一條或多條明確的規則：

- **測試通過**——一旦評估器的測試套件變綠，目標便達成，
  循環隨即停止；
- **評分準則分數達到某門檻**——例如，當評分準則分數
  達到至少 10 分中的 9 分時便停止；
- **達到最大迭代次數或預算上限**——例如，在
  20 次迭代之後、或者一旦 token／成本預算耗盡便停止，並升級上報給人
  而非繼續下去。

```bash
# 例子：每一次迭代都會檢查的複合停止條件。
if tests_pass; then            stop "目標達成"
elif [ "$iteration" -ge 20 ]; then  stop "達到最大迭代次數 — 升級上報給人"
elif over_budget; then         stop "預算耗盡 — 升級上報給人"
fi
```

第一條規則定義*成功*；其餘的定義*安全*。兩者合起來
保證了那個圓圈會停止轉動。

## 八者如何拼合在一起

按次序去讀，這八個組成部分勾勒出單一工作單元流經
循環的路徑。**自動化／觸發器**啟動一次迭代。**工作樹／隔離**
給予那次迭代一個安全的運行之處。**生成代理**在那裏做工作，
由**技能／知識**指引，並透過**連接器**伸手接觸外部世界。
**評估器**評斷結果；**狀態／記憶**記下發生了甚麼事，
好讓下一次迭代可以在其上繼續；而**停止條件**則
決定循環是要再轉一圈，還是就此完成。

並非每一個循環都需要每一個組成部分——第 08 章會展示一個只有
其中寥寥幾個的極簡循環。但這八個合起來就構成了部件的完整詞彙。
知道每一個是*為了甚麼*而存在，正是讓你在設計自己的循環時，能夠決定要納入哪些、
要把哪些組合起來、又要把哪些略去的關鍵。下一章
會令這些部件動起來，追蹤它們如何在循環的生命週期之中互相作用。

## 重點摘要

- 一個循環由**八個組成部分**裝配而成，每一個都接手了一項
  人在人手提示時所執行的判斷或工作。
- **自動化／觸發器**決定工作在何時、以何種方式進入循環——例如一個 cron
  時間表、一個針對新議題的 webhook，或者一個 `/loop` 指令。
- **工作樹／隔離**令每一次代理運行保持分隔而且可還原，這樣各次運行就不會
  互相干擾——例如 git worktrees 與獨立分支。
- **生成代理**做產出代碼與編輯的實際工作——例如
  Claude Code 或一個 Grok 代理。循環*並不是*那個代理；代理只是循環的其中一個
  部件。
- **評估器**（批評者／驗證者）從外部檢查輸出，這樣「完成」就是被
  驗證出來，而非被假定——例如一個專門跑測試的獨立代理、一個代碼審閱代理，
  或者一個評分準則計分器。
- **狀態／記憶**把進度持久化在任何代理的脈絡之外，這樣各次迭代就能
  累積——例如一個 `TODO.md`、一個草稿簿、一個 Linear／Notion 看板，或者磁碟上的
  檔案。
- **技能／知識**把專案的規則與模式編碼下來，這樣每一次迭代都
  跟從它們——例如一個 `SKILL.md`、一個 `AGENTS.md`，或者書面編碼標準。
- **連接器**把循環與外部工具整合起來——例如 GitHub、一個議題
  追蹤系統，或者 Slack。
- **停止條件**告訴循環何時停止或升級上報——例如測試
  通過、評分準則分數達到某門檻，或者觸及最大迭代／預算上限。
- 八者合起來追蹤一個工作單元由觸發直至終止的歷程；並非每一個循環
  都需要全部八個，但知道每一個的用途，正是讓你能夠刻意地
  設計一個循環的關鍵。
