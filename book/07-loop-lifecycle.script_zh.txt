在这一章里，我们会看看上一章那八个组件是怎样活起来、变成一个有次序、会不断重复的生命周期的。这是一个工作单位在一个 Loop（循环，也就是一个自动运行的工作循环）运行时所要经过的七个阶段。首先，工作会被发现或者被触发。接着，代理会在一个隔离的工作空间里被生成出来。一个代理会动手做事，产生输出。那些输出会被评估。如果做得不够好，反馈就会送回去，再试一次。状态会被保存。然后这个 Loop 就会决定，到底是继续下去、生成一个 sub-agent（子代理），还是停下来。看完这一章，你就会明白最后那个三选一的决定背后的准则，也会懂得追踪一份工作，从它被触发的那一刻，一直到它结束的那一刻，并且解释每一个阶段发生了什么，以及为什么会这样。

那我们现在就从零件讲到运转。上一章我们打开了这台机器，逐一讲了它八个部分的名字。现在我们要让它们转动起来。一张组件清单告诉你一个 Loop 是由什么组成的。而这个生命周期则告诉你，这些部分随着时间是怎样互相配合的。它告诉你，控制权是按什么次序从一个部分传到下一个部分，也告诉你，每个部分在绕圈的每一程里贡献了什么。

这个生命周期，其实就是一个很简单的问题的答案。一个 Loop 运行的时候，一步一步，到底实际发生了什么？当你亲手去提示一个 AI 的时候，那个次序是你自己提供的。你决定有事情要做。你叫这个代理去做。你看结果。你判断它做得好不好。你记住自己做到哪里。然后你决定要不要再叫一次。一个 Loop 就是把这个完全一样的次序，编码成一个会重复的循环，这样它就可以在你不必坐在旁边的情况下运行。我们接下来要逐个走过的阶段，正是那些判断，只不过把它们讲得清清楚楚，并且排好了次序。

一个 Loop 会把工作推过七个有次序的阶段。第一到第六个阶段，组成一次迭代的主体，而第七个就是结束这次迭代、并且选定下一步的那个决定。每一个阶段，都由上一章那一个或者几个组件去负责。

第一阶段，是发现或者触发工作。当工作进入这个 Loop，循环就开始了。可能是这个 Loop 发现有事情要做，比如清单上下一项未完成的事；也可能是一件外部事件去触发它。这是自动化与触发，也就是 Automation and Trigger 组件的职责。它可能是一个排好时间、半夜启动的定时任务；一个挂在新提交的 issue 上面的 webhook；又或者是一条 slash-loop 式的指令。在这个阶段启动迭代之前，什么都不会发生；而且这里不需要有人手动敲一句“开始”。

第二阶段，是在一个隔离的工作空间里生成代理。手头有了工作，这个 Loop 就会准备一个干净、密封的地方让这件工作去发生，并且在里面启动一个或者几个代理。这个阶段用到工作树与隔离，也就是 Worktrees and Isolation，也就是一份独立的代码副本和一条专用的分支，这样无论代理做什么，都会被限制住、也都可以还原。被生成出来的那个代理，就是生成代理，也就是 Generator Agent；当它开始的时候，它会收到项目的技能与知识，也就是 Skills and Knowledge，也就是那些编码规范和代理指引文件，这样它从第一个动作开始，就已经按照你的规则去做了。

第三阶段，是代理动手产生输出。在它那个隔离的工作空间里，Generator Agent 亲手做事。它读任务，它改代码库，它产生一组改动。当它需要接触外面的世界，比如从一个源代码控制主机读代码，又或者从一个问题追踪系统取细节，它就会通过连接器，也就是 Connectors 去做。这是唯一一个真正有新输出被创造出来的阶段。其他每一个阶段的存在，都是为了启动它、检查它、记住它，或者决定怎样处理它。

第四阶段，是评估器检查输出。刚刚产生出来的输出，现在会由外部去判断。评估器，也就是 Evaluator，它的角色是一个评审或者验证者；它会跑整个测试套件、审查那些改动，或者按一份评分标准去打分，然后给出一个裁决。到底这东西够好，还是不够好？这个阶段保护这个 Loop，不让它过于自信地接受了错误的或者低质量的工作。而最关键的一点是，评审和做事的是分开的。Generator Agent 不能批改自己的作业。

第五阶段，只会在输出不够好的时候才发生。在那种情况下，这个 Loop 不会把这件工作扔掉。它会从失败里学东西。到底错在哪里的实际信号，无论是失败的测试输出、审查者的评语，还是评分标准上那些低分备注，都会作为额外的上下文送回给 Generator Agent，然后循环就回到第三阶段，再试一次。这条反馈路径，正是让一个 Loop 成为一个 Loop 的核心。每一次被否决的尝试，都让下一次有更好的信息基础，而不是盲目地重复。而这个阶段是有条件的。当输出已经足够好，这个 Loop 就会直接跳过它。

第六阶段，是保存状态。无论这次迭代是完全成功，还是只取得了部分进展，这个 Loop 都会在任何单个代理的上下文之外，记下发生了什么，用的是状态与记忆，也就是 State and Memory 组件。它可能会更新一个待办文件、在一个草稿簿写下笔记，或者在一个项目看板上面移动一张卡片。这件事让下一次迭代可以在这一次的基础上继续，而不必从零开始。也让你随时都可以读到这件工作目前的状态。

第七阶段，是那个决定。当迭代完成、状态也保存起来了，这个 Loop 就到达它的决定点。主要由停止条件，也就是 Stopping Condition 去引导，它会在三条路里选一条。它可以继续去做多一次迭代；它可以生成一个 sub-agent，也就是子代理，去处理一件独立的工作；又或者它可以停下来。这个三选一的选择，重要到值得我们单独拿出来看。

如果这样有帮助，你可以把整个次序想象成一条流程：工作在顶部进来，每当输出不够好就在里面那条重试路径绕回去，最后到达底部那个决定点。不过，我们刚刚走过的那些步骤，本身就是这条流程，所以我们集中讲讲那个决定。

每一次绕这个 Loop，都在同一个分岔口结束；而它选哪条路，是由清晰的准则决定的，而不是靠猜。Stopping Condition 在这里是主要的权威，但这个决定确确实实是三选一。

第一个选项，是停止。当两种规则其中一种被触发，这个 Loop 就会结束。第一种是成功。目标已经达到。Evaluator 的测试通过了，又或者评分标准的分数已经到了它的门槛。这件工作做完了，所以这个 Loop 就有意识地结束，并且交付它的成果，比如通过一个 Connector 开一个 pull request。第二种是安全。一个防护栏被碰到了。迭代次数到了上限，又或者代币和成本的预算用完了。在这里，这个 Loop 在没有成功的情况下停下来，并且升级交给一个人去处理，而不是在那里空转下去。这条规则，就是防止一个 Loop 变成一个失控过程的东西。

第二个选项，是继续。如果成功规则和安全规则都还没被触发，那就意味着还有有用的工作要做，而且还有预算去做，所以这个 Loop 就再绕一圈。在实际操作上，继续就是在第五阶段的反馈之后所选的那条路。目前的任务还没完成，尝试次数和预算都在界限之内，所以这个 Loop 就用它刚刚学到的东西，在同一个任务上开始一次全新的迭代。

第三个选项，是生成一个 sub-agent。有时候最合适的做法，既不是重试同一个任务，也不是放弃，而是去委派。当这件工作自然地分裂成一件独立、自成一体的事，一件够大、值得有自己的隔离工作空间和自己评估的聚焦子任务，这个 Loop 就会生成一个 sub-agent 去处理它。这个 sub-agent 会运行它自己一个迷你的生命周期，有自己的生成、动手、评估和保存步骤，并且把它的成果回报给母 Loop。倾向于去生成的准则有两个：一个是可分解性，也就是这件任务可以干净地分开；另一个是独立性，也就是那件事可以在不必不断协调的情况下完成。当一个代理逐条线性地慢慢磨，会比把工作分派出去更慢、更乱的时候，就选这条路。

所以，简单讲。做完了或者预算用完了，就停。当这件工作应该被分割和委派，就生成一个 sub-agent。当手头上的任务还有事情要做、也还有资源去继续，就继续下去。

值得留意的是，这里其实有两个 Loop 在运作。有一个内层 Loop，也就是第三到第五阶段，在那里一个单一任务会带着反馈不断重试，直到 Evaluator 接受它，或者一个安全上限被触发为止。也有一个外层 Loop，也就是第一到第七阶段，它从一个工作单位移到下一个。第七阶段的决定，就是 Stopping Condition 居住的地方。它是唯一一个选择继续、生成，还是停止的位置；也是让整个结构会结束、而不会永远运行下去的原因。

而现在，这个生命周期并不是另外一个机制，硬装在这些组件上面。它本身就是这些组件，只是在运转之中被观察到。每一个阶段，纯粹就是某一个组件做它本职工作的那一刻。第一阶段，发现或者触发工作，就是 Automation and Trigger 组件。第二阶段，在隔离之中生成代理，就是把 Worktrees and Isolation、Generator Agent、以及 Skills and Knowledge 集合到一起。第三阶段，代理动手产生输出，就是 Generator Agent 和 Connectors 并肩工作。第四阶段，检查输出，就是 Evaluator。第五阶段，把反馈送回并且重复，就是 Evaluator 把它的裁决交回给 Generator Agent。第六阶段，保存状态，就是 State and Memory 组件。而第七阶段，决定到底是继续、生成还是停止，就是 Stopping Condition，并且有 Connectors 在旁边随时准备好做交付。

从顶到底地追踪一次，你就跟过了一份工作，从准许它进来的那个触发，一直到放它走的那个决定。这正是上一章预告的那段旅程，那时它说，八个部分追踪着一份工作单位穿过这个 Loop 的路径。这些组件是名词，而这个生命周期就是动词。两样都在手，你就准备好迎接下一章了，在那里你会建造一个真正运行这个循环的 Loop，从一个最基本的版本开始，逐步演化成一个生产级别的系统。

那以下就是几个重点。一个 Loop 是以一个有次序、会重复、共七个阶段的生命周期去运行的。第一，发现或者触发工作。第二，在一个隔离的工作空间生成代理。第三，代理动手产生输出。第四，Evaluator 检查输出。第五，如果不够好，就把反馈送回并且重复。第六，保存状态。第七，决定是继续、生成一个 sub-agent，还是停止。

第三到第五阶段，组成一个内层的反馈循环。一次被否决的尝试不会被丢弃。它会作为额外的上下文回到 Generator Agent，这样每一次重试都比上一次有更好的信息基础。

第七阶段的决定点是三选一。这个 Loop 会在成功的时候停止，也就是当测试通过，又或者评分标准的门槛达到；又或者它会在一个安全上限的时候停止，也就是当迭代次数到顶，又或者预算用完，并且升级交给一个人。当任务还没完成而且还有资源，它就继续。而当这件工作够可分解、也够独立，可以作为它自己一个迷你生命周期去委派，它就生成一个 sub-agent。

Stopping Condition 在决定点是唯一的权威。它就是保证这个 Loop 会有意识地结束，而不是永远运行下去的东西。

最后，每一个生命周期阶段，纯粹就是八个组件其中一个正在做它本职工作。这些组件是那些部分，而这个生命周期，就是那些部分在运转之中。
