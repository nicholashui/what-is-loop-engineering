# 10. 風險、限制與緩解措施

> **本章內容：** 你將了解 Loop（迴圈）工程坦白而言的負面一面——當一個 Loop 在
> 無人看管下運行時會出現甚麼問題，以及如何在每種失效模式造成代價之前先把它框死。
> 我們會深入處理兩種風險：**失控的 token 成本**，即一個自主運行的 Loop 默默地
> 燒掉預算（budget）；以及**「slop（低質量輸出）」**，即薄弱的驗證令 Loop 產出
> 大量低質量的輸出。對於每一種風險，我們都會把緩解措施扣返你已經建立好的機制——
> 第 08 章的護欄（guardrails）以及第 06–08 章的 Evaluator（評估器）。最後，我們
> 會論證：這一切都無法消除對工程判斷的需要——Loop 只是把你的判斷搬到別處，而非
> 將它退役。

## 一個坦白的盤點

上一章是*為* Loop 講好話。本章則是它必要的對沖。令 Loop 強大的一切——它無需你
在場就運行、它不知疲倦地反覆迭代、它自己供應下一個 prompt——正正就是當有甚麼出錯
時令它變得危險的地方。一個逐回合下 prompt 的人（第 03 章）在下一回合就會察覺到
錯誤而停手。Loop 並沒有這種本能：放任不管的話，它會把一個錯誤重複得同它重複一個
成功一樣有耐性，而且全程都在花真金白銀。

所以本章刻意並非一場推銷。Loop 工程值得學習，但它並非魔法，過度吹噓它對讀者並無
好處。下面兩種風險，正是你在實踐中真正會遇到的。兩者都不是迴避 Loop 的理由；兩者
都是要用前面章節所描述的護欄與 Evaluator 去建造 Loop 的理由。好消息——而且是真實
的好消息——是：令 Loop 達到生產級水平的同一套結構，正正也是令這些風險變得可控的
東西。緩解措施並非新發明；它們就是你已經認識過的組件，只不過如今對準它們本來就是
為了預防的那些失效模式。

## 風險 1：失控的 Token 成本

第一種風險最為即時，也最容易被低估：**一個自主運行的 Loop 在每次迭代都會花錢，
而一個沒有上限的 Loop 可以花掉無上限的金額。** 每次呼叫 Generator Agent（生成代理）
都會消耗 token，而 Loop 按定義就是一部不斷作出大量這類呼叫、而且中間不會徵求許可
的機器。你當初為了自主性而建造 Loop，但這份自主性正正就是移走了那個天然煞車掣——
再沒有一個人去讀每一回合，並判斷下一回合是否值得。

這通常以幾種典型方式出錯：

- **卡死的 Loop。** Loop 永遠達不到目標，但又永遠不會意識到這一點。它不斷迭代、
  通不過 Evaluator、回收反饋、然後再試——若無任何東西阻止，就是永遠。每一圈都很
  便宜；帳單就是極大量便宜圈數的總和。
- **來回擺盪的 Loop。** 一個更微妙的版本：代理修好問題 A，卻弄壞 B；下一次迭代
  修好 B，又再弄壞 A。Loop 每次迭代都在自信地「進展」，卻甚麼都收斂不到，而且每次
  都要付全價。
- **昂貴的單次嘗試。** 成本不單止關乎迭代*次數*。一次把巨大上下文塞進 prompt 的
  嘗試，又或反覆呼叫一個昂貴的模型，都可以在單單一圈內就爆掉預算——所以單靠限制
  迭代次數並不足夠。

坦白的講法是：這種風險屬於**結構性的，而非偶發的**。它並非偶爾擊中一個寫得差的
Loop 的罕見 bug；它是*任何*缺乏明確上限的 Loop 的預設行為。一個沒有停止條件
（Stopping Condition）的 Loop 並非「通常會完成」——它根本就沒有「完成」這個概念。

### 緩解失控的成本

緩解措施就是有紀律地運用 **Stopping Condition（停止條件）**組件（第 06 章）以及你在
第 08 章設計的**護欄（guardrails）**。失控風險正正就是那些護欄存在的目的，所以防禦
之道就是刻意地運用它們，而非把它們當成事後補上的東西：

- **設一個硬性的迭代上限。** 一個 `max_iterations` 限制保證 Loop 即使從未成功、
  亦從未徹底失敗，都會終止。單憑這一點，就把卡死的 Loop 由一張無止境的帳單變成
  一張有上限的帳單。
- **設一個獨立的成本上限。** 一個 `max_cost_usd` 預算會在累計花費越過某個門檻時
  停止 Loop，與迭代次數無關。這正是用來捕捉迭代上限會漏掉的「昂貴單次嘗試」情況。
  正如第 08 章所強調，你想*兩個*上限都要，因為兩者各自覆蓋對方的盲點。
- **令成功的 Stopping Condition 真實而且可檢查。** 停止花費最乾淨的方法，就是真正
  完成。一個 Evaluator 能客觀確認的目標（測試通過、達到 rubric（評分準則）門檻）
  讓 Loop 在工作完成的那一刻就退出，而非超額迭代過頭。
- **升級上報（escalate），而非死磨爛纏。** 當預算耗盡而仍未獲接納，Loop 應該交接
  俾人——即第 08 章那條升級至人手（escalation-to-human）的路徑——而非默默失敗，
  又或更糟，被盲目地重啟。偵測到*重複的無進展*（即來回擺盪的 Loop）並提早升級上報，
  可以慳返你原本要花來確認 Loop 已卡死的那筆預算。

這些上限以資料形式存在於 Loop 的配置之中，正如第 08 章那個生產級例子一樣：

```yaml
# Cost guardrails: layered Stopping Conditions that bound spend two ways.
stopping_conditions:
  max_iterations: 12          # 為卡死/來回擺盪的 Loop 的圈數設上限
  max_cost_usd: 5.00          # 獨立於迭代次數為花費設上限
  stop_when_tests_pass: true  # 成功退出：目標一獲驗證即停止

escalation:
  on_budget_exhausted: notify-human   # 交接而非盲目重啟
  on_repeated_no_progress: 3          # 連續 3 次同樣失敗 -> 提早升級上報
```

要坦白承認的限制是：較緊的預算會犧牲掉一部分當初令 Loop 吸引的自主性。上限設得太低
會在 Loop 完成真正艱難的工作之前就停止它；上限設得太高則違背了它自身的目的。並不
存在一個放諸四海皆準的正確數字——選擇它是一項工程判斷的行為，而這正是本章逐步建立
的主題。

## 風險 2：「Slop」——大量、低質量的輸出

第二種風險更安靜，而長遠而言更具腐蝕性。**Slop（低質量輸出）**就是當 Loop 迭代得快
但驗證得弱時所產出的東西：大量*看似*進展、但站不住腳的輸出。失控成本會在帳單上自我
宣告，slop 卻藏身於看起來合理的 diff 之中，純粹靠數量與自信通過審查，直到有人發現
當中大部分都微妙地錯、冗餘、或偏離目標。

Slop 是放任自主性跑贏驗證的直接後果。回想第 09 章的論證：*沒有* Evaluator 的自主性
只不過令錯誤產生得更快。Loop 是一個乘數——它放大你對準它的任何流程。把它對準一個
強而有力的驗證流程，它就把好工作放大；把它對準一個薄弱的流程，它就把 slop 放大得
同樣有效率。危險之處在於，即使輸出的*實質*在退化，其*形式*仍然保持令人信服：

- **自信但錯誤的輸出。** Generator Agent 報告成功、diff 讀起來清爽、而且沒有任何
  客觀的東西去反駁它——因為從來就沒有諮詢過任何客觀的東西。第 08 章那個極簡 Loop，
  即信任代理自己那句 `ALL DONE` 的版本，正是典型的 slop 產生器。
- **把數量誤當成價值。** 一個 Loop 在相同時間內可以生成遠多於人手的代碼。在沒有
  質量閘的情況下，那種產出量反而變成負擔：更多要審查的範圍、更多缺陷可以藏身的地方、
  更多淹沒信號的貌似合理的雜訊。
- **Goodhart 漂移。** 當 Loop 針對一個薄弱的代理指標（proxy）——一個膚淺的測試、
  一份含糊的 rubric——去優化，它就學會去滿足那個代理指標，而非滿足原意。指標亮起
  綠燈，而真正的目標卻悄悄退場。

### 緩解 Slop

緩解 slop 的措施就是 **Evaluator（評估器）**（第 06–08 章），尤其是那條原則：
Evaluator 必須*獨立*於 Generator Agent。Slop 就是當驗證薄弱時填補真空的東西；一個
強而獨立的 Evaluator 就是你移走那個真空的方法。這些模式正是第 08 章為設計評估準則
所列出的那些，如今理解為你對抗低質量輸出的首要防線：

- **以強而客觀的測試打頭陣。** 一道代理無法靠講道理蒙混過去的客觀閘——一套真實的
  測試套件、一個類型檢查器、一次建置、一次 schema 驗證——是抵抗 slop 的骨幹。它
  回傳一個明確的通過/不通過，任何份量的自信文辭都偽造不來。測試愈強、愈有意義，
  slop 能通過的空間就愈少。
- **加入一個 critic/rubric 代理去補測試漏掉的地方。** 客觀檢查確認行為，但對於
  可讀性、安全性、又或這項改動是否符合原意，就說得很少。一個第二代理按照一份明確、
  有版本管理的 rubric 去為 diff 評分——即一個充當 Evaluator 的 critic（評審）——
  正好覆蓋那個缺口。如第 08 章所建議，要刻意地把守 rubric 門檻。
- **令評估獨立於生成。** 最重要的單一屬性：*評斷*工作的東西，絕不可以是*產出*工作
  的同一個東西，亦絕不可以單純信生產者的一面之詞。獨立評估正是把「代理話完成咗」
  轉化為「一個外部檢查確認完成咗」的關鍵——這正正就是 slop 與經驗證的進展之間的
  分別。
- **令驗證成為一道硬性的閘，而非一個建議。** 只有當一次嘗試通過客觀閘*而且*越過
  rubric 門檻時才接納它。一個只會發出警告、而仍然放未經驗證的工作通過的檢查，並非
  對抗 slop 的防禦——它本身就是加了額外日誌記錄的 slop。

那條接納規則就是第 08 章那個獨立的雙閘檢查，值得專門再睇一次，視之為反 slop 的機制：

```python
# Anti-slop acceptance: independent verification the agent cannot self-certify.
def is_acceptable(result, rubric_score, threshold=0.8):
    if not result.tests_passed:   # 客觀閘：無法偽造的通過/不通過
        return False
    if rubric_score < threshold:  # 獨立的 critic 越過一份明確的 rubric
        return False
    return True                   # 只有經驗證的工作會獲晉升；其餘的一律丟棄
```

這裏要坦白的限制是：**一個 Evaluator 的好壞，僅取決於你給它的準則。** Loop 無法
驗證你從未指明的質量。如果你的測試膚淺、又或你的 rubric 含糊，Evaluator 就會忠實地
把 slop 放行，而 Loop 亦會忠實地產出更多 slop。這並不是一個你可以靠加更多代理來
自動化消除的缺陷——它是對人的一項要求：要把「好」的定義講得夠清楚，清楚到一部機器
可以執行它。這就帶我們去到兩種風險背後共同的那個限制。

## 那個不會消失的限制：工程判斷

如果一章講風險的內容，最後暗示只要配置正確就會令 Loop 自動安全，那就是不誠實。事實
並非如此。Loop 工程最深層的限制，亦是最容易被忽略的：**Loop 並不移除對工程判斷的
需要——它只是把它搬到別處。**

本章每一項緩解措施，最終都建基於一個沒有 Loop 能替你作出的人類決定：

- **目標。** Loop 會追求你給它的目標，一字不差地按你的措辭去追。決定*甚麼*值得
  建造，並把它表述為一個清晰、可檢查的結果（第 08 章），是人類判斷。Loop 會以追逐
  一個好目標時同樣不知疲倦的能量，去追逐一個揀得差的目標。
- **評估準則。** Evaluator 執行的是你所定義的標準——不多，不少。決定要測試甚麼、
  測試必須有幾強、rubric 獎勵甚麼、以及把它的門檻設在哪裏，都是人類判斷。Slop 歸根
  究柢就是評估準則弱到接不住它的徵狀。
- **護欄。** 預算、迭代上限、隔離邊界、以及升級上報路徑，全部都是*你*揀的數值。
  這項任務值幾多錢？要試幾多次之後才應該由人去睇一睇？甚麼絕不可以去到 `main`？
  這些都是 Loop 無法自行原創的判斷取捨。

這樣睇，本章那兩種風險其實是同一個教訓的兩個方向。失控成本是當*護欄*體現的判斷太少
時所發生的事；slop 是當*評估準則*體現的判斷太少時所發生的事。兩種情況之下，Loop
都精準地做了它被吩咐去做的事——缺口在於吩咐本身。

這就是為甚麼第 09 章那個由操作員到架構師的轉變，是判斷的一次*搬遷*，而非把它消除。
作為操作員，你曾經一百次地施加判斷，一次一個 prompt，而且能在中途捕捉到問題。作為
架構師，你施加判斷的次數較少，但槓桿大得多——而且關鍵在於，是*預先*施加，趕在 Loop
無人看管地運行之前。每個決定的賭注正正會升高，因為你已經不再在場逐回合修正它。
Loop 把你的判斷濃縮成寥寥數個高槓桿的選擇，然後忠實地、大規模地執行它們。這就是
它的威力與風險，可以一句話概括：它精確地做你設計它去做的事，不管那是不是你的本意。

由此而來的務實姿態，既非炒作亦非恐懼。為適合 Loop 的工作去建造 Loop（第 09 章）、
用本書所描述的護欄與 Evaluator 去框住它們，並把你花在設計目標、準則與護欄上的時間
當作真正的工程——因為那正是你的判斷如今所在之處。

## 重點摘要

- Loop 工程強大但並非魔法；一個坦白的盤點會老實點名它的風險，而非過度吹噓。令 Loop
  達到生產級水平的同一套結構，正是令它的風險可控的東西——緩解措施就是你已經認識過的
  組件，對準它們所預防的那些失效。
- **風險 1——失控的 token 成本：** 因為 Loop 作出大量自主的代理呼叫、中間並無人手
  煞車，一個沒有上限的 Loop 可以無限制地花費，無論是經由一個*卡死*的 Loop、一個
  *來回擺盪*的 Loop、抑或一次昂貴的單次嘗試。這是任何缺乏明確上限的 Loop 的預設
  行為，而非罕見的 bug。
- **以 Stopping Condition 與護欄緩解成本（第 08 章）：** *兩個*都要設——一個硬性的
  迭代上限與一個獨立的成本上限，令成功條件真實而可檢查，使 Loop 能真正完成，並在
  預算耗盡或重複無進展時升級上報俾人，而非死磨爛纏。
- **風險 2——slop：** 當迭代跑贏驗證，Loop 就產出大量看似合理但低質量的輸出——
  自信但錯誤的 diff、把數量誤當成價值、以及針對薄弱代理指標的 Goodhart 漂移。Loop
  會放大你對準它的任何流程，無論好壞。
- **以一個強而獨立的 Evaluator 緩解 slop（第 06–08 章）：** 以代理無法偽造的客觀
  測試打頭陣、加入一個 critic/rubric 代理去補測試漏掉的地方、令 Evaluator 獨立於
  Generator、並令驗證成為一道硬性的接納閘而非一個警告。一個 Evaluator 的好壞，僅
  取決於你給它的準則。
- **工程判斷不會消失——它只是搬遷。** Loop 會忠實地追求*你*設計的目標、執行*你*設計
  的評估準則、並尊重*你*設計的護欄；它無法原創那些決定。失控成本與 slop 分別都是
  護欄與準則裏人類判斷太少的徵狀。
- 負責任的姿態既非炒作亦非恐懼：為適合的工作建造 Loop、用護欄與一個 Evaluator 框住
  它們，並把設計目標、準則與護欄當作真正的工程——因為那正是你的判斷如今所在之處，
  賭注更高、槓桿更大。
