# 詞彙表

> **本章內容：** 本詞彙表是全書關鍵術語的唯一權威依據。每個術語只在此處定義一
> 次；每一章在使用這些術語時，都採用以下所列的含義。條目按英文字母順序排列，縮
> 寫會在首次提及時寫出全稱。

**Agent Harness Engineering（代理框架工程）**
: 通往 Loop Engineering 的演進譜系中的第三個階段：為單一代理運行建立一個可靠執
行環境（即 Harness）的實踐。它在精心整理好的脈絡之上，再於單一工作階段內加入動
作與回饋（工具、觀察到的結果、多步驟的控制）。

**AI Coding Agent（AI 編程代理）**
: 一種以大型語言模型為基礎的系統，能夠就軟件任務進行推理並對其採取行動——執行
指令、讀取及編輯檔案、查核結果——而不只是輸出文字。在 Loop 之內，AI 編程代理通
常擔當 Generator Agent 的角色。

**Author-Asserted Claim（作者主張陳述）**
: 一項事實陳述——某個有日期的事件、某項具名的歸屬，或一段直接引文——取自所提供
的原始素材，本書透明地將其呈現為需要由讀者自行獨立查證，因為在撰寫環境中無法對
照即時來源加以核實。此類陳述會帶有行內標記，並會收錄於 References 檔案之中。

**Automation/Trigger（自動化／觸發）**
: 決定工作何時及如何進入 Loop 的 Loop 組件。例子包括 cron 排程、在有新議題時觸
發的 webhook，以及按需呼叫的斜線式指令。

**Book（本書）**
: 完整的教學成品，由排好順序的 Markdown 章節檔案，加上輔助檔案（目錄、本詞彙表
及參考資料）組成。

**Chapter（章節）**
: 涵蓋本書某一主要主題範疇的單一頂層 Markdown 檔案。

**Connectors（連接器）**
: 將 Loop 與外部系統整合起來的 Loop 組件。例子包括源碼控制主機、議題追蹤系統，
以及聊天平台。

**Context Engineering（脈絡工程）**
: 通往 Loop Engineering 的演進譜系中的第二個階段：刻意組織放進模型脈絡視窗中的
資訊——相關檔案、檢索到的文件、先前的對話、範例及指示——使模型在恰當的素材上進
行推理的實踐。它在 Prompt Engineering 那些精挑細選的字眼之上，再加入正確的資訊。

**Evaluator (Critic / Verifier)（評估器／評審／驗證者）**
: 從代理外部獨立查核 Generator Agent 輸出的 Loop 組件。Evaluator 可以是一套測試
組、一個負責程式碼審查或「評審」的代理，又或是一個 Rubric 評分器。正是它讓 Loop
能夠判斷工作是否真的做得好，而非只信賴代理對自己的評估。

**Generator Agent（生成代理）**
: 在 Loop 之內產出實際輸出——程式碼、修改，或其他成品——的 Loop 組件。它通常是一
個 AI 編程代理，例如 Claude Code 或 Grok 代理。

**Harness（框架）**
: 供單一代理運行使用的可靠執行環境：它向代理提供工具、把工具結果回饋到其脈絡之
中、管理多步驟工作階段那種一來一回的互動，並使該次運行保持穩定。Loop 的編排層
正是建構在 Harness 之上。

**Iteration（迭代）**
: 走過 Loop 一遍的過程——發現或接收工作、運行代理、評估輸出、保存狀態，並決定接
下來該做甚麼。Loop 是透過多次迭代去推進一個目標，而非在單一次運行中完成。

**Loop（迴圈）**
: 由 loop engineer 所建立的編排系統。它發現或觸發工作、在隔離環境中衍生代理、查
核它們的輸出、跨多次運行保存狀態，並決定要繼續、衍生一個子代理，抑或停止。

**Loop Engineering（迴圈工程）**
: 設計一套控制系統，使其能自主地對 AI 編程代理進行提示、觀察、查核及迭代，而非
逐回合人手提示它們的實踐。它是這條演進譜系的最頂層階段——這條譜系由 Prompt
Engineering 起，經 Context Engineering 與 Agent Harness Engineering 一路發展上
來。

**Manual / Turn-by-turn Prompting（人手／逐回合提示）**
: 與 AI 編程代理協作的傳統做法：由人發出一個指示、閱讀結果，再發出下一個指示
——逐步監督每一步。Loop Engineering 以一套能自行對代理進行提示和迭代的系統，取
代這種親力親為的監督。

**Prompt Engineering（提示工程）**
: 通往 Loop Engineering 的演進譜系中的第一個階段：精心打磨單一指示用詞的實踐
——措辭、範例、輸出格式、角色設定——以引導模型就一個提示、一個回應得出好答案。

**Ralph Loop (Ralph Wiggum technique)（Ralph 迴圈／Ralph Wiggum 手法）**
: Loop Engineering 由之演化而來的前身手法：一個簡單的 shell 迴圈，反覆向代理餵
入新的脈絡，連同上一次運行的輸出與錯誤，然後重新運行，直至任務完成為止。它示範
了把代理放進迴圈內運行並將結果回饋進去這個核心構想，但沒有隔離、沒有獨立的
Evaluator，亦沒有保存狀態。

**ReAct (Reason + Act)（ReAct，推理＋行動）**
: 把推理步驟與行動交錯進行的代理模式：代理思考下一步該做甚麼、透過工具採取一個
具體行動、觀察結果、再次推理，如此持續直至任務完成。它是 Loop 的概念前身；Loop
把這個推理－行動循環從單一代理對話中提取出來，放進一個外部、可控的系統之中。

**Reader（讀者）**
: 目標學習者——一位在使用者層面熟悉 AI 編程代理、但對自主編排它們仍屬新手的軟件
工程師或技術從業者。

**Rubric（評分準則）**
: 一套明確、附有分數的評估標準，用以判斷 Generator Agent 的輸出。Evaluator 可運
用評分準則去給出分數，而評分準則的分數門檻亦可充當 Stopping Condition。

**Skills/Knowledge（技能／知識）**
: 向代理提供可重用指引與標準的 Loop 組件。例子包括一個技能檔案、一個代理指引檔
案，以及編程標準。

**Slop（劣質輸出）**
: 當代理在缺乏強力查核的情況下運行時所產生的大量低質輸出。它是 Loop 的一大風
險，可藉測試與評審代理把關 Loop 所接受的內容來加以緩減。

**State/Memory（狀態／記憶）**
: 跨迭代保存進度的 Loop 組件，使任何單一次運行結束時都不會遺失工作成果。例子包
括一個 TODO 檔案、一個記事本（scratchpad）、一個項目看板，以及磁碟上的檔案。

**Stopping Condition（停止條件）**
: 令 Loop 終止的規則。例子包括測試通過、Rubric 分數達到某個門檻，以及達到迭代次
數上限或耗盡預算。

**Sub-agent（子代理）**
: Loop 為處理某個聚焦的工作部分而衍生的次級代理——例如從主運行分支出去的一項專
門任務。衍生子代理是 Loop 決策點上三種結果之一（另外兩種是繼續與停止）。

**Token Budget（Token 預算）**
: 對 Loop 可消耗的 token 數量（以至成本）所設的上限。緊縮的預算配合 Stopping
Condition，可緩減 token 成本失控增長的風險。

**Worked Example（示範例子）**
: 一個具體、由頭到尾的示例——程式碼、配置，或附註解的逐步講解——用以示範某個概念
在實踐中如何運作。本書提供一個極簡的 Ralph 風格例子，以及一個生產級的例子。

**Worktrees/Isolation（工作樹／隔離）**
: 使每一次代理運行都保持隔離且可還原的 Loop 組件。例子包括 git worktree 與分開
的分支，它們讓代理的工作得以進行而不會干擾主開發線。
