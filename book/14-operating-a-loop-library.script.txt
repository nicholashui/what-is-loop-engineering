In this chapter, we assume you have a library that can describe and run loops, and we focus on keeping it good as it grows and as more people use it. We'll cover the contribution workflow that lets a team add loops safely, the quality bar and review standards that keep slop out, how to version and deprecate entries, the metrics that tell you whether a loop is worth keeping, and the governance and security concerns that come with autonomous loops running across a team. By the end, you'll be able to operate a loop library as shared infrastructure, not just a personal folder of scripts.

So let's start with a mindset shift: a library is a product, not a folder. A handful of loops in a directory is a convenience. But a library that a whole team trusts and runs against real systems is infrastructure, and infrastructure needs an owner, a quality bar, a way to change safely, and a way to measure whether it's earning its keep. The failure mode of a successful library is not too few loops. It's too many low-quality ones, so that nobody can tell which entry to trust. Curation is the work that prevents that.

This chapter assumes the mechanics from the previous chapter, the schema, the store, the runner, the CI validator, and adds the human layer around them: how loops get in, how they stay good, and how the whole thing scales past one author.

First, the contribution workflow. The same discipline you apply inside a loop, propose a change, verify it independently, accept only on a pass, applies to changes to the library itself. Treat a new or modified loop entry like any other code change: a pull request, a review, and an automated gate.

Here's a workflow that scales from one contributor to many. First, propose via pull request. A contributor adds or edits an entry file on a branch and opens a PR. Nothing merges directly to the main library. Second, automated gates run first. CI runs the validator: schema conformance, a present stopping condition, and an existing prompt file. A PR that fails these never reaches a human reviewer. The cheap checks go first. Third, a maintainer reviews against the quality bar. A human checks what automation cannot: is the goal genuinely checkable, is the evaluator independent of the generator, are the guardrails honest, is the attribution correct. Fourth, dry-run before acceptance. For any non-trivial loop, the reviewer runs it once through the runner against a safe target and confirms it behaves as described, especially that it stops. A loop that has never been executed is a hypothesis, not a library entry. And fifth, merge promotes it to stable, and the index is rebuilt.

The principle is recursive, and worth saying plainly: a loop library is itself maintained by a loop. Propose, verify independently, accept only on a pass. The devil's-advocate and builder-reviewer patterns apply as much to library entries as to the code those entries operate on.

Now the quality bar. Automated validation proves an entry is well-formed. The quality bar proves it is good. Every entry admitted to the library should clear a written, visible checklist, kept in the repo so contributors can self-check before opening a PR. The goal is externally checkable, phrased as an outcome a machine or independent reviewer can confirm, not "improve X." The evaluator is independent of the generator, so "done" never rests on the agent's own report. Stopping conditions are explicit and layered: a success exit plus at least one budget cap. An unbounded entry fails the bar outright. Isolation and a human-escalation path are defined. The blast radius is documented. And attribution and source are correct.

Make this checklist a PR template, so every contribution is reviewed against the same criteria. A loop that cannot clear the bar is not rejected forever. It's admitted as experimental or draft, clearly marked, so users know it has not earned full trust yet.

Next, versioning and deprecation. Loops change: a prompt improves, an evaluator tightens, a better approach supersedes an old one. Because each entry carries a semantic version and a maturity field, the library can evolve without breaking the people who depend on it. Version every meaningful change: bump the version when behavior changes, so callers that pin a version get reproducibility, and callers that track latest get improvements. Use the maturity ladder: new entries enter as draft or experimental, graduate to stable once proven, and end as deprecated when superseded. And deprecate, don't delete. When a loop is replaced, mark it deprecated, point to its successor, and keep it for one release cycle. Deleting an entry silently breaks anything that referenced it. The maturity ladder is what lets a library be welcoming to new loops without being reckless with trust.

Now, measuring whether a loop earns its place. A loop in a library is a claim that it's worth running. Once loops run regularly, you can replace that claim with evidence. Because the runner already drives every loop, instrument it once and you get telemetry for the whole library for free. Track the acceptance rate: of runs that reached a stopping condition, how many ended in success versus hitting a budget cap or escalating. Track iterations and cost to success: rising cost over time is an early warning of drift. Track the escalation rate: some escalation is healthy, but constant escalation means the loop is mis-scoped. And track usage: an entry nobody runs is documentation, not infrastructure, and a candidate for deprecation. These tie directly back to the stopping conditions: the same exits that bound a loop's behavior are the events you count to judge it.

Then there's governance and security at scale. The moment a library is shared and its loops run autonomously across many repositories, the risks stop being one engineer's problem and become an organizational one. A personal loop that misbehaves costs you a worktree removal. A shared loop with broad permissions that misbehaves can affect everyone. The controls that matter most are these. Least-privilege connectors: a loop's connectors should grant only the access it needs, read-only where possible, with no standing production credentials baked into entries. Central budget ceilings: beyond per-entry cost caps, enforce an organization-wide cap so the aggregate of many loops can't run away. An audit trail: log every run, which entry, which version, which inputs, who triggered it, what it changed, how it stopped. Human approval gates for high-blast-radius loops: anything touching production data, public copy, or irreversible state should require explicit human sign-off. And a clear owner: someone accountable for the quality bar, the deprecation decisions, and the security posture. None of these are new ideas. They're the earlier mitigations, lifted from a single loop to a fleet of them.

Finally, scaling the library itself. As it grows past a few dozen entries, two pressures appear: discovery gets harder, and duplication creeps in. Lean on shapes, not entries: when two entries are the same shape with different goals, prefer one parameterized entry over two near-duplicates. The library should grow in parameters and prompts, not in copies. Curate against sprawl: periodically run a housekeeping pass over the library, itself a loop, fittingly, to find entries nobody runs, duplicates, broken references, and stale prompts, and deprecate or consolidate them. And federate by domain, share the runner: large organizations can let teams own their own category folders while sharing one schema and one runner.

The endgame is a library that stays small relative to its usefulness: a curated set of trusted, parameterized, measured loops that the whole team reaches for, not an ever-growing pile of one-off scripts. That's the same subtractive discipline this book has argued for throughout, applied one level up. Add a loop to the library when not having it causes real duplication, keep it while it earns its place, and retire it when it stops.

So let's bring the key ideas together. A shared loop library is infrastructure, not a folder, and its failure mode is too many low-quality entries, so curation is the central job. Changes to the library follow the same loop discipline as the loops themselves: propose via PR, run automated gates, review against a written quality bar, and dry-run to confirm the loop actually stops before merging to stable. The quality bar requires a checkable goal, an independent evaluator, explicit layered stopping conditions, defined isolation and escalation, a documented blast radius, and correct attribution. Version and deprecate rather than break, using a maturity ladder exposed as a search facet. Measure each loop with the events its own stopping conditions already produce, and retire loops that thrash or that nobody runs. Governance scales the earlier mitigations to a fleet: least-privilege connectors, organization-wide budget ceilings, an audit trail, human approval gates, and a clearly accountable owner. And scale by shapes, not copies, keeping the library small relative to its usefulness.
