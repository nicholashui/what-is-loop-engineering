# 04. 源流與 ReAct 的關係

> **本章內容：** 你會了解 Loop Engineering（迴圈工程）由何而來——由 *Prompt
> Engineering*（提示工程）到 *Context Engineering*（脈絡工程）再到 *Agent Harness
> Engineering*（代理框架工程），最後到 *Loop Engineering* 的四階段源流，以及每個
> 階段相比前一個階段額外帶來咗啲乜。你亦會了解到一個 Loop（迴圈）同 **ReAct
> (Reason + Act，即推理 + 行動)** 模式有咩關係、一個 Loop 點樣將推理—行動嘅循環
> 由單一代理對話中抽出嚟、放入你親手設計嘅可控系統入面，以及前身技術 *Ralph
> Loop*（Ralph 迴圈）點樣為之後嘅一切鋪路。

## 一個不斷向上攀爬技術棧嘅領域

Loop Engineering 並唔係憑空出現。佢係一個穩步演進嘅最新一步——呢個領域年復一年，
不斷將 *由人類負責設計嘅範圍* 向上、向外推，由個別字詞推向整個系統。

呢條源流入面，每一個階段都解決咗前一個階段遺留落嚟嘅樽頸。當某一層變得夠可靠、
可以放心當佢理所當然嘅時候，注意力就會移上去更上一層。理解呢個攀爬過程，係最快
明白 *點解* Loop Engineering 會係而家咁樣嘅方法：佢係底下一切終於運作得夠好、可
以喺其上繼續建構之後，自然而然嘅結果。

呢條源流有四個階段：

```text
  Prompt          Context          Agent Harness          Loop
  Engineering ──▶ Engineering  ──▶ Engineering       ──▶  Engineering
  (字詞)          (輸入)           (一次可靠嘅運行)         (多次運行，
                                                            經編排調度)
```

等我哋逐個階段，沿住個技術棧向上行。

### 階段 1 — Prompt Engineering：雕琢字詞

任何人喺接觸語言模型時，最先學到嘅就係：*你點樣問* 會改變 *你得到啲乜*。**Prompt
Engineering（提示工程）** 就係雕琢單一條指令嘅用字，去引導模型俾出好答案嘅實踐
——揀措辭、俾例子、指定格式、賦予一個角色（「你係一位專家評審」）。

呢度嘅工作單位係 **一個提示加一個回應**。工程師嘅功夫係用字上嘅：搵到啲字眼令呢
一次對話回合成功。

- **佢帶來咗乜：** 認識到模型嘅輸出可以透過刻意嘅輸入嚟控制，以及一套用嚟做呢種
  控制嘅技巧詞彙（少量範例 few-shot、逐步指示、角色框定）。
- **佢未解決嘅問題：** 一條措辭再好嘅提示，始終只係喺訊息入面碰巧有嗰啲資訊上面
  運作。一旦任務需要模型從未被告知過嘅事實——你嘅程式碼庫、你嘅文件、尋日嘅決定
  ——再聰明嘅用字都唔夠。

### 階段 2 — Context Engineering：篩選整理輸入

下一個樽頸係 *模型答嘢嗰陣識啲乜*。**Context Engineering（脈絡工程）** 就係刻意
組裝放入模型脈絡窗口（context window）入面嘅資訊嘅實踐：相關嘅檔案、檢索到嘅文
件、之前嘅對話、例子同指示——令模型係基於正確嘅材料嚟推理，而唔淨係靠精心挑選嘅
字詞。

工作單位由單一提示擴展到 **模型睇到嘅整個輸入**。功夫由措辭轉移到 *揀選同組裝*：
要檢索啲乜、要包含啲乜、要捨棄啲乜，以及點樣將佢哋塞入一個有限嘅窗口入面。

- **佢相比 Prompt Engineering 額外帶來咗乜：** 佢將成個脈絡窗口當成設計嘅範圍，而
  唔係淨係指令嗰句句子。用字再好，但基於錯嘅資訊一樣會失敗；Context Engineering
  確保正確嘅資訊有出現。
- **佢未解決嘅問題：** 就算係措辭完美嘅提示，配上篩選整理得完美嘅脈絡，依然只係
  *一個回合*。真正嘅工作需要模型 *去行動*——執行一條命令、讀一個檔案、改程式碼、
  檢查結果——然後再根據發生咗嘅事再行動。靜態嘅輸入做唔到呢樣嘢。

### 階段 3 — Agent Harness Engineering：令一次運行變得可靠

要俾模型可以 *行動*，你要將佢包裹喺一個 **harness（框架）** 入面：即係單一代理運行
嘅可靠執行環境。Harness 俾工具予模型（一個 shell、一個檔案編輯器、執行測試嘅能
力）、將工具結果反饋返入佢嘅脈絡、管理一個多步驟工作階段嘅一來一往，並令成個運
行保持穩定同可重現。**Agent Harness Engineering（代理框架工程）** 就係將呢個環境
建構得好嘅實踐。

工作單位再次擴展——由單一輸入擴展到 **一次完整嘅代理運行**：一個工作階段，模型喺
入面推理、透過工具採取行動、觀察結果，並一直繼續落去，直到佢認為任務完成。

- **佢相比 Context Engineering 額外帶來咗乜：** *單一工作階段內嘅行動同反饋*。模
  型唔再局限於喺固定輸入上產出文字；佢可以改變世界（編輯檔案、執行命令），並對佢
  所觀察到嘅嘢作出回應。Harness 令呢個行動—觀察嘅循環喺一次運行入面變得可靠。
- **佢未解決嘅問題：** harness 令 *一次* 運行變得可靠，但單一運行始終受限於一段對
  話。佢會結束。佢嘅脈絡會塞滿。代理以外冇人去決定個結果係咪真係好、要唔要由乾淨
  狀態重新再試、定係要繼續再行多一百次迭代。判斷同堅持依然落喺睇住個工作階段嘅人
  類身上。

### 階段 4 — Loop Engineering：編排多次運行

呢度就係本書開始嘅地方。**Loop Engineering（迴圈工程）** 係設計一個控制系統嘅實
踐，呢個系統會自主噉提示、觀察、驗證並迭代 AI 編碼代理。佢坐喺 harness *之上*：
Agent Harness Engineering 令單一代理運行變得可靠，而 Loop Engineering 就建構嗰個
編排調度層，將代理一次又一次噉運行、由外部判斷佢哋嘅輸出、喺各次運行之間記住進
度，並決定要繼續、分叉出一個子代理（sub-agent），定係停止。

工作單位最後一次擴展——由單一運行擴展到 **朝住一個目標推進嘅多次運行**。功夫唔再
係用字上、甚至唔再係資訊上嘅；佢係 *系統性* 嘅。你係喺設計嗰部操作代理嘅機器。

- **佢相比 Agent Harness Engineering 額外帶來咗乜：** *跨運行嘅外部控制同迭代*。
  harness 信任代理自行決定幾時做完；一個 Loop 就加入一個獨立嘅 **Evaluator（評估
  器）** 去檢查工作、加入能夠比任何單一運行活得更耐嘅 **state（狀態）**，以及一個
  明確嘅 **停止條件**。佢將「一次好嘅運行」變成「持續產出好嘅運行，直到目標達成」。

留意嗰條貫穿始終嘅主線。喺每一個階段，人類嘅設計目標都向上移：由 *字詞*，到 *輸
入*，到 *單一運行*，再到 *運行各次運行嘅系統*。Loop Engineering 不過係呢個攀爬過
程到目前為止嘅頂點——去到呢一點，你所工程嘅嘢唔再係俾代理嘅一條訊息，而係一個替
你管理代理嘅系統。

## ReAct 的關係

要明白一個 Loop 所外部化嘅 *機制*，幫到手嘅做法係為每個現代編碼代理嘅核心模式改
個名：**ReAct**，全寫係 **Reason + Act（推理 + 行動）**。

ReAct 模式將兩種步驟交錯進行。代理 *推理*——用自然語言諗清楚下一步要做乜——然後 *
行動*——透過一個工具採取一個具體行動，例如執行一條命令或者編輯一個檔案。佢跟住觀
察嗰個行動嘅結果，按住佢學到嘅嘢再推理一次，再行動一次，並一直交錯進行思考同行
動，直到佢斷定任務完成。呢個推理—行動—觀察嘅節奏，正正令一個代理感覺上唔似一個
自動完成（autocomplete），而更似一個工作者：佢唔淨係答嘢，佢係 *逐步噉攻克問題*。

ReAct 係 Loop 嘅一個概念上嘅先祖。兩者都建基於同一個基本形態——一個思考、執行、
檢查結果嘅循環。但喺 *呢個循環活喺邊度* 同 *邊個控制佢* 呢方面，有一個關鍵嘅分別。

### 單一對話內 vs. 外部系統

喺一個普通嘅 ReAct 代理入面，推理—行動嘅循環活喺 **一段對話入面**。代理喺單一工
作階段、喺佢自己嘅脈絡窗口入面進行呢個循環，並自行判斷幾時停。咁係好強大，但佢承
繼咗單一運行嘅所有限制：對話最終會結束、脈絡窗口會塞滿，而代理同時係工作者 *同
埋* 佢自己工作嘅唯一裁判。冇一把獨立嘅聲音去問「呢樣嘢究竟啱唔啱？」，亦冇任何記
憶喺工作階段關閉之後仲存活。

一個 Loop **將嗰個推理—行動循環由單一代理對話中抽出嚟，並將佢變成一個外部、可控
嘅系統**。同樣嘅 思考 → 執行 → 檢查 節奏依然喺度行——但而家佢係喺代理 *周圍* 行，
而唔淨係喺佢入面行：

| 喺單一 ReAct 對話入面…… | 喺一個 Loop 入面，循環被抽出，所以…… |
|---------------------------------|------------------------------------------|
| 代理喺內部推理 | Loop 框定目標並提示代理 |
| 代理透過佢嘅工具行動 | 代理運行（harness）執行嗰個行動 |
| 代理判斷自己嘅結果 | 一個外部 **Evaluator（評估器）** 驗證輸出 |
| 脈絡係唯一嘅記憶 | **state（狀態）** 被持久化保存喺任何單一運行之外 |
| 代理決定佢「做完咗」 | Loop 套用一個明確嘅 **停止條件** |

重點唔係話內層嘅 ReAct 循環會消失——每次代理運行依然喺內部推理同行動。重點係一個
Loop 喺成個運行周圍包裹咗一個 *外層* 嘅推理—行動循環，而最關鍵嘅係，**將控制權由
代理手中移出嚟、移入一個你設計嘅系統入面**。關於「下一步做乜」嘅推理，同埋「再次
運行代理」嘅行動，唔再被困喺一段你要監督嘅對話入面；佢哋變成一個系統嘅組成部分，
而呢個系統可以自行運作、由外部判斷，並按目標所需橫跨任意多次運行噉持續落去。

如果第 03 章俾咗你一個會自己轉動嘅圓圈嘅畫面，咁 ReAct 就係嗰個圓圈嘅形態——推
理、行動、觀察——而一個 Loop 就係當你將嗰個圓圈由單一聊天窗口中抽出嚟、並建造一
部機器去轉動佢之後，你所得到嘅嘢。

## Ralph Loop：指明方向嘅前身

仲有最後一塊源流要介紹，而佢係當中最直接嘅先祖：**Ralph Loop（Ralph 迴圈）**。

Ralph Loop 係 Loop Engineering 所演化自嘅前身技術。喺佢最原始嘅形態，佢簡單到令
人驚訝：一個基本嘅 shell 迴圈，反覆噉餵俾代理新鮮嘅脈絡、加上上一次運行嘅輸出同
錯誤，然後一次又一次噉再運行佢，直到任務完成。佢冇咩精巧嘅框架——淨係一個洞見：如
果你不斷用代理上次產出嘅嘢去重新啟動佢，佢就可以橫跨多次迭代、一步步噉磨向目標，
而唔係喺一次運行結束時就放棄。

嗰個簡單嘅 bash 迴圈係本書一切嘅種子。佢以可以想像得到嘅最精簡方式，示範咗個核心
構想——*喺一個迴圈入面運行代理，並將結果反饋返入去*。Loop Engineering 就係當你認
真對待嗰粒種子、並將佢培育成一個生產級系統之後所得到嘅嘢：你加入隔離（isolation）
令每次運行都安全、加入一個獨立嘅 Evaluator 令「做完」係經判斷而非假設、加入持久化
嘅 state 令進度係真實嘅而唔淨係脈絡窗口入面碰巧存活落嚟嘅嘢，並加入明確嘅停止條
件令 Loop 知道幾時收手。Ralph Loop 係概念驗證（proof of concept）；Loop
Engineering 就係嗰份工程。

我哋喺呢度淨係簡短噉介紹 Ralph Loop，當佢係由「一個聰明嘅 shell 小技巧」過渡到「一
個經設計嘅系統」嘅概念橋樑。佢更完整嘅故事——邊個原創咗佢、喺幾時、以及 *Loop
Engineering* 呢個詞本身點樣被命名同推廣開嚟——係下一章嘅主題。

## 重點摘要

- Loop Engineering 係一條四階段源流嘅頂點，當中每個階段都將人類嘅設計目標沿住技
  術棧再向上推：**Prompt Engineering**（雕琢一條指令嘅 *字詞*）→ **Context
  Engineering**（篩選整理脈絡窗口入面嘅 *輸入*）→ **Agent Harness Engineering**
  （透過工具同反饋令 *一次代理運行* 變得可靠）→ **Loop Engineering**（以外部驗
  證、持久化狀態同停止條件去編排 *多次運行*）。
- 每個階段都補足咗前一個階段所欠缺嘅嘢：Context Engineering 為好嘅用字加入正確嘅
  資訊；Agent Harness Engineering 為靜態輸入加入行動同反饋；Loop Engineering 為單
  一可靠運行加入跨運行嘅外部控制同迭代。
- **ReAct** 全寫係 **Reason + Act（推理 + 行動）**——喺呢個模式入面，代理將推理步
  驟同工具行動交錯進行，觀察結果並一直繼續，直到任務完成。佢係 Loop 嘅概念先祖。
- 一個 Loop **將推理—行動循環由單一代理對話中抽出嚟**，並將佢變成一個外部、可控嘅
  系統：內層代理依然推理同行動，但 Loop 喺佢周圍包裹咗一個外層循環，去提示、由外
  部驗證、持久化狀態，並決定幾時停——將控制權由代理手中移出嚟、移入一個你設計嘅系
  統入面。
- **Ralph Loop**——一個精簡嘅 shell 迴圈，反覆噉用新鮮脈絡加上之前嘅輸出同錯誤去重
  新運行一個代理——係 Loop Engineering 所演化成生產級系統嘅直接前身。佢完整嘅起源
  故事，緊接住就嚟。
