在这一章里，你会了解到 Loop Engineering（循环工程）究竟从何而来。它有一条由四个阶段构成的发展脉络，从 Prompt Engineering（提示工程）开始，发展到 Context Engineering（上下文工程），再到 Agent Harness Engineering（智能体框架工程），最后才是 Loop Engineering。我们会一个阶段一个阶段地看，看看每一个阶段相比前一个，究竟多出了些什么。你还会了解到，一个 Loop 和 ReAct 这个模式有什么关系，一个 Loop 是怎样把“推理加行动”这个循环，从单个 agent 的对话里抽出来，变成一个由你亲手设计、可以控制的系统，还有一个叫做 Ralph Loop 的早期技巧，是怎样为后面的一切铺好了路。

我们就从一个不断向上攀爬的领域讲起。

Loop Engineering 不是凭空冒出来的。它是一个稳步发展的过程当中最新的一步。这个领域年复一年，一直把“人类负责设计的范围”向上、向外推，从一个个字词，慢慢推向整个系统。

在这条脉络里，每一个阶段都是在解决前一个阶段遗留下来的瓶颈。当某一层做得足够可靠、可以把它当成理所当然的时候，注意力就会向上转移到更高的一层。要明白 Loop Engineering 为什么会是现在这个样子，最快的办法就是去理解这个向上攀爬的过程。它其实是一个自然而然的结果，是在下面的一切终于做得足够好、足以在上面继续盖楼之后，才会出现的。

这条脉络有四个阶段。它从 Prompt Engineering 开始，讲的是字词。从那里发展到 Context Engineering，讲的是输入的内容。接着是 Agent Harness Engineering，讲的是怎样让一次运行做到可靠。最后爬到顶端，就是 Loop Engineering，讲的是多次运行，并且把它们一起编排起来。我们就一个阶段一个阶段，一层一层往上走。

第一个阶段是 Prompt Engineering，也就是雕琢字词的手艺。任何人开始使用语言模型的时候，学到的第一件事，就是你怎么问，会直接影响你得到什么。Prompt Engineering 就是去精心设计一句指令的措辞，引导模型给出一个好的答案。也就是说，你要选对用词、给出例子、指明格式，或者给它派一个角色，比如告诉模型，它是一名专业的审查员。

这里的工作单位，就是一个 prompt 加一个回应。工程师的手艺是语言层面的。你要找到那些字词，让这一个回合做得成功。这个阶段加入了什么呢？就是认识到，模型的输出是可以通过刻意设计的输入来控制的，同时也发展出一整套用来控制的技巧词汇，比如 few-shot 示例、逐步指示，以及角色设定。但它还有一些问题没有解决。一句措辞再好的 prompt，也只能处理刚好出现在消息里的资料。一旦任务需要一些模型从来没有接触过的事实，比如你的代码库、你的文档，又或者是昨天才做的决定，再聪明的措辞也不够用。

第二个阶段是 Context Engineering，也就是策划输入内容的手艺。下一个瓶颈，就是模型回答的时候究竟知道些什么。Context Engineering 就是刻意去组装放进模型 context window（上下文窗口）的资料。包括相关的文件、检索回来的文档、之前的对话、示例，以及指示，让模型基于正确的材料去推理，而不是只靠选得好的字词。

工作单位从单个 prompt，扩展到模型看到的整个输入。手艺也从措辞，转向挑选和组装。要检索什么、要包含什么、要舍弃什么，以及怎样把所有东西塞进一个有限的 window 里。相比 Prompt Engineering，它加入了什么呢？就是它把整个 context window 当成设计的范围，而不只是那句指令。再好的措辞配上错误的资料，一样会失败，而 Context Engineering 就是确保正确的资料一定要在场。不过它同样留下了一些没有解决的问题。就算措辞完美、配上策划得完美的 context，到头来还是只有一个回合。真正的工作，要求模型去行动。去执行指令、读文件、改代码、检查结果，然后再基于发生的事情去再次行动。一个静态的输入，根本做不到这一点。

第三个阶段是 Agent Harness Engineering，讲的是怎样让一次运行做到可靠。要让模型能够行动，你就要把它包在一个 harness（框架）里。那就是单次 agent 运行的可靠执行环境。Harness 给模型提供工具，比如一个 shell、一个文件编辑器，以及执行测试的能力。它把工具的结果反馈回模型的 context，它管理一个多步骤工作环节里的来来回回，还让整次运行保持稳定、可以重现。Agent Harness Engineering，就是把这个环境做好的手艺。

工作单位再一次扩展，从单个输入，扩展到一次完整的 agent 运行。也就是一个工作环节，其中模型会推理、通过工具采取行动、观察结果，然后一直继续，直到它认为任务完成为止。相比 Context Engineering，它加入了什么呢？就是在单个工作环节里加入了行动和反馈。模型不再只局限于基于固定输入去生成文字。它可以通过改文件和执行指令去改变世界，也可以对它观察到的东西作出回应。Harness 让这个行动和观察的循环，在一次运行里变得可靠。但它留下了什么没有解决呢？Harness 让一次运行可靠，但单次运行始终受限于一个对话。它会结束。它的 context 会被塞满。没有任何处在 agent 之外的人去判断这个结果是不是真的好、要不要从一张白纸重新来过，又或者要不要再继续一百个迭代。判断力以及坚持下去这件事，仍然落在盯着这个工作环节的人类身上。

第四个阶段是 Loop Engineering，也就是多次运行的编排。这里就是这本书的起点。Loop Engineering 就是去设计一个控制系统，去自主地向 AI 编码 agent 发出提示、观察、验证和迭代。它坐在 harness 之上。Agent Harness Engineering 是让单次 agent 运行做到可靠，而 Loop Engineering 则是搭起那一层编排层，一次又一次地去运行这些 agent，从外部去判断它们的输出，在各次运行之间记住进度，并且决定要继续、要分支出一个 sub-agent（子智能体），还是要停下来。

工作单位最后一次扩展，从单次运行，扩展到为了一个目标而推动的多次运行。手艺不再是语言层面的，甚至不再是资料层面的。它是系统层面的。你是在设计那台操作 agent 的机器。相比 Agent Harness Engineering，它加入了什么呢？就是跨越多次运行的外部控制和迭代。Harness 信任 agent 自己决定什么时候做完，但一个 Loop 加入了一个独立的 Evaluator（评估器）去检查成果、加入了可以超越任何单次运行而存在的状态、还有一个明确的停止条件。它把一次好的运行，变成一个会不断产生好运行的流程，一直到目标达成为止。

留意一下贯穿其中的这条主线。在每一个阶段，人类的设计目标都在向上移动。从字词，到输入，到单次运行，再到那个运行着各次运行的系统。Loop Engineering，纯粹就是到目前为止这个攀爬过程的最顶端。它就是那一刻，当你工程设计的东西，不再是一条发给 agent 的消息，而是一个替你管理 agent 的系统。

现在我们就转去讲 ReAct 的关系。要理解一个 Loop 把什么机制外部化了，先说出每一个现代编码 agent 核心那个模式的名字，会很有帮助。那个模式就是 ReAct，也就是 Reason plus Act，意思是推理加行动。

ReAct 这个模式，把两种步骤交织在一起。agent 会推理，也就是用自然语言去想清楚下一步应该做什么，然后它会行动，也就是通过一个工具去采取一个具体的动作，比如执行一个指令或者改一个文件。接着它会观察那个动作的结果，根据它学到的东西再推理一次，再行动，然后一直把思考和行动交织下去，直到它认为任务做完为止。这种推理、行动、观察的节奏，就是让一个 agent 不像一个自动补全，而像一个工人的原因。它不只是回答你，它是一步一步地去处理这个问题。

ReAct 是 Loop 在概念上的祖先。两者都建立在同一个基本形态之上，就是一个思考、做事、再检查结果的循环。但有一个很关键的区别，就是这个循环究竟在哪里运行，以及是谁去控制它。

想一想单个对话和一个外部系统之间的对比。在一个纯粹的 ReAct agent 里，推理和行动的循环是在一个对话里运行的。agent 在单个工作环节里、在它自己的 context window 里去执行它，自己判断什么时候应该停。这样很有威力，但它继承了单次运行的所有局限。对话最终会结束，context window 会被塞满，而 agent 既是那个工人，也是唯一一个判断自己成果的人。没有一个独立的声音去问一句，这件事是不是真的对，也没有任何记忆可以在工作环节关闭之后继续存活下去。

一个 Loop，把那个推理和行动的循环，从单个 agent 的对话里抽出来，变成一个外部的、可以控制的系统。同样的思考、做事、检查的节奏仍然在运行，但现在它是围着 agent 去运行，而不只是在它里面运行。看一下每一个部分是怎样转变的。在单个 ReAct 对话里，agent 是在内部推理，而一个 Loop 则是从外部去设立目标、向 agent 发出提示。本来 agent 会通过自己的工具去行动，在一个 Loop 里则是由 agent 那次运行，也就是 harness，去执行那个动作。本来 agent 会判断它自己的成果，一个 Loop 则引入一个外部的 Evaluator 去验证输出。本来 context 就是唯一的记忆，一个 Loop 则把状态保存在任何单次运行之外。本来是 agent 自己决定它做完了，一个 Loop 则套用一个明确的停止条件。

重点不是说里面那个 ReAct 循环会消失。每一次 agent 运行，在内部仍然会推理和行动。重点是，一个 Loop 在整次运行外面包住一个外层的推理和行动循环，而且最关键的是，它把控制权从 agent 里面移出来，移进一个由你设计的系统里。关于下一步做什么的推理，以及再运行 agent 一次的行动，不再困在一个你需要在场监督的对话里。它们变成一个可以自己运行、从外部去判断、并且可以在目标所需的任何多次运行里持续下去的系统的一部分。

如果第三章给了你一个画面，是一个自己会转的圆圈，那么 ReAct 就是那个圆圈的形状，推理、行动、观察，而一个 Loop 就是当你把那个圆圈从一个聊天窗口里拿出来、再搭起一台机器去转动它的时候，所得到的东西。

这就带我们去讲 Ralph Loop，那个指明了方向的前身。还有最后一块脉络要介绍，而它是所有里面最直接的祖先。

Ralph Loop 就是 Loop Engineering 由它演化出来的那个前身技巧。它最初的形态出奇地简单。它就是一个基本的 shell 循环，不断把新鲜的 context，加上上一次运行的输出和错误，喂给 agent，然后一次又一次地运行它，直到任务完成为止。没有什么复杂的框架。只有一个洞见，就是如果你不断用 agent 上次产出的东西去重新启动它，它就可以跨越很多个迭代，慢慢磨向一个目标，而不是在一次运行的末尾就放弃。

那个简单的 shell 循环，就是这本书里一切的种子。它用一个你能想象到的最精简的方式，示范了这个核心想法，就是在一个循环里运行一个 agent，再把结果喂回去。Loop Engineering，就是当你认真对待这粒种子、再把它培育成一个生产级别的系统的时候，所得到的东西。你加入隔离，让每一次运行都安全；加入一个独立的 Evaluator，让“完成”是经过判断的，而不是靠假设；加入保存下来的状态，让进度是真实的，而不是只在一个 context window 里残存的那种；还加入明确的停止条件，让 Loop 知道什么时候应该收手。Ralph Loop 是概念验证。Loop Engineering 才是工程。

我们在这里只是简短地介绍 Ralph Loop，把它当成从一个聪明的 shell 小把戏，到一个经过设计的系统之间那道概念桥梁。它更完整的故事，包括是谁原创的、在什么时候，以及 Loop Engineering 这个词本身是怎样被命名和推广开来的，就是下一章的主题。

我们来总结一下重点。Loop Engineering 是一条四个阶段脉络的最顶端，在每一个阶段，人类的设计目标都在整个 stack 里再向上移动。它从 Prompt Engineering 开始，在那里你雕琢一句指令的字词。接着是 Context Engineering，在那里你策划 context window 里的输入。再来是 Agent Harness Engineering，在那里你通过工具和反馈，让一次 agent 运行做到可靠。最后就是 Loop Engineering，在那里你用外部验证、保存下来的状态，以及停止条件，去编排多次运行。

每一个阶段，都加入了前一个阶段所欠缺的东西。Context Engineering 在好的措辞上面加入了正确的资料。Agent Harness Engineering 在静态输入上面加入了行动和反馈。而 Loop Engineering 则在单次可靠的运行之上，加入了跨越多次运行的外部控制和迭代。

记住，ReAct 也就是 Reason plus Act，推理加行动。它就是那个模式，agent 在里面把推理步骤和工具动作交织在一起，观察结果，然后一直继续，直到任务完成为止，而它就是 Loop 在概念上的祖先。

一个 Loop，把推理和行动的循环从单个 agent 的对话里抽出来，变成一个外部的、可以控制的系统。里面那个 agent 仍然会推理和行动，但 Loop 在它外面包住一个外层循环，去发出提示、从外部去验证、保存状态，并且决定什么时候应该停，把控制权从 agent 里面移出来，移进一个由你设计的系统里。

最后，Ralph Loop，一个精简的 shell 循环，会不断用新鲜的 context，加上之前的输出和错误，去重新运行一个 agent，它就是那个直接的前身，让 Loop Engineering 演化成一个生产级别的系统。它完整的起源故事，就在下一章。
