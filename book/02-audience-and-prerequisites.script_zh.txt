在这一章里，你会搞清楚这本书是写给哪些人看的、它假设你已经具备哪些背景知识，以及它如何处理任何超出这个背景的概念。读到最后，你就能在投入时间之前，确认这些内容是否适合你目前的水平。

那么，我们就从"这本书是写给谁的"开始讲起。这本书是写给软件工程师和技术从业者的，他们已经在使用 AI 编程代理（AI coding agent），但才刚刚开始接触如何自主地去调度它们。

如果你曾经打开一个 AI 编程助手，描述一个改动，看它的回应，然后再输入一条跟进指令去引导它，那么你正是这本书心目中的那位读者。你习惯了去做循环里的那个"人"。你审视每一个结果，你判断哪里有问题，然后你亲手提示下一步。你现在还做不到的，是把这个监督的角色，交给一个由你自己设计的系统。

这一步跨越，正是这本书要帮你实现的。它不会教你什么是 AI 编程代理，也不会教你如何写好一个提示。它假设你已经会做这些。相反，它会带你从"一次一回合地去操作一个代理"，进化到"工程化那个替你操作它的循环（Loop）"。这里的重心，是围绕着代理的那个调度系统（orchestration system）。也就是说，发现要做的工作、在隔离环境里运行代理、验证结果、记住进度，以及决定是继续还是停止。它讲的不是如何写好一个提示这种手艺。

有几类读者会在这里觉得格外自在。第一类，是每天都依靠 AI 编程代理的在职工程师，他们想从"一回合一回合地盯着代理"当中把自己的专注力夺回来。第二类，是技术主管（tech lead）和平台工程师（platform engineer），他们正在评估，自主代理工作流是否值得在自己团队里采用，也想看到一幅关于好处和风险的诚实图景。第三类，就是那些喜欢折腾的人和早期采用者，他们在一个 shell 脚本里跑过一个简单的代理循环，想了解如何把它进化成一些可靠的东西。

那么，如果你从来没用过 AI 编程代理，你一样可以跟着读下去。不过，只要你有一小段时间亲手用过它，每一章都会落地得更加扎实。

我们聊聊你应该已经会的东西。为了让本书聚焦在 Loop Engineering（循环工程）本身，它假设你有一点先备知识。没有一样是高深的。全都是一个在职工程师通常已经具备的背景。

第一，是版本控制，特别是 Git。你应该对基本操作有信心：提交（commit）、分支（branch），以及"工作可以隔离在一条分支上，之后再合并回去"这个概念。循环很依赖隔离，也就是说把每一次代理的尝试放在一个独立的地方去运行，这样一个糟糕的结果就永远不会污染到你的主代码。本书会用 Git 的概念，比如分支和 worktree，去解释这种隔离是如何做到的。顺便说一句，worktree 是 Git 的一个功能，可以让你同时把多于一条分支签出到不同的工作目录里。本书在需要的地方都会提醒你，所以对分支有一点认识就已经够开始了。

第二，是命令行（command line）的使用。你应该会在终端（terminal）里读和运行指令。那些实操例子从一个简单的 shell 循环开始，再慢慢搭建起来，所以假设了你看得懂一段短的 bash 脚本。也就是说：变量、一个循环，以及调用一个指令。你不需要是 shell 脚本专家。你只需要对终端不感到害怕，又能看得出几行代码在做什么。

第三，也是最重要的一样，就是同 AI 编程代理互动。你应该亲手用过至少一个 AI 编程代理。比如，提示它去写或者改代码，再读它做出来的东西。这是最重要的先备条件，因为整本书就是把这个你熟悉的经验重新框定。你今天亲手做的那件事，就是一个循环将会替你去做的那件事。你需要有一种切身的感觉：这个代理做得好的地方在哪、它出错的地方在哪、以及为什么现在一定要有个人留在循环里。这样你才能体会到，要把这个人抽走，到底需要付出什么。

也有一些东西是你不需要的。你不需要有搭建代理框架（agent framework）的经验。你不需要懂任何特定的调度工具或者供应商。你不需要有机器学习或者模型训练的背景。而你也不需要任何特定的编程语言，只要你看得懂实操章节里那些简短而有注释的例子就行。

现在我们来看看这本书如何处理新概念。Loop Engineering 是建立在几个概念之上的。有些你会懂，有些你可能还不懂。本书遵守一条很硬性的规则：任何超出我们刚刚讲的那些先备知识的概念，在被依赖之前都会先定义清楚。

具体做法上，这分三个方面进行。第一，那些术语会在被使用之前先介绍。当一章需要用到一个概念，比如生成代理（Generator Agent）、评估器（Evaluator）、执行框架（Harness），又或者停止条件（Stopping Condition），它会在第一次出现的那一刻，用浅白的语言去解释这个术语，而不是假设你已经记得它的定义。第二，缩写在第一次出现时就会写出全称。比如说 ReAct 模式，它代表 Reason 加 Act（也就是推理加行动），在第四章第一次介绍的时候会讲出全称，本书才会用那个简写。第三，有一个词汇表（glossary）支撑着整本书。每一个关键术语都在词汇表里有一个单一而一致的定义，所以就算你跳到后面的章节，你也随时可以查回这个术语，而不必到处去找它第一次在哪里解释过。

目标就是要让你永远都不会撞上一堵解释不了的行话高墙。如果你满足了全部三个先备条件——版本控制、命令行，以及亲手用过 AI 编程代理的经验——那么本书需要的其他所有东西，它都会在过程中慢慢教给你。

那我们就来总结一下重点。这本书是写给那些已经在用 AI 编程代理，但还没搭建起系统去自主调度它们的软件工程师和技术从业者。它假设了三个先备条件：通过 Git 对版本控制有认识、对命令行的使用有信心，以及有亲手同 AI 编程代理互动的经验。它没有假设你有搭建代理框架的经验、懂任何特定工具或者供应商，又或者有机器学习的背景。最后，任何超出那些先备条件的概念，在被使用之前都会先定义，缩写在第一次使用时会写出全称，而词汇表则会充当术语的单一事实来源。
