In this chapter, you're going to learn where Loop Engineering came from. There's a four-stage lineage that runs from Prompt Engineering, to Context Engineering, to Agent Harness Engineering, and finally to Loop Engineering, and we'll look at what each stage adds beyond the one before it. You'll also learn how a Loop relates to the ReAct pattern, how a Loop lifts the reason-act cycle out of a single agent conversation and turns it into a controllable system that you design, and how an earlier technique called the Ralph Loop set the stage for everything that follows.

Let's start with a field that kept climbing the stack.

Loop Engineering did not appear from nowhere. It's the latest step in a steady progression. This is a field that, year after year, has pushed the boundary of what the human designs up and outward, away from individual words and toward whole systems.

Each stage in this lineage solves the bottleneck left by the stage before it. When one layer becomes reliable enough to take for granted, attention moves to the next layer up. Understanding this climb is the fastest way to understand why Loop Engineering looks the way it does. It's the natural consequence of everything underneath it finally working well enough to build on.

The lineage has four stages. It begins with Prompt Engineering, which is about the words. From there it moves to Context Engineering, which is about the inputs. Next comes Agent Harness Engineering, which is about making one run reliable. And it tops out at Loop Engineering, which is about many runs, orchestrated together. Let's walk up that stack one stage at a time.

Stage one is Prompt Engineering, the craft of the words. The first thing anyone learns when working with a language model is that how you ask changes what you get. Prompt Engineering is the practice of crafting the wording of a single instruction to steer the model toward a good answer. That means choosing your phrasing, giving examples, specifying a format, or assigning a role, like telling the model it's an expert reviewer.

The unit of work here is one prompt and one response. The engineer's craft is verbal. You find the words that make this one turn succeed. What this stage adds is the recognition that the model's output is controllable through deliberate input, along with a whole vocabulary of techniques for that control, like few-shot examples, step-by-step instructions, and role framing. But what it leaves unsolved is this. A single well-worded prompt still operates on whatever information happens to be in the message. The moment a task needs facts the model was never given, like your codebase, your docs, or yesterday's decision, clever wording is just not enough.

Stage two is Context Engineering, the craft of curating the inputs. The next bottleneck is what the model knows when it answers. Context Engineering is the practice of deliberately assembling the information placed in the model's context window. That includes the relevant files, retrieved documents, prior conversation, examples, and instructions, so the model reasons over the right material, not just well-chosen words.

The unit of work expands from a single prompt to the whole input the model sees. The craft shifts from phrasing to selection and assembly. What to retrieve, what to include, what to leave out, and how to fit it all into a limited window. What this adds beyond Prompt Engineering is that it treats the entire context window as the design surface, not just the instruction sentence. Good wording over the wrong information still fails, and Context Engineering makes sure the right information is present. But it still leaves something unsolved. Even a perfectly worded prompt over perfectly curated context is still just one turn. Real work requires the model to act. To run a command, read a file, edit code, check a result, and then act again based on what happened. A static input simply can't do that.

Stage three is Agent Harness Engineering, which is about making one run reliable. To let a model act, you wrap it in a harness. That's the reliable execution environment for a single agent run. The harness gives the model tools, like a shell, a file editor, and the ability to run tests. It feeds tool results back into the model's context, it manages the back-and-forth of a multi-step session, and it keeps that whole run stable and reproducible. Agent Harness Engineering is the practice of building this environment well.

The unit of work expands again, from a single input to one complete agent run. That's a session in which the model reasons, takes actions through tools, observes the results, and continues until it believes the task is done. What this adds beyond Context Engineering is action and feedback within a single session. The model is no longer limited to producing text over fixed inputs. It can change the world by editing files and running commands, and it can respond to what it observes. The harness makes that loop of acting and observing dependable for one run. But here's what it leaves unsolved. A harness makes one run reliable, but a single run is still bounded by one conversation. It ends. Its context fills up. Nobody outside the agent decides whether the result was actually good, whether to try again from a clean slate, or whether to keep going for another hundred iterations. The judgment and the persistence still live with the human watching the session.

Stage four is Loop Engineering, the orchestration of many runs. This is where this book begins. Loop Engineering is the practice of designing a control system that prompts, observes, verifies, and iterates AI coding agents autonomously. It sits on top of the harness. Where Agent Harness Engineering makes a single agent run reliable, Loop Engineering builds the orchestration layer that runs agents over and over, judges their output from the outside, remembers progress between runs, and decides whether to continue, branch off a sub-agent, or stop.

The unit of work expands one final time, from a single run to many runs driven toward a goal. The craft is no longer verbal, or even informational. It's systemic. You are designing the machine that operates the agent. What this adds beyond Agent Harness Engineering is external control and iteration across runs. A harness trusts the agent to decide when it's finished, but a Loop adds an independent Evaluator that checks the work, state that outlives any single run, and an explicit stopping condition. It turns one good run into a process that keeps producing good runs until the goal is met.

Notice the through-line here. At every stage, the human's design target moves up. From the words, to the inputs, to the single run, to the system that runs runs. Loop Engineering is simply the top of that climb so far. It's the point at which the thing you engineer is no longer a message to an agent, but a system that manages the agent for you.

Now let's turn to the ReAct connection. To understand the mechanism a Loop externalizes, it helps to name the pattern at the core of every modern coding agent. That pattern is ReAct, which stands for Reason plus Act.

The ReAct pattern interleaves two kinds of steps. The agent reasons, which means it thinks through what to do next in natural language, and then it acts, which means it takes a concrete action through a tool, such as running a command or editing a file. It then observes the result of that action, reasons again in light of what it learned, acts again, and keeps interleaving thought and action until it concludes the task is done. This reason, act, observe rhythm is what makes an agent feel less like an autocomplete and more like a worker. It doesn't just answer, it works the problem step by step.

ReAct is a conceptual ancestor of the Loop. Both are built on the same fundamental shape, a cycle of thinking, doing, and checking the result. But there's a crucial difference in where that cycle lives and who controls it.

Think about the contrast between a single conversation and an external system. In a plain ReAct agent, the reason-act cycle lives inside one conversation. The agent carries it out within a single session, in its own context window, judging for itself when to stop. That's powerful, but it inherits all the limits of a single run. The conversation eventually ends, the context window fills, and the agent is both the worker and the only judge of its own work. There's no independent voice asking whether this is actually correct, and no memory that survives once the session closes.

A Loop lifts that reason-act cycle out of the single agent conversation and turns it into an external, controllable system. The same think, do, check rhythm still runs, but now it runs around the agent rather than only inside it. Consider how each part shifts. Where a single ReAct conversation has the agent reasoning internally, a Loop frames the goal and prompts the agent from the outside. Where the agent would act through its own tools, in a Loop the agent run, that is, the harness, performs the action. Where the agent would judge its own result, a Loop brings in an external Evaluator to verify the output. Where context is the only memory, a Loop persists state outside any single run. And where the agent itself decides it's done, a Loop applies an explicit stopping condition.

The point is not that the inner ReAct cycle disappears. Each agent run still reasons and acts internally. The point is that a Loop wraps an outer reason-act cycle around the whole run, and crucially, it moves the control out of the agent and into a system you design. The reasoning about what to do next, and the acting of running the agent again, are no longer trapped in one conversation that you have to supervise. They become parts of a system that can run on its own, judge from the outside, and persist across as many runs as the goal requires.

If chapter three gave you the image of a circle that turns on its own, then ReAct is the shape of that circle, reason, act, observe, and a Loop is what you get when you take that circle out of a single chat window and build a machine to turn it.

That brings us to the Ralph Loop, the precursor that pointed the way. There's one more piece of lineage to introduce, and it's the most direct ancestor of all.

The Ralph Loop is the precursor technique that Loop Engineering evolves from. In its original form it's strikingly simple. It's a basic shell loop that repeatedly feeds an agent fresh context, plus the previous run's output and errors, and then runs it again and again until the task is done. There's no elaborate framework. Just the insight that if you keep restarting an agent with what it produced last time, it can grind its way toward a goal across many iterations, rather than giving up at the end of one.

That simple shell loop is the seed of everything in this book. It demonstrated the core idea, run an agent in a loop and feed results back in, in the most minimal way imaginable. Loop Engineering is what you get when you take that seed seriously and grow it into a production-grade system. You add isolation so each run is safe, an independent Evaluator so done is judged rather than assumed, persisted state so progress is real and not just whatever survives in a context window, and explicit stopping conditions so the Loop knows when to quit. The Ralph Loop is the proof of concept. Loop Engineering is the engineering.

We're introducing the Ralph Loop only briefly here, as the conceptual bridge from a clever shell trick to a designed system. Its fuller story, who originated it, when, and how the term Loop Engineering itself came to be named and popularized, is the subject of the next chapter.

Let's recap the key takeaways. Loop Engineering is the top of a four-stage lineage, where each stage moves the human's design target further up the stack. It starts with Prompt Engineering, where you craft the words of one instruction. Then comes Context Engineering, where you curate the inputs in the context window. Next is Agent Harness Engineering, where you make one agent run reliable through tools and feedback. And finally there's Loop Engineering, where you orchestrate many runs with external verification, persisted state, and stopping conditions.

Each stage adds what the previous one lacked. Context Engineering adds the right information to good wording. Agent Harness Engineering adds action and feedback to static inputs. And Loop Engineering adds external control and iteration across runs to a single reliable run.
