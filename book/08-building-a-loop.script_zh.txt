在这一章里，我们会从"理解一个 Loop（循环）"走到"真正动手做一个出来"。我们会从最简单、最有可能行得通的东西开始，做一个秉承 Ralph Loop 精神的骨架版 Loop，然后再慢慢把它养大，变成一个生产级别的系统，一个有适当隔离、有独立评估器、有状态可以保存下来、又有清晰停止条件的系统。到最后，我们会把设计一个 Loop 的目标、它的评估准则以及护栏的常见模式，提炼出来。我们的目标，是让你自己也可以搭出一个行得通的 Loop，并且能够清晰地去思考怎样让它变得稳健。

我们就从概念走到实现吧。前面几章给了你各种零件，又示范了这些零件是怎样运作的。这一章就是把一把螺丝刀放到你手里。要真正理解一个 Loop，最快的方法就是做一个最小、可以跑得起来的版本，看清楚它在哪里脆弱，然后逐个逐个地加结构去补上每一个弱点。

所以我们特意从小开始。第一个例子就是最小化的 Loop。它只有几行，抓住了最核心的想法，就是叫一个 agent（代理）做事、把结果喂回给它、再叫它做一次。没有别的东西。它是特意缺少了每一样生产上要关心的东西。把它剥到一丝不挂，就会让那些缺少了的关注点变得很明显，而这一点正正就是为了铺垫之后那个更认真的版本。

我们就从这个最小化 Loop 开始，也就是 Ralph 风格的例子。Ralph Loop 是这本书里面每一个 Loop 的精神祖先。它是一个简单的 shell loop，会不断地把新鲜的上下文，加上上一次跑出来的结果，交给一个 agent，然后一直跑下去，直到这件工作做完为止。

用浅白一点的说法，这段小小的 script 在做的事情是这样的。它握着一条长期指令，类似"实现我们待办清单上的任务，并且让整个测试套件通过"这样。它会保存着一段不停更新的上下文，把上一次尝试发生过的事情带下去。然后它进入一个 Loop，会不停重复，直到一个特别的信号出现为止。在这个很粗糙的版本里面，要靠这个 agent 自己在它觉得做完之后，写出"all done"这一句字面文字，而这个 Loop 就只是单纯地盯着有没有出现这一句东西。

每一次跑过这个 Loop，都会发生三件事。第一，这个 Loop 叫那个写程序的 agent，把那条长期指令连同上一次尝试的输出和错误一起交给它，并且抓住这个 agent 打印出来的所有东西，包括它对改动的总结、它跑过的任何指令的输出、以及任何错误。第二，它把所有这些抓到的输出，折叠进下一轮的上下文里面，这样下一次这个 agent 跑的时候，就可以看到它刚刚做了什么、出了什么错。然后这个 Loop 又再跑一次。当那个信号终于出现，这个 Loop 就会宣布它做完了。

这里真的只有三行做完所有事情，而每一行都对应到前面几章的一个概念。Loop 的条件，也就是那个盯着有没有出现"all done"的判断，就是这个 Loop 的停止条件（Stopping Condition），也就是最最粗糙的形态。这个循环一直重复，直到那个哨兵字符串在这个 agent 的输出里面出现。这个就是生命周期那一章讲过的"继续还是停"的决定，只不过简化成一个字符串比对。叫 agent 的那一步，就是生成器 agent（Generator Agent，负责动手做事的那个）在那里动手干活，读任务、改文件、跑指令、打印结果。而上下文回收那一步，也就是上一次的输出被拉回到下一个提示那里，就是让一个 Loop 真正成为一个 Loop 的反馈路径。上一次尝试的输出和错误被折叠进下一次尝试的上下文里面，所以每一次尝试都是在上一次的基础上再走，而不是盲目地从头来过。

那么现在呢，这个例子之所以有用，正正是因为它省略了什么东西。它是特意没有隔离（Isolation）的。这个 agent 直接在工作目录里面改文件。没有一个独立的工作空间，也没有一条独立的分支（branch），所以单单一次差劲的迭代，就可以把你的工作弄脏，而且没法回滚。

它也没有一个独立的评估器（Evaluator，独立检查工作的角色）。"做完"纯粹是这个 agent 说做完就做完。盯着有没有"all done"这一句，也就是完全相信这个 agent 自己的说法。没有任何外部的东西去跑测试或者审视那些改动，所以这个 Loop 真的分不出哪样是好东西、哪样是很有自信的废话。

它也没有保存状态（State，也就是记忆／Memory）。唯一的记忆就是一个草稿文件，而每一次迭代都会覆盖它。没有一个耐用的进度记录，没有一份就算崩溃也能存活的任务清单，也没有办法从停下的地方继续做下去。

除了以上这些，它也没有一个真正的预算护栏（budget guardrail）。如果这个 agent 永远都不打印出那句魔法字句，这个 Loop 就会永远地跑下去。而这些呢，不是要在那里修修补补的 bug。它们正正就是生产级别的 Loop 要去填上的缺口。所以你心里就握着这个最小化版本，把它当成我们即将要加固的基准线。

我们把它进化成一个生产级别的 Loop，一个你真的可以信任它在没人盯着的情况下跑下去的 Loop。结构维持不变，叫生成器 agent、把结果折回到下一次尝试，但是最小化版本里面的每一个缺口，现在都会由一件明确的机器零件去填上。每一样新加的东西，都直接对应到解剖那一章那八个组件之中的一个，也对应到七个阶段的生命周期里面的一个阶段。

把这两个参考框架同时握在手里，会有帮助。我们现在加的，是四样生产上要关心的东西。第一，把每一次尝试在一个用完即弃、可以丢掉的工作空间和分支里面去跑，这个就是隔离组件，它对应到生命周期里面"在一个隔离的工作空间里面孵化 agent"那个阶段。第二，把"接不接受"这件事，关在一个独立检查上面，这个就是评估器，它对应到"评估器检查输出"那个阶段。第三，把进度记录到一个耐用的文件里面，这个就是状态和记忆，对应到"保存状态"那个阶段。第四，在成功、在一个门槛、或者在一个预算上限那里终止，这个就是停止条件，它对应到最后那个"我们决定继续、孵化、还是停"的阶段。

生成器 agent 仍然是中间那个工人。触发这个 script 的那个扳机，就是自动化组件。而被回收的反馈，也就是评估器的发现流入下一个提示那里，就是生命周期里面"我们把反馈喂回、再重复"那个阶段。现在不同了的，是没有任何这些东西还是隐含的。

第一步，是把配置外部化。我们把这个 Loop 的各个旋钮，从这个 script 里面抽出来，放进一个独立的配置文件。停止条件就是在这里以数据的形式存在，而不是埋藏在逻辑里面，这样就让那些预算可以被审计、又容易调校。

那个配置文件用浅白的声明把所有东西写出来。它列明这个目标，类似"实现待办清单上面下一个未勾的任务，并且让这个套件变绿"这样。它点名我们要叫哪个生成器 agent，以及它用的模型。它描述这个评估器，而这里就是最关键的那部分，这个评估器是独立于这个生成器的。它不相信这个 agent 自己的报告。它反而会跑真正的指令，然后给结果评分。所以这个配置文件点名一条客观的测试指令，这条指令必须成功；一条可选的评论家（critic）指令，拿一个第二意见；以及一个评分标准（rubric）门槛，也就是这件工作要过得了才会被接受的最低分数。

这个配置文件也铺排好那些停止条件，一个对迭代次数的硬上限、一个以美元计的预算护栏（如果花费太多就会停下这个 Loop）、以及一个成功条件（当测试套件变绿、而且评分标准也满足了，就接受这件工作）。它描述了隔离的设置，也就是基础分支以及那些用完即弃的工作空间应该放在哪里。它也点名那个在迭代之间保存进度的文件。这里每一个字段，都是把一样生产上要关心的东西明确化。单单是那些停止条件，已经填上了最小化版本那个"永远地跑"的缺口，因为这个 Loop 现在有一个成功出口、一个评分标准出口、以及两个独立的预算上限。

第二步，是那个会读这个配置文件、然后去跑加固版 Loop 的协调器 script。你可以把它当成一个有导游的游览，每一个有标签的段落，就是四样新加的东西之中的一样。

这个 script 一开始会从配置文件载入所有那些设置，也就是目标、测试指令、评论家指令、门槛、迭代上限、成本上限、基础分支、以及状态文件。它设好计数器，记住现在是第几次迭代以及累积的花费，又设好一段空的上下文去把反馈带下去。

接着就是停止条件，它就坐在这个 Loop 的最顶。这个 Loop 的头部不再是盲目地盯着一个哨兵，而是把那些预算护栏写成真正的、由数据驱动的判断。这个 Loop 只会在"我们还在迭代上限之下、又在成本上限之下"的时候才会继续跑。一到达任何一个上限，它就会停。

每一次跑里面，第一件发生的事就是隔离。每一次尝试都在它自己一个用完即弃的工作空间里面跑，在它自己一条用完即弃、用尝试编号命名、从基础分支生出来的分支上面跑。因为这样，一次差劲的迭代永远都不可能弄脏主线的工作，或者弄脏另一个并行的尝试。如果一次尝试失败，我们只是删掉它的工作空间就行。这个正正就是最小化 Loop 所缺少的那种可逆性。

之后，生成器 agent 出手。这个 script 把那条长期目标、保存了的状态文件、以及上一次评估被回收回来的反馈，一起交给这个 agent。然后这个 agent 就在那个隔离的工作空间里面做它的事，改文件、跑指令。

这个 agent 做完之后，评估器接手，而这个就是独立验证。这个 agent 自己的意见在这里不算数。这里有两道闸。第一道闸是客观的，真正的测试套件必须通过。第二道闸是质性的，一个评论家会根据一份写下来的评分标准给这件工作评分，然后产生一个数字，比如在零和一之间某个位置。这个 script 也会把这次尝试的成本加进去，这样这个预算护栏就保持准确。

接着就是决定点。如果测试通过，而且评论家的分数达到或者超过这个门槛，两道闸都满足了，那么这次尝试就会被升格，合并回基础分支，而这个 Loop 就停。这个就是成功出口。

如果这次尝试过不了两道闸，这个 script 就会把学到的东西保存下来。它记下测试结果以及评论家分数，append（追加）到那个耐用的进度文件里面，这样下一次迭代就可以继续下去，而不是盲目地从头来过，也让那些进度就算崩溃也能存活。同一份反馈也会被回收进下一个提示。最后，那次被拒绝的尝试，它的工作空间就会被丢弃，不过那条分支就可以留下来做事后取证，等你想之后查回出了什么问题的时候可以看。

至于如果这个 Loop 在没有撞到那个成功出口的情况下掉了出来，那么就是说有一个停止条件停下了它，可能是迭代上限，又或者是预算。这个是一个受控的停止，不是失控。这个 script 会报告跑了多少次迭代，以及总共花了多少钱。

那么我们就把这四样关心的东西，逐个逐个跟最小化的基准线对回去，看清楚每一个缺口是怎样被填上的。讲隔离，最小化 Loop 是在那里直接改工作目录。现在每一次迭代都拿到一个全新、用完即弃、在用完即弃分支上面的工作空间。一次失败的尝试只是单纯被移走，而只有被接受的尝试，才会合并回来。这个 agent 做的任何东西，都弄不脏主线或者另一个并行尝试。

讲独立评估器，"做完"不再是这个 agent 说了算。一道客观闸，也就是测试必须通过，加一道评论家闸，也就是分数必须过门槛，两道都要成功，这件工作才会被接受。这个是对抗"很有自信但是其实做错了"的输出的主要防线。

讲保存状态，进度在每一次迭代之后都会 append 到一个耐用的文件，而那个文件会被喂回到下一个提示。这个 Loop 可以在崩溃之后继续，而每一次尝试都是在一段记录下来的历史上面再走，而不是在一个被覆盖来覆盖去的单一草稿文件上面走。

讲停止条件，现在的终止是明确而且多管齐下。这个 Loop 可以在成功（两道闸都绿）的时候停，可以在用尽最大迭代次数的时候停，也可以在超出成本上限的时候停。最小化 Loop 那种"永远跑下去"的失控模式，在这里结构上是不可能发生的。

留意一下有些东西是没变过的。这个心跳依然是生成、评估、反馈、重复。生产级别的 Loop 不是跟 Ralph 风格的 script 不同的想法。它是同一个想法，只是把每一个隐含的假设，换成一个明确、可以被检视的组件。而这个就是 loop engineering（Loop 工程）的整套动作。你拿解剖那一章的零件，沿着生命周期排好它们，然后把每一个控制面变成你看得到、调得到、又信得过的东西。

现在我们讲一下怎样从零件去搭一个 Loop，也就是一条务实的路径。我们刚刚走过那两个例子，展示了目的地。这一部分讲的就是路线。你不是坐下来一次过写好这个生产 Loop。你是把它养大，一次加一个组件，用一个让你在每一步都握着主导权的顺序去加。指导原则很简单。从最小、可以对着你真实任务跑得起来的 Loop 开始，然后只在一个组件不在那里真的让你痛的时候，才加它。

这里有一个可靠的顺序，可以从那八个组件去组装一个 Loop。先从生成器 agent 加一个停止条件开始。这个就是最小化 Loop，一个循环、一次 agent 调用、加一个粗糙的停止判断。立刻拿它对着一个真实任务去跑。你现在还不是要追求正确，你是要在你自己机器上面，从头到尾看到这个心跳运作得起来。

接着加隔离，要在你信任这个 Loop 去做任何真东西之前加。一旦这个 agent 在没人盯着的情况下改文件，一次差劲的迭代就可以弄脏你的工作空间。把每一次尝试包在一个用完即弃、在用完即弃分支上面的工作空间里面，这样每一次迭代都是可逆的。这个是整个系统里面最便宜的保险，所以要早早加它。

然后加评估器。这个是单一最高杠杆的组件，因为它正正是把一个 Loop 跟一个随机输出生成器分开的那样东西。从你已经有的最强客观信号开始，通常就是你现有的测试套件，之后才再叠上一道评论家或者评分标准闸。

之后，加保存状态。当那些迭代已经隔离了、又有闸把守，就给这个 Loop 一个耐用的记忆，一个任务文件、一个草稿簿、又或者一个项目看板，这样进度就算崩溃也能存活，而每一次尝试都是在一段记录下来的历史上面走，而不是盲目地从头开始。

接着，把停止条件收紧成明确的护栏。把那个粗糙的哨兵，换成由数据驱动的上限，一个成功出口、一个迭代上限、以及一个预算上限。

最后，只在你的任务真的要求的时候，才加上其余的组件。自动化或者触发器，比如排程作业、webhook、又或者斜杠指令；连接器，接到你的问题追踪器、即时通讯、或者源代码控制主机；以及技能与知识，比如一个技能文件、又或者你的编程规范，这些东西就是把一个你要亲手去跑的 Loop，变成一个自己会跑的 Loop。它们对一个无人值守的 Loop 来讲是必需的，但是它们对正确性来讲是毫无贡献，所以就把它们押后，直到那个"生成和评估"的核心已经值得信任为止。

这里的纪律是减法，不是加法。你每加一个组件，都是一个你现在要去理解和维护的控制面，所以在每一步刚刚好的问题是：省略这样东西，会不会在我的任务上面造成一个具体的失败？如果一个在本机上面对着一个绿色测试套件跑的 Loop 已经做得了这件工作，那么你就不需要一个 webhook 触发器或者一个即时通讯连接器。只在你的任务真的推你到那个地步的时候，才向着那个生产例子去成长。

现在我们深入一点讲一下目标、评估准则、以及护栏的模式。有三个设计决定，比任何其他东西都更加决定一个 Loop 的成败，就是你给它的目标、决定它的工作什么时候才算可接受的评估准则、以及约束它行为的护栏。那些组件只是管道。这三样东西才是工程判断所在的地方。我们就走一走那些值得首先拿来用的模式。

先从设计目标开始。一个 Loop 是在无人值守之下跑，所以它的目标必须清晰又可核查，要这样去措辞，让那个 agent 自己的乐观以外的东西，都可以判断到它究竟达到了没有。

第一个模式，是让这个目标可以由外部验证。挑一个它的完成度可以由一台机器去确认的目标。类似"让测试套件干干净净地退出，没有任何被跳过的测试"是可核查的。类似"改善那些程序代码"就不是。一个评估器核查不了的目标，就是一个 Loop 完成不了的目标。

第二个模式，是把范围界定在一个连贯的工作单元。一个由"重构支付模块，让所有现有测试仍然通过"去驾驭的 Loop，会收敛。一个瞄准"现代化整个代码库"的 Loop，就会四处游荡。窄的目标，才给得到反馈路径在每一次迭代都有些具体的东西去顶住。

第三个模式，是把目标写成一个结果，而不是一个程序。描述你想要的最终状态，"这个套件是绿的，而且这个新端点成功了"，而不是去到那里的步骤。步骤由这个 agent 挑，这个 Loop 就验证那个结果。

一个这样写的目标，可以直接放进配置文件的目标字段。举个例子，一个好的目标可能是："给 orders 端点加分页功能，让它接受 page 和 limit 参数、最多返回所要求的项目数量、并且让现有测试保持绿色。"这个是可核查的，因为现有和新的测试就是"做完"的客观信号。它也是有界定的，因为它是一个端点、一个行为，而不是"改善 orders API"。

接着，设计评估准则。评估器是这个 Loop 对抗"很有自信但是做错了"的输出的防线。一个值得信任的评估器，有一个界定性的特征。它是独立于这个生成器 agent 的。它永远不会把这个 agent 自己的报告当成成功的证据。好的评估通常会叠两种检查。

第一种，是一道难以作假的客观闸。测试、类型检查、linter、一个构建步骤、又或者 schema 验证，任何会返回一个明确的"通过或者失败"、而这个 agent 怎么说都说不掉的东西。这个是值得信任的评估的骨干，所以从这里开始。

第二种，是一道评论家或者评分标准闸，用来补上客观检查漏了的东西。一个测试套件确认到行为，但是它对于可读性、安全性、又或者这个改动究竟有没有真的符合意图，就讲得很少。一个第二的 agent 根据一份写下来的评分标准去给这件工作评分，就补上这个缺口。把评分标准保持明确，并且为它做版本控制，这样这个 Loop 要达到的标准就看得到、又调得到。

两道闸组合成单一条接受规则。只有在客观闸通过、而且评分标准过得了它那个门槛的时候，才接受。用浅白的说法，规则是这样运作的。它首先检查客观闸，如果真正的测试套件没通过，答案就是不，拒绝。然后它检查质性闸，如果评论家的分数在门槛之下，答案又是不。只有当两道闸都满足的时候，它才返回是，升格这次尝试，然后停。这个是一个独立、两道闸的检查，这个 agent 根本没法自我认证。

还有两个模式，让一个评估器在务实上值得信任。把客观闸在这个 agent 工作那个一模一样的隔离工作空间里面去跑，这样你评分的就正正是它产出那样东西。并且要审慎地设定评分标准门槛。太低，那些随便的东西就会穿过。太高，这个 Loop 就会为了追一个它永远都达不到的分数，而烧光它整个预算。一个你守得住、又会随着你学习而调整的门槛，好过一个随便定的门槛。

最后，设计护栏。护栏就是那些停止条件，加上那些让一个无人值守的 Loop 不要造成破坏的隔离和升级规则。评估决定这件工作好不好，而护栏就决定这个 Loop 在有东西停下它之前，可以走到多远。拿这些模式来用吧。

永远设多过一个预算。为迭代设上限，又为花费设上限。单是其中一个，都有个盲点。一个便宜但是卡住了的 Loop，可以在它那个迭代次数里面打转；而一个昂贵的单一尝试，就可以一下子爆了那个美元预算。两个夹在一起，就把两种失控模式都箍住。

隔离每一次尝试，并且只在成功的时候才合并。可逆性本身就是一道护栏。如果一个被拒绝的尝试是住在一个用完即弃的工作空间里面，那么一次差劲的迭代，代价就只是删掉那个工作空间，而不是一条被弄脏了的主分支。

并且定义一条明确的路径，去升级回给一个人类。一个 Loop 应该知道什么时候要停下来发问。当它用尽了预算都还没被接受、当它在那里震荡但是没进步、又或者当它触发了一条高风险的引线，它就必须交回给一个人，而不是悄悄地失败，又或者继续烧 token。

这些规则是住在配置文件里面，连同一个升级钩子一起。那些停止条件设定迭代上限，抓住那些"卡住但是便宜"、在那里原地打转的 Loop；设定成本上限，独立于迭代次数地抓住一个昂贵的 Loop；以及设定成功出口，也就是客观和评分标准两道闸都过了的时候。升级设定就讲明，当一个预算撞到了但是还没被接受的时候应该怎么做，不要悄悄地失败，交回给人，可能是开一个 issue，又或者发一个消息去即时通讯。而如果同一个失败连续重复了某个既定次数，就升级。

留意一下这里每一个模式，都是绑回到解剖那一章的一个组件，以及生命周期的一个阶段。目标塑造那些在第一个阶段被发现的工作。评估准则就是评估器在第四个阶段的检查。而护栏就是最后一个阶段那个"继续、孵化、还是停"的决定。把一个 Loop 设计得好，到头来，就是把这三样东西设计得好。那些组件只是给每一个决定一个落脚的地方。

那么我们就把这一章的重点想法归纳到一起。要理解一个 Loop，最快的方法就是做一个最小、可以跑得起来的版本，盯着它在哪里出事，然后加结构去修正每一个弱点。

一个最小化、Ralph 风格的 Loop，只是一个简单的循环，它叫一个生成器 agent、抓住它的输出和错误、然后把它们回收进下一次迭代的上下文，直到一个哨兵停止条件被满足为止。

那个最小化 Loop 是特意没有隔离、没有独立评估器、也没有保存状态的，而正正是那些缺口，就是之后那个生产级别 Loop 的动机。

生产级别的 Loop 保持同一个"生成、评估、反馈"的心跳，但是它用一个明确的组件去填上每一个缺口。每一次尝试一个用完即弃的工作空间，用来做隔离。一个独立的评估器，它在测试上面、以及在一个评分标准分数上面把闸。一个保存进度的文件，它在崩溃之后仍然存活，并且为下一次尝试提供信息。以及明确的停止条件，成功、一个最大迭代次数、以及一个预算上限，让失控的 Loop 变成不可能。

每一样生产上的新增，都一对一地对应到解剖那一章的一个组件，以及生命周期的一个阶段。做一个 Loop，大部分时候就是把那些零件变得明确、又可以被检视。

用减法去做一个 Loop，一次一个组件。从一个生成器 agent 加一个停止条件开始。在信任它之前加隔离。然后加一个评估器，再加保存状态，再加明确的预算护栏。而自动化、连接器、以及技能与知识，就只在任务真的要求的时候才加。

还有，要记住，一个 Loop 的成败，是取决于三个设计决定，多过取决于任何组件的选择。一个清晰、可核查、以结果来措辞的目标。一套建基于一道独立客观闸（也就是你的测试）加一道评论家或者评分标准闸的评估准则。以及护栏，也就是层层叠的迭代和预算上限、每一次尝试的隔离、加上一条明确的升级给人类的路径，这样一个无人值守的 Loop 就永远都知道什么时候要停下来发问。
