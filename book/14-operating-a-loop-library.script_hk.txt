喺呢一章，我哋假設你已經有一個可以描述同運行 loop 嘅庫，而我哋會聚焦喺：點樣喺個庫增長、用嘅人愈嚟愈多嘅同時，令佢保持優質。我哋會講一個令團隊可以安全噉加 loop 嘅貢獻流程、令劣質產出入唔到嚟嘅品質標準同審查規範、點樣為條目做版本管理同棄用、邊啲度量指標可以話俾你聽一個 loop 值唔值得保留，以及當眾多自主 loop 喺一個團隊之間運行時隨之而嚟嘅治理同安全顧慮。讀完之後，你就有能力將一個 loop 庫當作共享基礎設施嚟營運，而唔只係一個個人嘅腳本資料夾。

噉就由一個心態轉變講起：一個庫係一件產品，唔係一個資料夾。幾個 loop 放喺一個目錄入面，係一個方便。但一個俾成個團隊信任、並針對真實系統運行嘅庫，就係基礎設施，而基礎設施需要一個負責人、一個品質標準、一個安全改動嘅方法，以及一個量度佢有冇值返條數嘅方法。一個成功嘅庫嘅失敗模式，唔係 loop 太少。而係劣質嘅太多，搞到冇人分得清邊個條目可以信。策展，就係防止呢樣嘢嘅工作。

呢一章假設你已經有上一章嗰套機制，schema、儲存、runner、CI 驗證器，並喺佢哋周圍加上人嘅一層：loop 點樣入嚟、佢哋點樣保持優質，以及成件事點樣擴展到一個作者以上。

首先，貢獻流程。你喺一個 loop 內部所套用嗰份紀律，提出一個改動、獨立驗證佢、只有喺通過時先接受，同樣適用於對個庫本身嘅改動。將一個新嘅或者修改過嘅 loop 條目，當作任何其他程式碼改動噉處理：一個 pull request、一次審查，以及一道自動關卡。

呢度有一個由一位貢獻者擴展到多位嘅流程。第一，透過 pull request 提出。一位貢獻者喺一個分支上新增或編輯一個條目檔案，並開出一個 PR。冇任何嘢直接合併入主庫。第二，自動關卡先運行。CI 運行嗰個驗證器：schema 合規、一個存在嘅停止條件，以及一個存在嘅 prompt 檔案。一個過唔到呢啲關卡嘅 PR，永遠到唔到人類審查者嗰度。平嘅檢查先行。第三，一位維護者對照品質標準審查。一個人類去檢查自動化檢查唔到嘅嘢：個目標係咪真係可被檢查、個評估器係咪獨立於生成器、啲護欄係咪老實、標注係咪正確。第四，接受之前先做試運行。對於任何非瑣碎嘅 loop，審查者透過 runner 將佢針對一個安全目標運行一次，確認佢嘅行為同所描述一致，尤其係佢真係會停。一個從未被執行過嘅 loop，係一個假設，唔係一個庫條目。第五，合併將佢晉升為 stable，並重新生成個索引。

呢個原則係遞迴嘅，而且值得講白：一個 loop 庫，本身就係由一個 loop 嚟維護。提出、獨立驗證、只有喺通過時先接受。魔鬼代言人同建造者審查者形態，套用喺庫條目身上，同套用喺嗰啲條目所作用嘅程式碼身上，係一樣咁多嘅。

而家講品質標準。自動驗證證明一個條目形式正確。品質標準就證明佢優質。每一個獲准入庫嘅條目，都應該通過一份書面、可見嘅清單，存放喺儲存庫入面，令貢獻者喺開 PR 之前可以自我檢查。個目標係外部可被檢查嘅，措辭為一個機器或者一個獨立審查者可以確認嘅成果，而唔係「改善 X」。個評估器獨立於生成器，於是「完成」永遠唔係靠 agent 自己嘅報告。停止條件係明確而分層嘅：一個成功出口加上至少一個預算上限。一個無界限嘅條目即時不過關。隔離同一條人類升級路徑都有定義。波及範圍有記錄。而標注同來源係正確嘅。

將呢份清單做成一個 PR 範本，咁每一個貢獻都會對照同一套準則去審查。一個過唔到標準嘅 loop 唔係永遠被拒絕。佢以 experimental 或者 draft 嘅身分獲准，清楚標明，令用戶知道佢仲未贏得完全嘅信任。

跟住，版本管理同棄用。Loop 會變：一個 prompt 改善咗、一個評估器收緊咗、一個更好嘅做法取代咗一個舊嘅。因為每一個條目都帶住一個語意化嘅 version 同一個 maturity 欄位，個庫可以喺唔搞冧依賴佢嘅人嘅前提下演化。為每一個有意義嘅改動做版本：當行為改變時提升個 version，咁釘住某個版本嘅呼叫者得到可重現性，而追蹤最新版嘅呼叫者得到改善。用成熟度階梯：新條目以 draft 或者 experimental 進入，一經證實就晉升為 stable，被取代時就以 deprecated 終結。而棄用，唔好刪除。當一個 loop 被取代時，將佢標為 deprecated、指向佢嘅後繼者，並保留佢一個發行週期。靜靜雞刪除一個條目，會悄悄噉搞冧任何引用咗佢嘅嘢。成熟度階梯，正正就係令一個庫能夠歡迎新 loop 而又唔對信任掉以輕心嘅嘢。

而家，度量一個 loop 值唔值得佢嗰個位。一個庫入面嘅 loop，係一個「佢值得運行」嘅聲稱。一旦啲 loop 定期運行，你就可以用證據去取代嗰個聲稱。因為個 runner 本身已經驅動每一個 loop，所以你只要為佢加裝一次儀表，就免費得到成個庫嘅遙測。追蹤接受率：喺到達一個停止條件嘅運行之中，有幾多係以成功結束，而唔係撞到預算上限或者升級。追蹤到達成功所需嘅迭代次數同成本：成本隨時間上升，係一個漂移嘅早期警告。追蹤升級率：一啲升級係健康嘅，但持續升級就意味住個 loop 範圍定錯咗。追蹤使用量：一個冇人運行嘅條目係文件，唔係基礎設施，係一個棄用嘅候選。呢啲直接連返嗰啲停止條件：嗰啲界定一個 loop 行為嘅出口，同時亦係你點算嚟判斷佢嘅事件。

然後就係規模化嘅治理同安全。一旦一個庫被共享、而佢啲 loop 喺好多儲存庫之間自主運行，啲風險就唔再係一個工程師嘅問題，而變成一個組織嘅問題。一個出事嘅個人 loop 頂多令你要移除一個 worktree。而一個有廣泛權限、出事嘅共享 loop，就可能影響到所有人。最重要嘅控制係以下幾個。最小權限連接器：一個 loop 嘅連接器應該只授予佢需要嘅存取，可能嘅話唯讀，並且唔好將長期生產憑證焗死喺條目入面。中央預算上限：喺逐條目嘅成本上限之上，執行一個組織層級嘅上限，咁眾多 loop 嘅總和就唔會失控。一條審計軌跡：記錄每一次運行，邊個條目、邊個版本、邊啲輸入、邊個觸發、佢改咗乜、佢點樣停。高波及範圍 loop 嘅人類審批關卡：任何掂到生產資料、公開文案或者不可逆狀態嘅嘢，都應該要求明確嘅人類簽核。以及一個清晰嘅負責人：有一個人為品質標準、棄用決定同安全姿態負責。呢啲都唔係新諗法。佢哋就係之前嗰啲緩解措施，由一個單一 loop 提升到一整隊 loop。

最後，擴展個庫本身。當佢增長到超過幾打條目時，兩股壓力會出現：發現變得更難，而重複亦會悄悄爬入嚟。倚靠形態，而非條目：當兩個條目係同一種形態、只係目標唔同嗰陣，寧可用一個參數化嘅條目，都唔好用兩個近乎重複嘅。個庫應該喺參數同 prompt 入面增長，而唔係喺複本入面。針對蔓延做策展：定期對個庫做一次清理巡查，啱啱好，呢樣嘢本身都係一個 loop，去搵出冇人運行嘅條目、重複、失效引用同過時 prompt，並棄用或合併佢哋。以及按領域聯邦化、共享 runner：大型組織可以容許各團隊擁有自己嘅類別資料夾，同時共享一個 schema 同一個 runner。

終局係一個相對於佢嘅有用程度而保持細小嘅庫：一套經過策展、可信、參數化、被度量嘅 loop，俾成個團隊去取用，而唔係一堆不斷增長嘅一次性腳本。呢個就係本書由頭到尾所主張嗰份減法紀律，套用喺高一個層級之上。當「冇將一個 loop 放入個庫」會造成真正嘅重複嗰陣，先將佢加入，喺佢值返個位嗰段時間保留佢，並喺佢唔再值得時退役佢。

噉等我哋將啲關鍵諗法收攏埋一齊。一個共享嘅 loop 庫係基礎設施，唔係一個資料夾，而佢嘅失敗模式係劣質條目太多，所以策展係核心工作。對個庫嘅改動，跟住同 loop 本身一樣嘅 loop 紀律：透過 PR 提出、運行自動關卡、對照一份書面品質標準審查，並喺合併到 stable 之前做試運行去確認個 loop 真係會停。個品質標準要求一個可被檢查嘅目標、一個獨立嘅評估器、明確而分層嘅停止條件、有定義嘅隔離同升級、一個有記錄嘅波及範圍，以及正確嘅標注。做版本同棄用，而非搞冧，用一條以搜尋分面暴露出嚟嘅成熟度階梯。用每一個 loop 自己嘅停止條件本身已經產生嘅事件去度量佢，並退役嗰啲喺度亂撞或者冇人運行嘅 loop。治理將之前嗰啲緩解措施擴展到一整隊：最小權限連接器、組織層級嘅預算上限、一條審計軌跡、人類審批關卡，以及一個清晰可問責嘅負責人。而按形態擴展，而非按複本，令個庫相對於佢嘅有用程度而保持細小。
