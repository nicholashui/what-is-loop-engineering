In this chapter, we move from building a single loop to studying a whole library of them. We'll tour a public collection of fifty production loops, the Forward Future Loop Library, organized into five categories: engineering, evaluation, design, content, and operations. For each loop, you'll get its author, its purpose, and a plain-language summary of how it works, mapped back to the eight components and the lifecycle from earlier chapters. By the end, you'll have a reference shelf of proven loop patterns to adapt, and the raw material for the implementation chapters that follow.

So let's start with why you'd want a library, not just a loop. Earlier we taught you to build one loop and harden it. But teams rarely run a single loop forever. The same handful of shapes, audit-and-fix, generate-and-score, build-and-review, clean-up-safely, keep recurring across projects with different goals plugged in. A loop library is the natural response. It's a curated, reusable collection of loop definitions that a team can browse, copy, and adapt, instead of re-deriving each one from scratch.

This chapter is a guided tour of one such library, so you have concrete patterns in hand before the next chapter shows you how to implement a library of your own. Each entry is a real, named loop with an attributed author. Read it as a pattern catalog. Notice how every loop, however different its domain, is still the same heartbeat from this book: generate, evaluate, feed back, stop. It just has a domain-specific goal and a domain-specific check bolted on.

As you read, keep asking the three diagnostic questions from earlier chapters. What is the checkable goal? What independent check decides "done"? And what bounds the run? Almost every loop answers all three, and that's exactly what makes them worth cataloging.

Let's start with the engineering loops. This is the largest category. These loops operate directly on code, repositories, and running systems, and they lean heavily on objective evaluators: test suites, builds, benchmarks, and production telemetry.

There's the docs sweep, by Matthew Berman, which reviews the codebase and updates stale documentation, then opens a pull request as its reviewable stopping artifact. There's Peter Steinberger's architecture satisfaction loop, which refactors in small, tested, committed checkpoints until the architecture is satisfactory. There's the sub-fifty-millisecond page-load loop, which optimizes every page and measures load time under identical conditions, stopping only when every page loads under fifty milliseconds. The production error sweep traces production errors to their root cause, fixes and verifies them, and opens a PR, or stops cleanly if there's nothing actionable. The hundred-percent test coverage loop adds tests until coverage hits its target. The logging coverage loop fills logging gaps on every important path. The nightly changelog loop updates the changelog each night. And the test-suite speed loop makes the suite faster without weakening coverage or behavior.

Several engineering loops focus on safe, verified change. The repository cleanup loop recovers valuable work and removes proven-stale branches and worktrees. Hiten Shah's ticket-to-PR-ready loop reproduces a failure in the smallest environment, proves the root cause, makes the smallest credible fix, and reruns the reproduction plus regression tests, and if it can't reproduce after two serious attempts, it says so. The Clodex adversarial-review loop has Codex review Claude's pull request until blocking findings are resolved, and crucially, never reports an errored or exhausted run as approved. The Loop Harness verification loop ships scheduled work only after an independent verification pass. The fresh-clone loop repeats clean onboarding from the README in a disposable environment, carrying nothing between attempts. The autonomy-loop builder-reviewer loop passes code between a builder and a reviewer, accepting a change only when a red-before, green-after test proves it on both passes. And the Codex completion-contract loop defines "done" up front and requires evidence for every reported result.

A few more round out the category. The five-minute repository maintainer loop wakes every five minutes to move repository work forward without interrupting active agents. The recent-feedback sweep turns recent user corrections into a project-wide audit. The propagation compliance loop, after one value changes, finds every other place still showing the old value. The Goal Forge loop turns a rough idea into measurable planning files before a long run starts. The cold-load trimmer loop reduces data downloaded before the first screen, keeping changes only if screenshots stay pixel-identical. The housekeeper loop cleans a project one proven, low-risk change at a time. The prepare-a-new-project loop strengthens documents until two independent engineers would build the same system. The test stabilizer loop finds flaky tests, fixes their root causes, and proves stability with repeated full-suite runs. And the Groundtruth loop, by Mohamed, runs a read-only audit of a project's real architecture, security, and operational assumptions, recording proved, no-issue, weak, or not-applicable for each area from current evidence rather than framework defaults.

Now the evaluation loops. These exist to judge: products, prompts, policies, claims, and even other agents. Their defining trait is a deliberately designed evaluator, things like holdout sets, rubrics, must-pass checks, and independent reviewers.

The quality streak loop fixes failures until a defined streak of tests passes in a row. The full product evaluation loop tests every major capability against criteria defined before testing. The self-improving champion loop promotes a change only when it beats the champion on fresh holdout cases without weakening a must-pass check. The devil's-advocate loop has a critic argue a design is wrong until every high-impact objection is resolved or explicitly accepted. The Revolve versioned-experiment loop improves prompts or code through comparable, checkpointed experiments. The promise-to-proof loop checks whether every customer-facing claim is true and fixes the riskiest mismatch first. The multi-LLM convergence loop has two genuinely different AI systems review the same work until both approve one unchanged version. The easy onboarding loop acts like a first-time user, fixes one obstacle, and retries from a clean session. The Axelrod subagent arena loop tests whether agents learn to cooperate or retaliate in a repeated game. The artifact-to-skill loop extracts the method behind a strong artifact and proves it works on a fresh case. And the Strip Miner loop, by Alex Burkhart, mines explicitly authorized coding-agent history for workflows with at least three independent successes, treats the transcripts as untrusted evidence, and validates each candidate with a fresh replay before keeping it.

Next, the design loops. These target visual, interaction, and front-end fidelity. Their evaluators are unusual: repeatable screenshots, pixel comparisons, scoring rubrics, and accessibility scans stand in for a test suite.

The Boeing 747 benchmark builds and improves a Three.js model across nine repeatable views, fixing the weakest feature each round. War Loops reconstructs a real interface and repairs its weakest visual and motion mismatches. The Infinite Clickbait thumbnail loop iterates thumbnail concepts until one clears the quality bar without misleading viewers. The UI/UX Score Loop walks a real user task, scores each screen, and improves the weakest safe area. The pixel-safe CSS trim loop shrinks the CSS sent to users while keeping every screen visually identical. And the accessibility repair loop finds barriers for keyboard, screen-reader, and low-vision users, and fixes the most harmful first, never silencing a check.

The content category is small but distinct: loops that produce or maintain published content, gated on accuracy and source-grounding rather than tests. The SEO/GEO visibility loop fixes the highest-impact gaps in search and AI-answer visibility. And the product update podcast loop turns meaningful product updates into a short, source-grounded episode, making no episode if nothing meaningful shipped.

Finally, the operations loops. These manage releases, production data, and customer-facing AI workflows, emphasizing controlled rollout and monitoring. The stale-safe batch release loop batches valid changes and excludes stale or unfinished work. The production data cleanup loop removes disallowed data and improves the classification logic so the error doesn't return. The post-release baseline loop benchmarks each release and records a reproducible baseline. The customer AI deployment loop moves one customer AI priority through validation, controlled rollout, and monitoring. The Living Story loop, by Buddy Hadry, turns repository activity, goals, and prior open threads into a verified, recurring narrative for the next agent, carrying every thread forward instead of silently dropping it. The Recovery Proof loop, by Eric Lott, restores randomly selected real backups from zero in a disposable clean-room, measures the actual recovery point and recovery time objectives, and repeats until each scenario passes a consecutive-success streak. And the refund follow-up loop, by Jason, starts a refund claim through an approved channel and keeps following up through every reply, promise, and deadline until the money actually arrives.

Now step back from the fifty individual entries, and the same handful of shapes appears again and again. Recognizing the shape is what lets you reuse a loop you've never seen for a problem it was never written for.

There's audit-rank-fix-reverify: scan a scope, rank findings by impact, fix the highest-leverage one, then rerun the same scan. There's generate-score-improve-the-weakest: produce several candidates, score them on a fixed rubric, and improve the weakest dimension of the best ones until a threshold is met. There's build-then-independently-review: one agent produces, a different agent or model verifies, and work is accepted only on an independent pass. There's reproduce-from-clean-state: start from a pristine environment every attempt, fix one obstacle, and retry from scratch. There's streak or threshold convergence: keep iterating until N successes in a row, a numeric target, or a coverage figure is reached. And there's plan-before-you-build: refuse to start implementation until measurable, reviewed planning artifacts exist.

Each shape is a reusable template, waiting for a goal, an evaluator, and a stopping condition. And that's the bridge to the next chapter. If loops cluster into a small number of reusable shapes, you can catalog them deliberately, and that's exactly what implementing a loop library means.

So let's bring the key ideas together. A loop library is a curated, reusable collection of loop definitions, and this chapter surveyed one public example, the fifty-loop Forward Future Loop Library, as a pattern catalog. The library sorts loops into five categories: engineering, gated on tests and telemetry; evaluation, gated on rubrics and independent reviewers; design, gated on screenshots and accessibility scans; content, gated on accuracy and source-grounding; and operations, gated on controlled rollout and monitoring.

However different their domains, every loop is still the book's heartbeat, generate, evaluate, feed back, stop, with a domain-specific goal, evaluator, and stopping condition plugged in. The fifty loops collapse into a few reusable shapes, and recognizing the shape is what makes a loop reusable beyond its original purpose. And because loops cluster into a small number of shapes, they can be cataloged deliberately, which is what implementing a loop library is all about.
