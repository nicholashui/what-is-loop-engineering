在这一章，我们会从建造单一个 loop，走向研究整个由很多 loop 组成的库。我们会导览一个公开的集合，里面有五十个生产级 loop，也就是 Forward Future Loop Library。它分成五大类：工程、评估、设计、内容和运营。对每一个 loop，你都会看到它的作者、它的目的，以及一段用浅白语言写的运作摘要，再对应回之前几章那八个组件和生命周期。读完之后，你就会有一个由经过验证的 loop 形态组成的参考书架，也是之后几章"实作"所需要的原材料。

那就从"为什么你会想要一个库，而不只是一个 loop"讲起。之前我们教你搭一个 loop 并且把它巩固。但团队很少会永远只跑一个 loop。同样那几种形态——审计再修正、生成再评分、建造再审查、安全地清理——会在不同项目里反复出现，只是换上不同的目标。一个 loop 库就是对这件事的自然回应。它是一个经过策展、可重用的 loop 定义集合，给团队去浏览、复制和改编，而不用每次都从零开始重新推导。

这一章是其中一个这样的库的导览，让你在下一章学会怎么实作自己的库之前，先掌握一些具体的形态。下面每一个条目都是一个真实、有名有姓、又标明了作者的 loop。请把它当成一个形态目录来读。留意每一个 loop，无论它的领域有多不同，骨子里都仍然是本书那个心跳节奏：生成、评估、反馈、停止。它只是加装了一个针对特定领域的目标，和一个针对特定领域的检查。

讲多一句关于来源。这里每一个 loop，都是取材自公开发布的 Forward Future Loop Library，五十个 loop，最后更新是 2026 年 6 月 21 日，里面每一个 loop 都标明了贡献者。这些名字和作者是照搬自那个来源。至于"怎么运作"的摘要，则是为了符合授权限制而用我们自己的文字重新表述，并且把每一个 loop 对应回本书的词汇。如果想要逐字、可即用的 prompt 原文，就请查阅原始的库条目。

读的时候，请持续问回之前几章那三条诊断问题。可被检查的目标是什么？由谁独立检查去裁定"完成"？以及，什么去界定运行的范围？下面几乎每一个 loop 都答得到这三条，而这一点，正好就是它们值得编入目录的原因。

我们先讲工程类的 loop。这是最大的一类。这些 loop 直接作用在程式码、储存库和运行中的系统上，并且大量依赖客观的评估器：测试套件、构建、基准测试，以及生产环境的遥测数据。

有文件清扫，作者是 Matthew Berman，它会审视程式码库、更新过时的文件，然后开出一个 pull request 作为它可供审查的停止产物。有 Peter Steinberger 的架构满意 loop，它以小巧、有测试、有提交的检查点不断重构，直到架构令人满意。有五十毫秒以下载入 loop，它优化每个页面，在相同条件下量度载入时间，只有当每页都在五十毫秒以下载入时才停。生产错误清扫会把生产错误追查到根因、修正并验证、再开一个 PR，如果没有可行动的东西就干净地停。一百个百分比测试覆盖率 loop 会不断加测试直到覆盖率达标。日志覆盖 loop 会在每一条重要路径补回日志的缺口。每晚变更日志 loop 每晚更新变更日志。而测试套件提速 loop 就在不削弱覆盖率或行为的前提下，让套件跑得更快。

有几个工程 loop 着重安全、经验证的改动。储存库清理 loop 会回收有价值的工作，并移除已证实过时的分支和 worktree。Hiten Shah 的工单到 PR 就绪 loop，会在最小的环境重现故障、证明根因、做最小而可信的修正，再重跑重现步骤加回归测试；如果认真试了两次都重现不了，它就直接说明。Clodex 对抗式审查 loop 会由 Codex 审查 Claude 的 pull request，直到阻塞性的发现都解决了，而最关键的是，它绝对不会把一次出错或者耗尽的运行报告成"已批"。Loop Harness 验证 loop 只有在通过一次独立验证之后才交付排程工作。全新克隆 loop 在一个用完即弃的环境里，从 README 重复干净的入门流程，两次尝试之间什么都不带。自主 loop 建造者审查者 loop 把程式码在建造者和审查者之间传递，只有当一个"改前红、改后绿"的测试在两遍都通过、证明了那个修正时，才接受改动。而 Codex 完成契约 loop 就事前定义"完成"，并要求每一个报告的结果都有证据。

还有几个补全这一类。五分钟储存库维护者 loop 每五分钟醒一次，在不打断活跃 agent 的前提下推进储存库工作。近期反馈清扫把近期的用户更正变成一次全项目的审计。传播合规 loop 在改了一个值之后，找出仍然显示旧值的每一个其他地方。Goal Forge loop 在一次长运行开始之前，把一个粗略构思变成可度量的规划文件。冷载入瘦身 loop 减少第一屏之前下载的数据，只有当截图保持逐像素一致时才保留改动。管家 loop 一次一个经证实、低风险的改动地清理一个项目。筹备新项目 loop 强化文件，直到两个独立工程师都会建出同一个系统。测试稳定器 loop 找出不稳定的测试、修正它们的根因，并以反复的整套运行去证明稳定性。而 Groundtruth loop，作者是 Mohamed，就做一次只读审计，从当前的证据而非框架预设出发，把项目的真实架构、安全和运营假设逐个范畴记录为已证实、无问题、薄弱或不适用。

现在讲评估类的 loop。这些 loop 存在的目的就是去裁定：裁定产品、prompt、政策、声称，甚至其他 agent。它们的标志性特征，就是一个刻意设计的评估器，例如保留集、评分准则、必过检查和独立审查者。

品质连胜 loop 不断修正失败，直到一段指定长度的测试连胜达成。完整产品评估 loop 对照测试之前就定好的准则，测试每一项主要能力。自我改进冠军 loop 只有当一个改动在全新的保留个案上胜过冠军、而又没削弱任何必过检查时，才晋升它。魔鬼代言人 loop 会由一个评论者去论证这个设计是错的，直到每一个高影响的反对都解决或者明确接受。Revolve 版本化实验 loop 透过可比较、有检查点的实验去改进 prompt 或程式码。承诺到佐证 loop 检查每一个面向客户的声称是不是属实，并先修风险最高那个落差。多 LLM 收敛 loop 由两个真正不同的 AI 系统审查同一份工作，直到两者都批准同一个未经改动的版本。轻松入门 loop 扮演首次使用的用户，修正一个障碍，并从一个干净的工作阶段重试。Axelrod 子 agent 竞技场 loop 测试 agent 在一个重复的博弈中会不会学会合作或报复。产物到技能 loop 抽取一件出色产物背后的方法，并在一个全新个案上证明它有效。而 Strip Miner loop，作者是 Alex Burkhart，就从获明确授权的编程 agent 历史中，开采出至少有三次独立成功的工作流程，把那些纪录稿当作不可信的证据，并在保留之前用一次全新重放去验证每一个候选。

接着是设计类的 loop。这些 loop 针对视觉、交互和前端的保真度。它们的评估器很特别：可重复的截图、像素比对、评分准则和无障碍扫描，取代了测试套件的角色。

Boeing 747 基准用 Three.js 建造并改进一架模型，跨越九个可重复的视角，每一轮修正最弱那个特征。War Loops 重建一个真实界面，并修补它最弱的视觉和动态落差。无限标题党缩图 loop 迭代缩图概念，直到有一个在不误导观众的前提下通过品质标准。UI/UX 评分 loop 走一次真实的用户任务，为每个画面评分，并改进最弱的安全区域。像素安全 CSS 瘦身 loop 缩减送给用户的 CSS，同时让每个画面在视觉上保持一致。而无障碍修复 loop 就找出键盘、屏幕阅读器和低视力用户的障碍，并先修最有害那个，绝不静音任何检查。

内容类很小但很独特：这些 loop 产出或维护已发布的内容，关卡是准确性和来源依据，而不是测试。SEO/GEO 能见度 loop 修正搜索和 AI 答案能见度中影响最大的缺口。而产品更新播客 loop 就把有意义的产品更新变成一集短小、有来源依据的节目，如果没有有意义的东西上线就不制作。

最后，运营类的 loop。这些 loop 管理发行、生产数据和面向客户的 AI 工作流程，着重受控的推出和监察。过时安全批次发行 loop 批量处理有效的改动，并排除过时或未完成的工作。生产数据清理 loop 移除不允许的数据，并改进分类逻辑让那个错误不会再回来。发行后基准 loop 为每个发行做基准测试，并记录一个可重现的基准。客户 AI 部署 loop 把一个客户 AI 优先事项，经过验证、受控推出和监察地推进。Living Story loop，作者是 Buddy Hadry，把储存库活动、目标和先前的未结线索，变成一个经验证、会反复更新的叙事给下一个 agent，把每一条线索带着向前而非悄悄地丢掉。Recovery Proof loop，作者是 Eric Lott，在一个用完即弃的洁净室中，从零复原随机挑选的真实备份，量度实际的复原点和复原时间目标，并重复直到每个情境都通过一段连续成功的次数。而退款跟进 loop，作者是 Jason，就透过一个已批准的渠道开展一宗退款申索，并捱过每一个回复、承诺和限期持续跟进，直到钱真的到手。

现在从五十个个别条目退后一步，同样那几种形态会一次又一次地出现。懂得辨认那个形态，正好就是让你能够重用一个你从未见过的 loop，去解决一个它从未为之而写的问题。

有审计、排序、修正、再验证：扫描一个范围，按影响为发现排序，修正杠杆最大那个，然后重跑同样的扫描。有生成、评分、改进最弱者：产生几个候选，用一个固定准则评分，并改进最佳几个的最弱维度，直到达到一个门槛。有先建造、再独立审查：一个 agent 负责产出，一个不同的 agent 或模型负责验证，只有在通过一次独立检查时才接受工作。有从干净状态重现：每次尝试都从一个全新环境开始，修正一个障碍，再从零重试。有连胜或者门槛收敛：不断迭代直到连续 N 次成功、一个数值目标，或者一个覆盖率数字达成。还有先规划、后建造：在存在可度量、经审查的规划产物之前，拒绝开始实作。

每一种形态都是一个可重用的模板，等着填入一个目标、一个评估器和一个停止条件。而这个，就是通往下一章的桥梁。如果这些 loop 会聚集成一小撮可重用的形态，那你就可以刻意地把它们编入目录，而这件事，正好就是实作一个 loop 库的意思。

那让我们把这些关键想法收拢到一起。一个 loop 库是一个经过策展、可重用的 loop 定义集合，而这一章审视了一个公开例子，就是五十个 loop 的 Forward Future Loop Library，作为一个形态目录。该库把 loop 分成五大类：工程，以测试和遥测为关卡；评估，以评分准则和独立审查者为关卡；设计，以截图和无障碍扫描为关卡；内容，以准确性和来源依据为关卡；以及运营，以受控推出和监察为关卡。

无论领域有多不同，每一个 loop 都仍然是本书那个心跳节奏，生成、评估、反馈、停止，只是插入了一个针对特定领域的目标、评估器和停止条件。这五十个 loop 会收束成几种可重用的形态，而懂得辨认那个形态，正好就是让一个 loop 能够超越它原本目的去重用的关键。而因为 loop 会聚集成一小撮形态，所以它们可以被刻意地编入目录，这件事就是实作一个 loop 库的核心所在。
