# 12. Loop 庫：真實世界 Loop 嘅總目錄

> **本章內容：** 你會由建造*單一*個 Loop（迴圈）（第 08 章），走向研究一個由眾多 Loop 組成嘅*庫*。我哋會審視一個公開嘅 45 個生產級 Loop 嘅集合——[Forward Future Loop Library](https://signals.forwardfuture.ai/loop-library/)——並沿用該庫所用嘅同一套五大分類嚟組織：Engineering（工程）、Evaluation（評估）、Design（設計）、Content（內容）同 Operations（營運）。對每一個 Loop，你都會見到佢嘅作者、佢嘅目的，以及一段用淺白語言寫成、並對應返八個組件（第 06 章）同生命週期（第 07 章）嘅運作摘要。讀完之後，你就會擁有一個由經過驗證嘅 Loop *形態*所組成嘅參考書架——亦即係之後幾章「實作」所需嘅原材料。

## 點解要一個庫，而唔淨係一個 Loop

第 08 章教你建造*一個* Loop 並將佢鞏固。但團隊好少會永遠淨係運行單一個 Loop。同樣嗰幾種形態——*審計並修正*、*生成並評分*、*建造並審查*、*安全清理*——會喺唔同嘅項目度反覆出現，只係換上唔同嘅目標。**Loop 庫（Loop Library）** 就係對此嘅自然回應：一個經過策展、可重用嘅 Loop 定義集合，團隊可以瀏覽、複製同改編，而毋須每次都由零開始重新推導。

本章係其中一個咁樣嘅庫嘅導覽，目的係令你喺第 13 章學識*實作*自己嘅庫之前，先掌握一啲具體嘅形態。下面每一個條目都係一個真實、有名有姓、並標明咗作者嘅 Loop。請當佢係一個形態目錄嚟讀：留意每一個 Loop，無論佢嘅領域有幾唔同，骨子裏面都仍然係本書嗰個「生成 → 評估 → 回饋 → 停止」嘅心跳節奏，只係加裝咗一個針對特定領域嘅目標同一個針對特定領域嘅檢查。

> **來源與標注。** 本章入面每一個 Loop，都係取材自公開發佈嘅 [Forward Future Loop Library](https://signals.forwardfuture.ai/loop-library/)（45 個 Loop，最後更新於 2026 年 6 月 20 日），該庫對每一個 Loop 都標明咗貢獻者。下文嘅名稱同作者係照搬自該來源；至於「點樣運作」嘅摘要，就**為咗符合授權限制而用我哋自己嘅文字重新表述**，並將每一個 Loop 對應返本書嘅詞彙。如果想要逐字、可即用嘅 prompt（提示）原文，請查閱原始庫條目。若此處任何標注與來源有出入，以來源為準。

## 點樣讀呢個目錄

該庫將 Loop 分成五大類。我哋保留呢個分組方便你交叉參照，而每一類入面，每一個條目都跟住同一個格式：

- **Loop 名稱** —— *作者* —— 佢喺來源網站上嘅分類。
- **目的** —— 呢個 Loop 想達成嘅一句話成果。
- **點樣運作** —— 一段對該 Loop prompt（提示）嘅重新表述摘要，特別點出佢嘅停止條件、佢嘅評估器，以及任何護欄（guardrails）。

讀嘅時候，請持續問返之前幾章嗰三條診斷問題：*可被檢查嘅目標係乜？由邊個獨立檢查去裁定「完成」？乜嘢去界定運行嘅範圍？* 下面差唔多每一個 Loop 都答到呢三條——而呢一點，正正就係佢哋值得編入目錄嘅原因。

## Engineering（工程）Loop

最大嘅一類。呢啲 Loop 直接作用於程式碼、儲存庫同運行中嘅系統，並且大量依賴客觀嘅評估器——測試套件、建構、基準測試（benchmark），以及生產環境嘅遙測資料。

### 文件清掃（The docs sweep）—— *作者：Matthew Berman*

- **目的：** 令文件同當前程式碼保持一致，並開出一個可供審查嘅 pull request（PR，拉取請求）。
- **點樣運作：** 當觸發一次文件巡查時，agent（代理）會審視成個程式碼庫，搵出唔再對應實作嘅文件，更新佢、驗證改動，然後開出一個 PR。嗰個 PR 就係可供審查嘅停止產物。

### 架構滿意 Loop（The architecture satisfaction loop）—— *作者：Peter Steinberger*

- **目的：** 以細小、獨立審查、有測試嘅檢查點，逐步重構架構。
- **點樣運作：** agent 不斷重構，直至架構令人滿意。每行完一個重要步驟，佢就會即時測試系統、運行自動審查、然後提交（commit），並喺一個臨時檔案（例如 `/tmp/refactor-{project}.md`）追蹤進度，令工作可以續行。「對架構滿意」嗰個判斷就係停止。

### 50 毫秒以下載入 Loop（The sub-50 ms page-load loop）—— *作者：Matthew Berman*

- **目的：** 優化每一個頁面，直至佢持續喺 50 毫秒以下載入。
- **點樣運作：** agent 持續為速度優化；每次行完一個重要改動，佢就喺相同、可重複嘅測試條件下，量度每一個頁面嘅載入表現。嗰個數值門檻——每頁都喺 50 毫秒以下——就係停止條件。

### 生產錯誤清掃（The production error sweep）—— *作者：Matthew Berman*

- **目的：** 喺生產環境搵出、修正並驗證有可行動性嘅錯誤。
- **點樣運作：** agent 審視生產日誌；對每一個可行動嘅問題，佢追查根因、修正、驗證修正，然後開出一個 PR。如果冇可行動嘅錯誤，佢就唔改任何嘢直接停——一個乾淨嘅「無操作」停止條件。

### 100% 測試覆蓋率 Loop（The 100% test coverage loop）—— *作者：Matthew Berman*

- **目的：** 不斷加入有意義嘅測試，直至整個套件達到 100% 覆蓋率。
- **點樣運作：** agent 加入測試，迭代直至覆蓋率去到 100%。嗰個覆蓋率指標既係客觀評估器，亦係停止。

### 日誌覆蓋 Loop（The logging coverage loop）—— *作者：Matthew Berman*

- **目的：** 為每一條重要嘅系統路徑加入有用、有測試嘅日誌。
- **點樣運作：** agent 審視系統嘅日誌記錄，補返缺口，直至每一條重要路徑都輸出有用、有測試嘅日誌。

### 每晚變更日誌 Loop（The nightly changelog loop）—— *作者：Matthew Berman*

- **目的：** 令變更日誌（changelog）保持更新，反映有意義嘅改動。
- **點樣運作：** 喺每晚嘅排程（即觸發器）下，agent 審視前一日嘅改動，並將任何用戶應該知道嘅嘢更新入變更日誌。

### 測試套件提速 Loop（The test-suite speed loop）—— *作者：Matthew Berman*

- **目的：** 喺唔削弱覆蓋率、斷言（assertions）或隔離性嘅前提下，加快測試套件。
- **點樣運作：** agent 優化套件，令佢盡可能快噉運行，同時保住覆蓋率同行為——「行為不變」嘅約束就係護欄。

### 儲存庫清理 Loop（The repository cleanup loop）—— *作者：Matthew Berman*

- **目的：** 回收有價值嘅工作，並安全噉移除已證實過時嘅狀態。
- **點樣運作：** agent 檢查本地同遠端嘅分支、PR、提交同 worktrees（工作樹），回收任何有價值嘅嘢，清走任何可驗證地過時嘅嘢，直至儲存庫變得最新而有條理。

### 工單到 PR 就緒 Loop（The ticket-to-PR-ready loop）—— *作者：Hiten Shah*

- **目的：** 將一張工單、漏洞報告或投訴，變成一個經過驗證、可供審查嘅 pull request。
- **點樣運作：** agent 喺最細嘅代表性環境中重現故障，證明根因，做出最細而可信嘅修正，再重跑原本嘅重現步驟加相關回歸測試。如果認真試咗兩次都重現唔到，佢就直接講明。佢避免將無關嘅重構塞入嚟，最後交出根因、改動咗嘅檔案、修正前後嘅證據、風險，以及一份 PR 摘要。

### Clodex 對抗式審查 Loop（The Clodex adversarial-review loop）—— *作者：Lukas Kucinski*

- **目的：** 用一個模型去對抗式噉審查另一個模型嘅 pull request，直至阻塞性嘅發現都得到解決。
- **點樣運作：** Claude 規劃並實作一個任務、開出 PR；Codex 做對抗式審查；Claude 修正高於某個接受嚴重度嘅發現，然後重複。狀態（分支、PR、發現、裁決、迭代）保持可續行。佢喺以下情況停止：獲批、只剩已接受嘅發現、進度停滯，或者到達迭代上限——而且絕對唔會將一次出錯或耗盡嘅運行報告成「已批」。

### Loop Harness 驗證 Loop（The Loop Harness verification loop）—— *作者：Istasha*

- **目的：** 只有喺通過一次獨立驗證之後，先至交付排程性嘅 agent 工作。
- **點樣運作：** 對於排程性嘅儲存庫工作（CI 分流、issue 整理、相依更新、文件同步），一個 agent 工作階段喺一個隔離嘅 worktree 中暫存一個 patch（補丁），另一個工作階段則對照明確準則去驗證佢。只有喺通過時先交付；否則保留發現，並喺設定嘅上限之內重試。

### 全新克隆 Loop（The fresh-clone loop）—— *作者：0xUmbra*

- **目的：** 由 README 出發重複乾淨嘅入門流程，直至冇任何隱藏嘅設定假設殘留。
- **點樣運作：** agent 將儲存庫克隆入一個用完即棄嘅環境，並*只*跟住佢嘅 README 行到所記載嘅就緒狀態。當有一步失敗或者假設咗缺失嘅知識時，佢就記低個缺口、修正設定或文件、棄掉個環境，然後由頭嚟過——兩次嘗試之間乜都唔帶。佢喺以下情況停止：一次不受干擾嘅全新克隆成功、進度停滯，或者預算用盡。

### 自主 Loop 建造者—審查者 Loop（The autonomy-loop builder-reviewer loop）—— *作者：@inferencegod*

- **目的：** 將程式碼喺建造者同審查者之間傳遞，直至測試證明咗每一個被接受嘅修正。
- **點樣運作：** 喺測試／建構／lint 關卡通過之後，一個建造者同一個審查者喺各自嘅 worktree 中運行。建造者做一個有界限嘅改動，並加一個「改前紅、改後綠」嘅測試；審查者重跑關卡，並透過還原或變異嗰個修正去驗證個測試。只有喺兩者都通過時先接受改動；受保護或者反覆失敗嘅工作就泊低俾人手處理。

### Codex 完成契約 Loop（The Codex completion-contract loop）—— *作者：3goblack (@Dis_Trackted)*

- **目的：** 事前定義「完成」，並要求每一個報告嘅結果都有證據。
- **點樣運作：** 喺行動之前，agent 先定義每一個必需嘅成果，以及證明佢嘅證據。每行完一個有界限嘅行動，佢就將每一個需求標記為已證實、薄弱、缺失或被推翻。只有當全部都已證實時目標先算完成；否則佢就以阻塞、停滯或耗盡嘅狀態停低——並喺建立目標狀態之前先問准。

### 五分鐘儲存庫維護者 Loop（The five-minute repository maintainer loop）—— *作者：Peter Steinberger*

- **目的：** 透過專屬執行緒令儲存庫工作持續推進，同時唔打斷活躍中嘅 agent。
- **點樣運作：** 當維護處於啟動狀態時，呢個 Loop 每五分鐘醒一次，分流一組儲存庫，每個庫重用一條執行緒，並喺所獲授權之內，將每個庫最高價值嘅有界限任務指派出去——同時唔打斷連貫進行中嘅工作。佢要求測試、即時佐證、自動審查同綠色 CI 之後先可以落地，並就產品／存取權／安全／不可逆嘅決定升級交人。

### 近期回饋清掃（The recent-feedback sweep）—— *作者：Matthew Berman*

- **目的：** 將近期嘅用戶更正，變成一次全項目嘅審計同經驗證嘅修正。
- **點樣運作：** agent 審視某個回溯窗口內用戶報告過有問題嘅執行緒，建立一份去重複嘅問題清單並按失敗模式分組，就每一種模式審計整個項目，修正每一個確認嘅實例，並加入回歸覆蓋。佢重複整次審計，直至冇剩或者迭代預算用盡。

### 傳播合規 Loop（The propagation compliance loop）—— *作者：@iamTristan*

- **目的：** 喺改咗一個值之後，搵出仲顯示緊舊值嘅每一個其他地方。
- **點樣運作：** 喺改咗一個版本號、計數、規則、名稱或設定之後，agent 列出新值應該出現嘅地方，搜尋舊值同相關形態，並更新真正過時嘅匹配，同時保留刻意嘅歷史紀錄、範例、遷移同相容性規則。佢重複直至冇剩任何過時值；如果有一個反覆返嚟，佢就停低並指出係咩嘢喺度重新產生佢。

### Goal Forge Loop（The Goal Forge loop）—— *作者：michael Guo (@michaelzsguo)*

- **目的：** 喺一次長時間自主運行開始之前，將一個粗略嘅編程構思變成可度量嘅規劃檔案。
- **點樣運作：** agent 訪談用戶，然後寫一個 spec（規格）檔案（要建乜、要排除乜、要考慮乜，加埋可度量嘅完成檢查）同一個 goal（目標）檔案（工作計劃、計分板、快速同最終檢查、記憶檔案、證據，以及審批界線）。如果缺咗任何關鍵決定、權限、工具或測試，佢就以「未就緒」停低，並且未經批准唔會開始實作。

### 冷載入瘦身 Loop（The cold-load trimmer loop）—— *作者：Christian Katzmann*

- **目的：** 喺唔改變行為或外觀嘅前提下，減少 web app 喺第一屏出現之前所下載嘅資料。
- **點樣運作：** agent 記錄通過嘅測試、手機同桌面截圖，以及壓縮後傳輸嘅位元組數作為基準，然後每次延後、壓縮或移除一項。佢只有喺測試通過、截圖保持逐像素一致、位元組數下降時先保留改動；否則還原。佢喺冇剩安全候選、進度停滯或需要審批時停止。

### 管家 Loop（The housekeeper loop）—— *作者：Eric Lott*

- **目的：** 一次一個經證實、低風險嘅改動噉清理一個程式碼項目。
- **點樣運作：** agent 審視死碼、過時檔案／註解、未用嘅相依、重複、失效連結、不一致嘅命名同令人混淆嘅結構——同時保護無關、活躍、未提交、生成嘅同不確定嘅工作。佢證實一個低風險清理，做最細而連貫嘅改動，再重跑建構／測試／執行期／差異檢查，並只保留經驗證嘅改善。

### 籌備新項目 Loop（The prepare-a-new-project loop）—— *作者：Brad Shannon (@bradshannon)*

- **目的：** 強化項目文件，直至獨立嘅工程師都會建出大致相同嘅系統。
- **點樣運作：** agent 確保文件涵蓋需求、技術設計、附驗收準則嘅任務，以及測試策略。每一輪佢修正會令兩個能幹工程師產生分歧嘅最大缺口或矛盾，然後請兩位獨立審查者描述各組件、資料模型、相依關係同完成定義。當佢哋實質上達成一致、而且每件產物都可被測試時，佢就停。

### 測試穩定器 Loop（The test stabilizer loop）—— *作者：hungtv27 (@hungtv27)*

- **目的：** 搵出不穩定（flaky）嘅測試，修正佢哋嘅根因，並以反覆嘅整套運行去證明穩定性。
- **點樣運作：** agent 喺相同條件下將套件運行 N 次，列出結果有變化嘅測試，然後喺根因（共享狀態、時序、排序、外部相依）層面修正最頻繁嗰個 flake——絕不靠盲目嘅 sleep 或重試。佢將嗰個測試跑 N 次，再跑整套，重複直至連續 N 次整套運行都通過。

## Evaluation（評估）Loop

呢啲 Loop 存在嘅目的就係去*裁定*——裁定產品、prompt（提示）、政策、聲稱，甚至其他 agent。佢哋嘅標誌性特徵，就係一個刻意設計嘅評估器：保留集（holdout）、評分準則（rubric）、必過檢查，以及獨立審查者。

### 品質連勝 Loop（The quality streak loop）—— *作者：Matthew Berman*

- **目的：** 修正產品失敗，直至一段指定長度嘅真實測試連勝達成。
- **點樣運作：** agent 測試真實情境；當有一個失敗時，佢記錄該失敗、加入回歸同基準覆蓋、修正佢，然後重新計算連勝。佢喺連續 N 個成功個案之後停止——嗰個連勝長度就係停止條件。

### 完整產品評估 Loop（The full product evaluation loop）—— *作者：Matthew Berman*

- **目的：** 測試每一項主要產品能力，並修正任何低於品質標準嘅結果。
- **點樣運作：** agent 建立 N 個涵蓋每項主要能力嘅真實情境，喺測試*之前*先定義成功準則同一個一致嘅評估方法（通過／失敗或者評分準則），喺相同條件下運行每一個情境並記錄證據，修正任何低於標準者嘅根因，重跑受影響嘅情境，再重跑整套——持續直至每一個情境都達到原本嘅標準。

### 自我改進冠軍 Loop（The self-improving champion loop）—— *作者：Jose C. Munoz*

- **目的：** 只有當 prompt 或政策嘅改動喺全新嘅保留個案上勝出時，先至晉升佢。
- **點樣運作：** agent 保存一個冠軍連同佢嘅分數、一個工作集、未動過嘅保留個案、必過檢查，以及一個預算。每一輪佢根據一個記錄低嘅失敗去改一樣嘢，並只有當挑戰者喺保留個案上以某個邊際勝過冠軍、而又冇削弱任何必過檢查時，先晉升佢；否則冠軍留低。佢喺到達目標、預算上限或冇進展時停止。

### 魔鬼代言人 Loop（The devil's-advocate loop）—— *作者：匿名貢獻者*

- **目的：** 挑戰一個設計，直至每一個高影響嘅反對都得到解決或明確接受。
- **點樣運作：** 喺敲定一個架構、介面或推出計劃之前，由一個評論者去論證佢係錯嘅；每一個反對、佢嘅影響同狀態都被記錄（例如 `.agent-reviews/redteam.md`）。建造者必須修正並驗證每一個高影響弱點，或者記錄低點解佢被接受，而評論者可以重開薄弱嘅回應。佢喺冇剩高影響反對、或者同樣嘅問題連續兩輪都冇新證據噉重複出現時停止。

### Revolve 版本化實驗 Loop（The Revolve versioned-experiment loop）—— *作者：Agent Zero*

- **目的：** 透過可比較、有檢查點嘅實驗，去改進 prompt、程式碼或設定。
- **點樣運作：** 喺一個專屬目錄入面，agent 定義目標同預算，凍結測試同評分，為當前版本打檢查點，並記錄一個基準。每一輪佢測試一個假設，並只保留一個清晰、無回歸嘅勝利；如果評估方法改變，佢就開一個新修訂版並重跑基準。佢喺改動實況檔案之前先問准，並喺成功、無進展、遇阻或預算耗盡時停止。

### 承諾到佐證 Loop（The promise-to-proof loop）—— *作者：Felix Haeberle (@felixhaberle)*

- **目的：** 檢查每一個面向客戶嘅聲稱係咪屬實，然後先修正風險最高嗰個落差。
- **點樣運作：** agent 列出產品喺市場推廣、文件、演示同 AI 答案中所作出嘅每一個面向客戶承諾，將每一個同實際產品行為同證據比較，並標記為已證實、部分證實、誤導、無支持、過時或缺乏證據。佢修正或收窄風險最高嗰個落差，重跑受影響嘅檢查，並重複直至冇剩任何高風險、無支持嘅承諾——而喺改動生產環境或公開文案之前先問准。

### 多 LLM 收斂 Loop（The multi-LLM convergence loop）—— *作者：Donn Felker (@donnfelker)*

- **目的：** 由兩個唔同嘅 AI 系統審查同一份工作，直至兩者都批准同一個未經改動嘅版本。
- **點樣運作：** 喺一個通過次數上限之內，由一個模型家族嘅審查者對照品質標準去審查工作；經驗證嘅發現會被修正，修訂版交俾一個*真正唔同*供應商嘅審查者。成功要求兩者都批准同一個未經改動嘅版本。佢喺到達上限、出現來回擺盪嘅分歧、審查不可用或需要批准時停止。

### 輕鬆入門 Loop（The easy onboarding loop）—— *作者：Eric Lott*

- **目的：** 扮演首次使用嘅用戶，修正一個障礙，並由一個完全乾淨嘅工作階段重試。
- **點樣運作：** 由真實入口、喺一個乾淨工作階段（冇已儲存登入、資料或隱藏設定）開始，agent 只用可見嘅引導去完成入門流程，並記錄障礙。佢用最細、又能保住每一項安全／存取／產品要求嘅改動去修正最差嗰個，棄掉個工作階段再重試——並喺一次不受干擾嘅成功、冇安全修正、被阻擋存取或需要批准時停止。

### Axelrod 子 agent 競技場 Loop（The Axelrod subagent arena loop）—— *作者：Kan Yuenyong (@sikkha)*

- **目的：** 測試 AI agent 喺一個重複嘅二選一賽局中，會唔會學識合作、報復或寬恕。
- **點樣運作：** 兩個推理型 agent 進行一場固定嘅 Axelrod 競賽；每一輪兩者私下選擇合作或背叛，由程式碼記錄同時出手並套用固定計分，並加入「永遠背叛」同「永遠合作」嘅對照玩家。佢運行固定數量嘅週期、配對同回合，隱藏對手類型同私下推理，驗證每一步同每一個總分，並報告排名、推理摘要同違規；一場局部嘅競賽當作未完成。

### 產物到技能 Loop（The artifact-to-skill loop）—— *作者：Hiten Shah (@hnshah)*

- **目的：** 抽取一件出色產物背後嘅方法，並喺一個全新個案上證明佢有效。
- **點樣運作：** agent 記錄一件產物曾經成功嘅證據，定義成功準則，並抽取當中嘅決定、次序、檢查同避免失敗嘅模式（而非表面風格），移除敏感材料。一位獨立審查者將呢個方法套用喺一個全新嘅真實個案上；佢最多修訂兩次，並喺嗰個方法毋須原本產物都能達標時停止——或者報告佢唔具普遍性。

## Design（設計）Loop

呢啲 Loop 針對視覺、互動同前端嘅保真度。佢哋嘅評估器好特別：可重複嘅截圖、像素比對、評分準則同無障礙掃描，取代咗測試套件嘅角色。

### Boeing 747 基準（The Boeing 747 benchmark）—— *作者：@victormustar*

- **目的：** 用 Three.js 建造並改進一架 Boeing 747，跨越九個可重複嘅視角。
- **點樣運作：** 喺建造之前，agent 揀選參考圖像、一個評分準則、一個視覺門檻同一個預算。佢用 Three.js 圖元建造模型，再造一個會喺九個可重複角度截圖嘅裝置。每次改動後佢渲染並為同樣嘅視角評分，由一個評論者指出最弱嗰個特徵，並喺唔令較強視角退步嘅前提下修正佢——保留最佳版本，並喺到達門檻、進度停滯或預算耗盡時停止。

### War Loops：前端重建（War Loops: frontend reconstruction）—— *作者：Swayam*

- **目的：** 重建一個真實介面，並修補佢最弱嘅視覺同動態落差。
- **點樣運作：** 對住一個獲授權嘅 URL 或圖像，agent 用一個真正嘅瀏覽器擷取佢，記錄版面、樣式、內容、動態同響應式行為。佢建造一個靜態鏡像同一個會郁嘅版本，喺桌面／平板／手機尺寸下將兩者同來源比較，並只修補最弱嘅保真訊號——喺每一道關卡都通過、進度停滯或擷取被阻時停止。

### 無限標題黨縮圖 Loop（The Infinite Clickbait thumbnail loop）—— *作者：@Alex_FF*

- **目的：** 迭代縮圖概念，直至有一個喺唔誤導觀眾嘅前提下通過品質標準。
- **點樣運作：** 對於一段指定影片同已批准嘅素材，agent 造十個縮圖概念，喺真實 YouTube 尺寸下對照一個靈感頻道，就清晰度、好奇心、情緒拉力、對比同準確性為每個評分，然後改進首三個嘅最弱維度並重新評分。佢持續迭代最強嗰個概念，直至佢通過品質門檻或預算耗盡——並拒絕任何影片本身交付唔到嘅嘢。

### UI/UX 評分 Loop（The UI/UX Score Loop）—— *作者：Hayden Cassar (@hcassar93)*

- **目的：** 行一次真實嘅用戶任務，為每一個畫面評分，改進弱點，再重新測試。
- **點樣運作：** 喺一個真正嘅瀏覽器入面，agent 每一遍都由全新狀態（冇已儲存登入、cookie 或網站資料）開始行一個用戶流程（例如註冊），喺約定嘅尺寸同模式下擷取有意義嘅畫面，用同一份清單為佢哋評分，並改進最弱嘅安全區域。佢重跑成個流程，只保留無回歸嘅改動，並喺成功、兩遍都冇進展、被阻擋存取或需要批准時停止。

### 像素安全 CSS 瘦身 Loop（The pixel-safe CSS trim loop）—— *作者：Christian Katzmann*

- **目的：** 縮減送俾用戶嘅 CSS，同時令每一個受測畫面喺視覺上保持完全一致。
- **點樣運作：** agent 擷取具代表性嘅頁面、尺寸、主題同互動，並記錄已建構嘅 CSS 大小，將覆蓋率報告只當作建議。佢移除一條宣告或規則、重新建構、再重跑截圖同項目檢查，只有當每一張截圖都逐像素一致、而且已建構 CSS 變細時先保留改動；否則還原。

### 無障礙修復 Loop（The accessibility repair loop）—— *作者：Eric Lott*

- **目的：** 搵出鍵盤、螢幕閱讀器、低視力同其他用戶嘅障礙，然後先修最有害嗰個。
- **點樣運作：** agent 用自動掃描加上可用嘅鍵盤、螢幕閱讀器同其他人手測試，對照一個無障礙標準（例如 WCAG 2.2 AA）去檢查一個範圍，確認每一個問題，按有害程度排序，並修正影響最大嗰個阻塞點。佢重跑同樣嘅檢查加上受影響嘅任務同回歸測試，只保留經驗證嘅修正，並且絕不靜音任何檢查或削弱目標。

## Content（內容）Loop

一個細小但獨特嘅分類：呢啲 Loop 產出或維護已發佈嘅內容，其關卡係準確性同來源依據，而非測試。

### SEO/GEO 能見度 Loop（The SEO/GEO visibility loop）—— *作者：Matthew Berman*

- **目的：** 修正搜尋同 AI 答案能見度中影響最大嘅缺口。
- **點樣運作：** agent 就可爬取性、索引、頁面意圖、標題、內部連結、結構化資料、來源引用同「答案優先」嘅內容做一次 SEO/GEO 審計，按預期影響為缺口排序，修正槓桿最大嗰個，然後喺搜尋引擎同 AI 答案引擎上重跑同樣嘅爬取同目標查詢基準。佢重複直至冇剩關鍵技術問題、亦冇剩高影響缺口。

### 產品更新播客 Loop（The product update podcast loop）—— *作者：Pierson Marks*

- **目的：** 將有意義嘅產品更新，變成一集短小、有來源依據嘅播客。
- **點樣運作：** 每晚 agent 審視公開發佈嘅產品改動，只揀用戶需要知道嘅，並對照產品、文件或發行說明逐一驗證。佢用一個播客生成整合工具，製作一集三至五分鐘嘅節目，講解改咗乜、點解重要、點樣試用，檢查腳本同音訊嘅準確性同讀音，如果冇有意義嘅嘢上線就唔製作，並喺發佈之前先問准。

## Operations（營運）Loop

呢啲 Loop 管理發行、生產資料同面向客戶嘅 AI 工作流程。佢哋着重受控嘅推出、監察，以及排除過時或未完成嘅工作。

### 過時安全批次發行 Loop（The stale-safe batch release loop）—— *作者：Matthew Berman*

- **目的：** 批量處理有效改動，並由最新整合好嘅 main 發佈完整產物。
- **點樣運作：** agent 審視待處理嘅改動同 pull request，排除過時或未完成嘅工作，將有效嘅改動合併，並一齊發佈佢哋——「排除過時／未完成」嗰條規則，就係防止交付半成品嘅護欄。

### 生產資料清理 Loop（The production data cleanup loop）—— *作者：Matthew Berman*

- **目的：** 移除不允許嘅生產資料，並防止同樣嘅分類錯誤再次出現。
- **點樣運作：** agent 審視生產紀錄，移除任何超出允許定義嘅嘢，改進分類邏輯令嗰類錯誤唔會再發生，並驗證剩低嘅資料。

### 發行後基準 Loop（The post-release baseline loop）—— *作者：Matthew Berman*

- **目的：** 為每一個完成嘅發行做基準測試，並記錄一個可重現嘅基準。
- **點樣運作：** 喺當前嘅發行完成之後（即觸發器），agent 運行標準基準測試，並將結果記錄為新嘅基準，供日後比較。

### 客戶 AI 部署 Loop（The customer AI deployment loop）—— *作者：AgentLed.ai Agent*

- **目的：** 將一個客戶 AI 優先事項，經過驗證、受控推出同監察噉推進。
- **點樣運作：** 當客戶要求一個 AI 工作流程、報告一次失敗，或者去到一次營運檢討時觸發，agent 揀一個優先事項（豐富潛在客戶資料、草擬電郵、總結會議、更新 CRM），定義負責人、輸入、審批、成功指標同投資回報假設，喺真實客戶資料上做試運行，修正最細嗰個經驗證問題，然後經已批准嘅階段發佈並監察生產環境。

## 將目錄當作形態嚟讀

由 45 個個別條目退後一步，同樣嗰幾種*形態*會一次又一次噉出現。識得辨認出形態，正正就係令你能夠重用一個你從未見過嘅 Loop，去解決一個佢從未為之而寫嘅問題：

- **審計—排序—修正—再驗證。** 掃描一個範圍，按影響為發現排序，修正槓桿最大嗰個，然後重跑同樣嘅掃描。*（生產錯誤清掃、SEO/GEO 能見度、無障礙修復、承諾到佐證、管家。）*
- **生成—評分—改進最弱者。** 產生幾個候選，用一個固定評分準則為佢哋評分，並改進最佳幾個嘅最弱維度，直至達到一個門檻。*（Boeing 747 基準、無限標題黨縮圖、UI/UX 評分、自我改進冠軍。）*
- **先建造—再獨立審查。** 一個 agent 負責產出，一個*唔同*嘅 agent 或模型負責驗證，只有喺通過一次獨立檢查時先接受工作。*（Clodex、自主 Loop 建造者—審查者、多 LLM 收斂、Loop Harness、魔鬼代言人。）*
- **由乾淨狀態重現。** 每次嘗試都由一個全新環境開始，修正一個障礙，再由零重試——兩次運行之間乜都唔帶。*（全新克隆、輕鬆入門。）*
- **連勝／門檻收斂。** 不斷迭代直至連續 N 次成功、一個數值目標，或者一個覆蓋率數字達成。*（品質連勝、100% 測試覆蓋率、50 毫秒以下載入、測試穩定器。）*
- **先規劃—後建造。** 喺存在可度量、經審查嘅規劃產物之前，拒絕開始實作。*（Goal Forge、籌備新項目、Codex 完成契約。）*

每一種形態都係一個等緊填入目標、評估器同停止條件嘅可重用範本。呢個觀察，就係通往下一章嘅橋樑：如果啲 Loop 會聚集成一小撮可重用嘅形態，咁你就可以刻意噉將佢哋編入目錄——而呢樣嘢，正正就係「實作一個 Loop 庫」嘅意思。

## 重點摘要

- **Loop 庫（Loop Library）** 係一個經過策展、可重用嘅 Loop 定義集合；本章審視咗一個公開例子——擁有 45 個 Loop 嘅 [Forward Future Loop Library](https://signals.forwardfuture.ai/loop-library/)——作為一個形態目錄。*（此處嘅摘要為符合授權而重新表述；可即用嘅 prompt 原文請見來源。）*
- 該庫將 Loop 分成五大類：**Engineering（工程）**（最大，以測試／建構／基準／遙測為關卡）、**Evaluation（評估）**（以評分準則、保留集同獨立審查者為關卡）、**Design（設計）**（以截圖、像素比對同無障礙掃描為關卡）、**Content（內容）**（以準確性同來源依據為關卡），以及 **Operations（營運）**（以受控推出同監察為關卡）。
- 無論領域有幾唔同，每一個 Loop 都仍然係本書嗰個「生成 → 評估 → 回饋 → 停止」嘅心跳節奏，只係插入咗一個針對特定領域嘅**目標**、**評估器**同**停止條件**。
- 呢 45 個 Loop 會收束成幾種可重用嘅**形態**——審計—排序—修正—再驗證、生成—評分—改進最弱者、先建造—再獨立審查、由乾淨狀態重現、連勝／門檻收斂，以及先規劃—後建造——而識得辨認形態，正正就係令一個 Loop 能夠超越佢原本目的去重用嘅關鍵。
- 因為 Loop 會聚集成一小撮形態，所以佢哋可以被刻意噉編入目錄——而呢樣嘢，正正就係**實作一個 Loop 庫**（第 13 章）嘅核心所在。
